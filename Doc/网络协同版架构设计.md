# movTools 网络协同版架构设计

## 1. 目标

将现有 `D:\project\movTools` 单机版制片工具升级为可网络协同的系统，并保留原有核心能力：

- 项目与集管理
- 镜头资料维护
- 文件检查与绑定
- 提取与历史记录
- 本地预览与串片
- 本地 FFmpeg 相关处理

新增能力：

- 用户管理
- 导演审片
- 制作人员项目查看
- 多客户端协同同步

明确边界：

- **不集成 SVN**
- **服务端不处理文件扫描、预览、转码、引用检查**
- **客户端只面向本地/NAS 已有文件工作**

---

## 2. 推荐技术选型

### 客户端

- **Electron + React + TypeScript**

原因：

- 现有单机版已采用该技术栈，迁移成本最低
- 本地文件系统访问、预览、FFmpeg、路径解析都更适合桌面客户端
- 可最大化复用旧项目现有页面、类型、状态管理和主进程服务

### 服务端

- **C# + ASP.NET Core Web API**
- **SignalR** 用于实时协同通知
- **PostgreSQL** 作为协同主数据库

原因：

- 更适合你熟悉的技术背景
- 调试便利
- Docker 化成熟
- 内网/NAS 场景部署简单

### 部署

- **Docker Compose**
- 部署在 NAS 或内网服务器上

## 4. 职责边界

### 4.1 服务端负责

- 用户与角色管理
- 项目、集、镜头主数据
- 镜头状态流转记录
- 文件绑定元数据
- Layout 候选与摘要信息
- 导演审片流程与意见
- 客户端路径映射配置
- 操作日志与同步日志

### 4.2 客户端负责

- 本地文件扫描
- 文件匹配与绑定
- Layout 引用检查
- 视频/图片预览
- 镜头详情页预览源决策以客户端主进程返回结果为准，协同主数据场景下 production / layout 优先代理预览；当前 Electron 客户端通过 `movtools-preview://video?path=...` 自定义协议承载本地预览文件，并由主进程在 `app.whenReady()` 后注册处理器
- FFmpeg 转码、提取、串片
- 本地缓存与离线工作
- 将协同元数据同步到服务端

### 4.3 系统明确不负责

- SVN 更新、提交、冲突处理
- 服务端素材托管
- 服务端文件预览
- 服务端转码
- 云端大文件处理平台

## 5. 路径模型

这是系统设计的关键点。

### 当前原则

当前实现采用“协同元数据 + 客户端本机解析”的混合路径模型：

1. 服务端可以保存项目、集、根目录、binding 等协同路径元数据，用于多客户端回读同一份协同事实
2. 服务端保存的路径事实不等于当前机器一定能访问文件
3. 当前机器是否存在文件、能否预览、能否扫描，必须由客户端本机文件系统、路径映射和主进程预览能力最终判定
4. 客户端仍可通过路径映射把服务端回读的路径或相对路径解析为本机可访问路径
5. 服务端不执行素材扫描、媒体探测、视频预览、FFmpeg 转码或 Layout 引用检查

因此，文档中后续出现的“服务端路径”均应理解为协同元数据，不应理解为服务端可以直接打开或处理素材文件。

### 激活时路径重定位

项目激活时，客户端以服务端记录的绝对路径为基准，只允许做 Windows 盘符差异重定位：

- `Z:\foo\bar` 可自动重定位到 `D:\foo\bar`
- 盘符后面的路径必须保持一致
- 若所有盘符都不存在同一后缀路径，再提示用户创建对应目录
- UNC / NAS 路径暂按原样处理，不做自动改写

### 好处

- 不同机器盘符不同也能协同
- 适合 NAS 挂载差异
- 便于旧项目快速迁移
- 服务端保存协同事实，客户端保留本机路径最终解释权

## 7. 用户角色建议

最少角色：

- 系统管理员
- 制片
- 导演
- 制作人员
- 只读查看者

权限原则：

- 管理员：全部权限
- 制片：项目配置、成员管理、镜头管理、查看全局
- 导演：待审列表、审片、评论、通过/返修
- 制作人员：查看参与项目、提交版本、查看反馈
- 只读：仅查看

## 8. 审片流程

### 目标

让导演基于客户端本地/NAS 文件进行预览与反馈，服务端仅保存协同记录。

### 流程

1. 制作人员将镜头版本标记为待审
2. 客户端同步到服务端，生成 `review_task`
3. 导演客户端查看待审列表
4. 导演客户端解析路径后在本地/NAS 打开预览
5. 导演填写意见：
   - 通过
   - 返修
   - 文字评论
   - 时间点批注
6. 服务端保存审片记录并推送给相关用户

## 9. 同步机制

推荐采用：**本地优先 + 服务端同步**

### Push

客户端把以下变更推送到服务端：

- 镜头资料修改
- 状态流转
- 文件绑定元数据
- Layout 候选与检查摘要
- 审片意见

### Pull

客户端按递增序号拉取：

- 项目更新
- 镜头更新
- 状态变化
- 审片变化
- 成员变化

### 实时通知

使用 SignalR 推送：

- `project.updated`
- `lens.updated`
- `lens.statusChanged`
- `review.created`
- `review.updated`
- `review.commentAdded`

## 10. 冲突策略

### 建议策略

- 镜头基础资料：乐观锁（`row_version`）
- 状态流转：服务端规则校验
- 评论：追加，不覆盖
- 检查摘要：以最新上报为准
- 路径映射：按客户端独立保存

### 示例

如果两个人同时改同一镜头：

- 第一个提交成功
- 第二个提交时版本号不匹配
- 服务端返回冲突
- 客户端提示刷新后重试

## 11. 旧项目迁移原则

### 优先复用

- `src/pages/*`
- `src/types/*`
- `src/stores/*`
- `electron/ipc/*`
- 本地文件处理、预览、FFmpeg 逻辑

### 优先改造

- `electron/services/project/*`
- `electron/services/lens/*`
- `electron/services/file-check/*`

### 核心策略

先把直接写 SQLite 的服务改造成通过 Repository 访问数据，再逐步引入远程同步。

## 12. 第一版范围（MVP）

建议第一版只做：

1. 登录
2. 用户管理
3. 项目/集/镜头同步
4. 镜头状态流转
5. 路径映射
6. 导演审片
7. 制作人员查看反馈
8. 文件绑定和检查摘要上报

不建议第一版就做：

- 微服务
- 自动 SVN 处理
- 服务端文件预览/转码
- 复杂权限细粒度策略
- 高级报表系统

## 14. 最终结论

### 推荐定稿

- 客户端：**Electron + React + TypeScript**
- 服务端：**C# + ASP.NET Core**
- 数据库：**PostgreSQL**
- 实时同步：**SignalR**
- 部署：**Docker Compose on NAS**

### 核心原则

> 服务端只负责协同元数据；客户端只负责本地文件能力；SVN 完全在系统外由人工使用。
