# 审片陪审镜头修复第二阶段前端派工指令包

## 1. 文档用途

本文件用于把“导演工作台正确识别陪审镜头、制片入口与导演入口分离、完成按钮与提示收口”这一阶段的客户端工作直接派发给其他 AI 或开发执行。

目标目录：`D:\project\MovtoolsEx\movtools-client\`

参考文档：

- `D:\project\MovtoolsEx\Doc\项目固化文档.md`
- `D:\project\MovtoolsEx\Doc\通信协议文档.md`
- `D:\project\MovtoolsEx\Doc\审片陪审镜头修复第一阶段后端派工指令包.md`

---

## 2. 本阶段目标

本阶段只解决“导演工作台把陪审镜头误当正式镜头”“制片账号误入导演操作页”和“完成按钮提示不清楚”的问题。

完成后必须满足：

1. 任务详情中的 `context` 镜头会被稳定识别
2. `context` 镜头只可播放，不可显示正式反馈编辑、镜头通过、重新评审、提交反馈
3. 导演 `审片完成` 只受正式镜头约束
4. 制片页面不再误用导演详情/导演操作入口
5. 页面文案能明确区分 `context`、`review` 和 `layout` 补位

---

## 3. 工作边界

1. 本阶段只改客户端仓储映射、页面状态、按钮能力、交互文案与必要前端测试
2. 本阶段不重新设计导演审片主流程，不重写播放器或草稿体系
3. 本阶段必须消费第一阶段后端统一后的字段，不得再造一套本地临时真相源
4. 若发现前端多个地方重复判断参与类型，允许抽公共方法，但不要大规模重构无关逻辑

## 5. 必做事项

1. 收口任务详情镜头参与类型映射，确保 `reviewParticipationMode` / `participationMode` 的兼容读取不会把真实 `context` 静默回落成默认 `review`
2. 统一导演页里所有“是否陪审镜头”的判断真值源，至少覆盖：
   - 是否显示 `当前帧反馈` 编辑器
   - 是否显示 `当前镜头反馈列表` 正式操作
   - 是否允许 `提交反馈`
   - 是否显示 `镜头通过`
   - 是否显示 `重新评审`
   - 是否计入 `审片完成`
3. 核查 `buildPlaybackItems`、`buildPlaybackItemsWithLensDetails`、切镜、重载、预加载、状态刷新后 `reviewParticipationMode` 不会丢失
4. 保持导演 `handleCompleteTask` 调用正式 `completeTask`，不要再回退到 `closeTask` 或其他旧捷径
5. 梳理制片页任务点击入口，确保制片账号不会误走导演专属详情接口；若当前产品要求仅查看，则应走制片详情或只读链路
6. 收口页面文案：
   - `context` 只表示上下文陪审身份
   - `layout` 只表示播放源补位
   - 两者不能互相暗示
7. 优化完成按钮禁用原因，至少在正式镜头未完成时能明确指出阻塞镜头，而不是泛化成空提示
8. 如项目已有前端测试能力，补最少量回归覆盖；若没有自动化测试，则至少补详细自测矩阵

---

## 6. 禁止事项

1. 不要用“当前视频来源是 Layout”来推断镜头就是 `context`
2. 不要只修按钮显示，不修实际调用链和完成判定
3. 不要改坏制片侧 `close` 流程或制片任务管理页现有功能
4. 不要在没有协议依据的情况下新增另一套参与类型状态名
5. 不要大改 `ReviewPage.tsx` 的绘制、草稿、提交主流程结构

---

## 7. 验收标准

1. `context` 镜头在导演工作台只读
2. `context` 镜头不显示正式反馈编辑、镜头通过、重新评审、提交反馈
3. `审片完成` 按钮只受正式镜头影响
4. 制片账号不会再误入导演接口导致 403
5. `npm run typecheck` 与 `npm run build` 通过
6. 文案与禁用提示能清楚说明 `context` / `review` / `layout` 的区别
