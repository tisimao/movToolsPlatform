## 3. 当前已固化范围

### 3.1 协同总体边界

当前已经固化：

- 服务端负责：用户、角色、项目、集、镜头主数据、镜头状态流转、成员权限、协同元数据、binding 协同事实的持久化与回读
- 客户端负责：本地文件系统能力、本地扫描、本地解析、本地预览、本地目录创建、本地 FFmpeg / FFprobe、本地路径映射、本地 Layout 引用检查
- 文件能力不迁回服务端；服务端只记录协同主数据与 binding 事实，不直接做客户端本地文件扫描、路径解析、媒体探测或视频预览生成
- 后续业务文档不再使用“远程模式 / 本地模式”作为正式业务概念；统一使用“协同主数据”和“客户端本地文件能力”描述职责边界
- 不论镜头主数据当前来自何种数据源，文件扫描、预览、FFmpeg、路径映射都必须在客户端当前机器执行
- 同一账号在不同机器登录时，服务端返回的 binding 协同事实可以相同，但某台机器是否存在本地文件、是否可以预览，必须由该机器的本地扫描结果与路径映射结果决定

### 3.2 文件能力与协同主数据边界（定稿）

以下边界自本次起正式固化，后续实现不得再混淆：

1. 服务端只负责协同主数据，不负责客户端本地文件能力
2. 镜头状态、镜头主数据、成员权限、binding 元数据由服务端保存并回读
3. MA / MOV / Layout / LayoutVideo 的本地扫描、是否存在、绝对路径解析、媒体信息探测、预览地址生成，均由客户端在当前机器执行
4. 服务端返回 binding，并不等于当前机器一定可以打开或预览该文件
5. 详情页与列表页可以读取服务端返回的 binding 协同事实，但文件存在性、可播放状态、预览能力必须以客户端当前机器的本地结果为准
6. 不允许把 FFmpeg、FFprobe、路径映射、Layout 引用检查迁移到服务端
7. 后续若页面需要展示“可预览 / 不可预览 / 本机缺文件 / 缺路径映射”等状态，必须由客户端本地能力给出最终判定，不能由服务端代判

补充说明：maker 的只读权限仅针对协同写入；本机文件检查、扫描刷新、预览与 Layout 引用排查属于客户端本地能力，不应被误收口。

补充说明：账号切换后必须清空上一账号的镜头缓存，并重新按当前账号与当前激活集拉取镜头列表，不能继续沿用旧会话的本地镜头态。

补充说明：Layout 引用检查与缺失导出属于本机诊断动作，只用于当前机器上的问题排查，不得作为协同 binding 同步入口。

补充说明：当前版本文件状态的展示语义已固化为三档：

- MA / MOV 均未匹配时显示红色，并明确提示“缺失 MA 文件”“缺失 视频文件”
- 仅部分文件匹配时显示黄色，并提示“缺失 xx 文件”或“当前版本未完全匹配”
- MA / MOV 均匹配时显示绿色，并在下方折叠展示已匹配文件名

补充说明：详情页“下一步建议”中的当前版本补齐提示，必须优先展示缺失文件类型，不应仅使用笼统的“未完全匹配”文案。

补充说明：

- 当前代码实现中即使暂时保留不同数据源入口，也只属于实现层结构，不再作为正式业务概念写入后续文档或派工描述
- 业务语义统一为：服务端负责协同主数据，客户端负责全部本地文件能力

### 3.3 项目创建与成员管理

当前已经固化：

1. 制片可创建项目
2. 创建项目时可预置项目成员
3. 创建项目时成员字段主契约为：

```json
{
  "members": [
    { "userId": "...", "projectRoleCode": "maker" }
  ]
}
```

4. 项目成员管理接口为：
   - `GET /api/project-members?projectCode=...`
   - `POST /api/project-members`
   - `DELETE /api/project-members/{projectCode}/members/{userId}`
5. maker 作为项目成员时可编辑镜头
6. 非项目成员仍被拒绝
7. 创建项目时允许不预置任何制作人员；`members` 可为空或不传，由创建人先成为项目 producer
8. 创建项目 / 创建集时，如果 Excel 中的制作人员能匹配到系统账号，应自动将该账号加入当前项目成员，再进入镜头同步链路

### 3.4 第六批已固化主链路

第六批当前已经固化并可继续作为后续开发基础的主链路：

1. 制片登录后可创建项目
2. 创建项目时可同步创建首集
3. 客户端负责解析旧项目初始化 Excel
4. 客户端在本地完成初始化校验
5. 客户端在项目 / 首集创建成功后，将镜头批量同步到服务端
6. 服务端负责接收并落库首集镜头主数据
7. 客户端在镜头同步成功后，按真实镜头记录创建本地镜头文件夹
8. 页面使用真实初始化状态，不再以静态成功文案冒充初始化完成

### 3.5 第六批职责边界（已定稿）

客户端负责：

1. 解析旧 `.xls` / 旧表头 Excel
2. 生成镜头初始化数据
3. 调用镜头批量同步接口
4. 创建本地镜头文件夹
5. 展示真实初始化状态

服务端负责：

1. 创建项目
2. 创建首集
3. 返回后续初始化上下文
4. 接收镜头批量同步并落库
5. 维持成员与权限边界

服务端当前**不负责**：

1. 直接解析旧项目 `.xls`
2. 本地镜头文件夹创建

### 3.6 当前已确认稳定的运行状态

截至本次固化，以下运行状态已确认可作为后续测试基线：

1. 制片账号登录后可看到自己有权限的项目列表
2. 重启客户端 / 服务端后，项目列表仍可恢复显示，不再依赖本地临时缓存
3. 在协同模式下，项目激活会自动带出当前集上下文，不会出现“项目已激活但镜头页为空”的假空状态
4. 进入镜头页后，可按当前集读取已创建的镜头信息
5. 项目与镜头展示依赖真实 workspace / episode 上下文，不再使用静态成功文案或空数组兜底掩盖错误
6. 协同模式下镜头列表必须以服务端当前集镜头为准，不能回落展示本地历史残留镜头
7. 协同模式下进入镜头列表页时，必须先基于服务端当前集镜头触发一次本地文件匹配刷新，再合并展示本地文件状态

当前已确认的关键修复点：

- 登录后工作区刷新已改为读取仓储层统一入口
- 协同模式下项目激活会自动选择当前集或首个可用集
- 镜头页刷新链路已能拿到真实的当前集镜头数据
- 删除项目后，同一项目代码可以再次创建，不会再被“项目代码已存在”阻塞
- 协同模式下镜头页不再误读本地历史镜头残留，镜头列表必须受服务端项目 / 集权限约束

### 3.7 协同项目激活与本地工作区标识边界（已定稿）

第十一批返修后，项目 / 集激活链路正式固化以下规则：

1. 服务端项目主键语义是协同项目代码，如 `XJ3`
2. 客户端本地工作区主键语义是本地 registry / manifest 中的 `projectId`，不等于服务端项目代码
3. 任何会写入或刷新本地 SQLite 工作区的动作，必须使用本地 `projectId` / `episodeId`，不得直接拿服务端项目代码或服务端集 id 覆盖本地主键语义
4. 协同模式下创建项目时，必须先建立本地项目工作区，再执行服务端项目创建、镜头同步与本地回灌
5. 协同模式下切换项目 / 集时，必须先通过本地 `open()` 恢复本地工作区，再基于本地返回的 `projectId` / `episodeId` 执行 `applyInitialization`
6. 若本地工作区未建立成功，不允许继续伪装成“已激活项目”；必须明确返回失败
7. 后续任何派工若涉及项目激活、首集初始化、文件检查上下文刷新，都必须先检查“本地主键”和“服务端协同主键”是否被错误混用

### 3.8 文件检查协作台与镜头模块关系（已定稿）

第十一批后，文件检查协作台的产品定位正式固化如下：

1. 文件检查协作台不是独立业务域，必须服务于镜头模块
2. `ma` / `mov` / `layout` / `layoutVideo` 的匹配结果，必须进入镜头共用 binding 协同事实
3. 文件检查页、镜头列表、镜头详情页必须消费同一份 binding 回读事实，不允许各自维护独立真相源
4. Layout candidate、Layout video 绑定、批量文件检查结果一旦影响镜头可用文件，就必须回到镜头 binding 同步链路
5. 文件检查页允许展示本地扫描细节，但不能另起一套“只在文件检查页生效”的同步口径

### 3.9 初始化与镜头帧数容错规则（已定稿）

1. 初始化 Excel / 镜头同步中的 `singleFrame` 允许为 `0`
2. `0` 表示当前镜头帧数未填或暂未确认，不应因为本地校验而阻断项目创建、首集创建或本地回灌
3. 只有负数才属于非法帧数输入
4. 当镜头初始 `singleFrame = 0` 且已匹配到可解析帧数的 Layout 视频时，客户端可以使用该 Layout 视频帧数对镜头做一次性初始化
5. 这次初始化必须由客户端在本地完成真实落库，不允许只做界面展示层兜底
6. 一次性初始化成功后，客户端必须同时将 `frame_source_locked` 置为锁定状态，后续不得再被 Layout 视频自动覆盖
7. 镜头帧数一旦完成自动初始化，后续若要调整，只能由人工在镜头编辑链路中手动修改
8. 对于历史项目中“已匹配 Layout 视频但镜头帧数仍为 0”的漏初始化数据，客户端在打开项目 / 进入镜头列表 / 读取镜头详情时可以执行一次补偿初始化
9. 补偿初始化与首次初始化遵守同一规则：仅允许对 `singleFrame <= 0` 且尚未锁定的镜头生效，成功后立即锁定
10. 客户端自动初始化或补偿初始化得到的正式镜头帧数，必须继续同步到服务端，避免本地与协同主数据分叉

### 3.10 镜头制作人员语义（已定稿）

第八批开始前，镜头“制作人员”字段的业务语义正式定稿为：

1. 镜头正式制作人员必须对应**系统中的真实用户账号**，而不是只保存一段 Excel 名字文本
2. Excel 中的“制作人员 / 负责人”字段仅作为**导入匹配线索**，不是最终协同身份主键
3. 若 Excel 人名能匹配到当前项目成员中的账号，则镜头应直接绑定该账号
4. 若 Excel 人名当前无法匹配到账号，则镜头进入“未匹配”状态，但必须保留 Excel 原始姓名，不能简单丢失为纯空值
5. 后续可由 admin 创建用户、由制片将用户加入项目，再在镜头编辑中重新指定制作人员账号
6. “未匹配”是一个可回补的协同状态，不代表镜头永久无人负责
7. 镜头编辑页必须允许制片把 `unmatched` / `unassigned` 状态的镜头重新指定到当前项目成员账号
8. 镜头编辑页也必须允许清空当前制作人员，使镜头回到 `unassigned` 状态，但不得把非项目成员写入镜头负责人
9. 项目创建 / 创建集时，不应要求制片手工先把 Excel 中命中的制作人员逐个加入项目；客户端应自动把命中的系统账号并入项目成员
10. 在自动入组后，客户端必须重新按最新项目成员重算镜头制作人员匹配，避免因入组前的旧成员快照导致“未匹配”残留

因此，后续实现中不得再把“maker 纯文本字符串”视为镜头正式负责人的唯一来源。

### 3.11 协同镜头列表加载规则（已定稿）

第十一批返修后，协同模式下镜头列表入口规则正式固化如下：

1. 镜头列表的主数据基线必须来自服务端当前激活集，不允许直接以本地 SQLite 历史镜头作为协同展示结果
2. 若当前登录账号在服务端下没有可访问项目，或当前没有有效激活集，则镜头页必须显示空列表，而不是展示本地残留镜头
3. 协同模式下进入镜头页时，必须先读取服务端 `/api/episodes/{episodeId}/lenses` 的镜头集合，再以该集合为目标执行本地文件匹配刷新
4. 本地文件匹配刷新只负责补强当前机器的文件状态、Layout 状态、帧数回填、预览信息等本地能力字段，不得反向决定“有哪些镜头”
5. 镜头列表与镜头详情都必须遵守同一原则：服务端决定协同主数据范围，客户端决定当前机器文件状态与可预览状态
6. 后续若再次出现“切换账号后仍看到上一账号镜头”的问题，优先检查是否错误回退到了本地镜头列表

### 3.13 镜头列表与本地扫描合并规则（已定稿）

协同模式下镜头列表的展示规则进一步固化如下：

1. 镜头列表先从服务端当前集获取镜头主数据，再用当前机器的本地文件检查结果做补强
2. 本地扫描结果合并时，必须以镜头自身稳定标识为键进行合并，优先使用 `lensId`，不能依赖本地 / 服务端的 `episodeId` 是否一致来判断能否合并
3. 进入镜头列表页时触发的本地刷新，只是补强这批服务端镜头的文件状态，不得让本地历史镜头覆盖服务端镜头列表
4. 若本地存在与当前服务端镜头同 `lensId` 的记录，则可合并其 `currentVersionReady`、`layoutReady`、`layoutVideoReady`、`singleFrame` 等文件状态字段
5. 若本地没有同 `lensId` 的记录，则列表仍保持服务端主数据，不能因为本地上下文不一致而忽略合并

### 3.14 镜头详情预览源固化规则（已定稿）

1. 镜头详情页最终展示的预览源必须以客户端主进程返回结果为准，页面层不得自行拼接或猜测文件 URL
2. 在协同主数据场景下，production / layout 的详情播放必须优先走代理预览
3. 代理预览生成失败时，不允许静默回退到原始 `file://` 直连播放
4. 详情页若未拿到主进程明确返回的可用预览源，必须继续显示不可预览状态，而不是伪装成已可播放
5. 当前 Electron 客户端已改用 `movtools-preview://video?path=...` 自定义协议承载本地预览文件，由主进程在 `app.whenReady()` 后注册处理器，避免启动期协议注册报错
6. 当前稳定状态下，镜头详情页使用主进程返回的最终预览 URL 直接播放，不再自行回退到原始 `file://` 路径

### 3.15 审片返修与镜头返修语义固化（已定稿）

1. 审片页的“返修”是审片任务结论，对应协议动作 `reject`
2. 镜头页的“返修”是镜头主状态流转，对应状态值 `REWORK` / `返修`
3. 两者可以同时存在，但不能互相替代或共用一套状态枚举
4. 返修后若进入镜头状态流转链路，服务端必须实际写入“返修”状态并回读给列表/详情
5. 批量返修必须作用于所有选中镜头，不允许只更新第一条记录
6. 镜头返修时服务端必须同步递增并持久化 `VersionNum`，客户端只能以服务端返回值为准，不能再用本地缓存覆盖
7. 返修完成后的本地文件匹配与绑定同步必须基于服务端最新版本号执行

### 3.16 审片任务草稿保存收口（已定稿）

1. 制片创建审片任务时，导演字段必须选择系统真实用户，不再允许自由文本冒充导演主键
2. 草稿任务保存前，服务端必须校验导演用户存在且具备导演角色
3. 草稿任务创建与任务镜头写入视为同一事务单元，避免出现主表成功、子表失败后留下半写入状态
4. 若数据库结构或历史迁移未对齐，服务端必须返回可读业务错误，不得继续只暴露原始 `DbUpdateException`
5. 任务草稿失败时，客户端应将错误视作保存失败，而不是已创建成功
6. 开发环境启动阶段允许直接重建开发库；默认优先尝试用业务账号连接 PostgreSQL 维护库删除并重建 `movtools_server_dev`，若权限不足则回退到 `postgres` 维护库继续联调，避免单纯因为缺库而阻塞启动
7. 当前本机开发库口令基线固定为 `movtools / movtools_dev`，并要求角色具备 `LOGIN + CREATEDB`，同时 `movtools_server_dev.public` 必须授权给 `movtools`
8. 若再次遗失本机 PostgreSQL 管理口令，允许临时将 `C:\Program Files\PostgreSQL\16\data\pg_hba.conf` 的本机认证改为 `trust`，重启 `postgresql-x64-16` 后重置角色与权限，再恢复为 `scram-sha-256`
9. 导演审片页是导演反馈卡片的正式编辑入口；镜头页只负责反馈回看与制作人员消费，不再承担导演侧主编辑职责
10. 镜头详情页中的 `镜头反馈` 与 `导演反馈` 必须作为同级分页表达；前者由 `记录` 分页承接镜头主状态链路中的返修记录，后者由 `导演反馈` 分页承接导演内部审片反馈卡片

### 3.17 审片任务截止时间弱化（已定稿）

2. 客户端创建 / 编辑审片任务时可以不展示截止时间输入

### 3.12 制作人员账号自动入组与匹配回补（已定稿）

第十一批返修后，制作人员账号与项目成员的关系固化如下：

1. Excel 中的制作人员文本不是最终身份主键，真正的负责人必须能落到系统用户账号
2. 项目创建 / 创建集时，客户端必须先依据当前项目成员与系统用户列表尝试匹配制作人员

4. 若制作人员文本存在别名、空格、分隔符或括号修饰，应尽量做归一化匹配，不能只依赖完全字面相等
5. 对于已经创建完成但成员缺失的历史项目，后续打开项目 / 创建集时也应补做一次成员回填，避免因为项目成员不完整导致镜头制作人一直未匹配
6. 只有在系统内确实找不到可匹配账号时，才允许保留“未匹配”状态

### 3.16 角色中文展示与稳定码分层（已定稿）

6. 若用户编辑仅修改密码，不得因为角色显示值回填而二次触发角色重置

### 3.17 制作人员本地文件检查与协同写操作分离（已定稿）

2. `检查文件`、`刷新列表`、`刷新全部文件匹配`、`检查当前 Layout 引用` 属于当前机器的本地检查能力，应继续可用
3. 本地检查可以刷新镜头列表中的版本状态、Layout 状态与预览状态，但不能直接回写协同 binding 主数据
4. 协同写操作仍包括：提交、通过、返修、关闭、绑定 MA/MOV/Layout、导入、删除
5. maker 若看不到本机状态刷新，会导致版本 / Layout 状态长期停留在未匹配，因此客户端必须保留本地扫描链路

### 3.18 协同项目激活路径回读与本机盘符重定位规则（已定稿）

1. 协同项目激活时，制作人员不应再被要求手工重新配置首集镜头根目录或 layout 根目录
2. 服务端必须向有项目访问权限的客户端返回项目 / 集 roots 的真实 `absolutePath`
3. 客户端激活项目时，优先读取服务端记录的 `projectRootPath`、`lensFolderRootPath`、`layoutCheckPath`
4. 若 `lensFolderRootPath` / `layoutCheckPath` 缺失，客户端必须回退使用 `lensRoots` / `layoutRoots` 中优先级最高且启用的 `absolutePath`
5. Windows 多机协同时允许盘符不同，但盘符后的路径结构必须一致；客户端可将 `Z:\foo\bar` 自动重定位到 `D:\foo\bar`
6. 当前阶段自动重定位仅处理 Windows 盘符差异，不处理 UNC / NAS 路径改写
7. 若服务端未返回任何可用绝对路径，属于服务端协同配置缺失，不能要求制作人员在激活时临时补配根目录
8. 若服务端已返回绝对路径但当前机器仅盘符不同，客户端必须优先自动重定位而不是直接报错

补充定稿：

9. `projectRootPath`、`lensFolderRootPath`、`layoutCheckPath` 属于正式协议字段，后续不得再借用 `description` 或其他非路径字段传输这些值
10. `description` 仅保留项目 / 集说明语义，不能继续承担路径存储职责
11. 项目 roots 与集 roots 的响应结构必须同构，并稳定包含 `fileKind`、`absolutePath`、`priority`、`isEnabled`、`initExcelPath`
12. 客户端不得再把服务端 roots DTO 直接当成本地扫描 roots 类型使用，必须经过显式映射后再进入激活和文件检查链路
13. 历史数据若曾把路径写入 `description`，只允许在迁移期做一次性兼容读取；迁移完成后必须回到正式路径字段
14. 激活失败后的手动重定位入口只允许重选 `projectRootPath`，镜头根目录和 Layout 根目录必须继续沿用服务端路径并按新盘符自动重映射
15. 项目页与仪表盘页都必须遵守同一条激活失败修复规则：只允许用户重选项目根目录，不允许分别手工改镜头根目录或 Layout 根目录
14. 激活失败后的手动重定位入口只允许重选 `projectRootPath`，镜头根目录与 Layout 根目录必须继续沿用服务端路径，仅按新项目盘符自动重映射

### 3.19 第十五批正式路径字段落库与迁移基线（已定稿）

1. 服务端数据库中的 `projects` 表必须正式包含：`ProjectRootPath`、`LensFolderRootPath`、`MaCheckPath`、`MovCheckPath`、`LayoutCheckPath`
2. 服务端数据库中的 `episodes` 表必须正式包含：`LensFolderRootPath`、`LayoutCheckPath`
3. 第十五批对应迁移为 `20260511000000_S15_ProjectEpisodeFormalPaths`，后续环境验收必须先确认该迁移已落库
4. 若数据库尚未应用该迁移，producer / maker 登录后读取项目列表时会因为缺失正式列而直接失败，这属于结构阻塞，不属于业务数据问题
5. `clear-project-data.bat` 仅负责清空项目相关业务数据，不承担数据库结构升级职责；清数据不能替代 migration
6. 服务端启动时必须先尝试执行数据库迁移；若迁移失败，日志中必须能明确暴露数据库初始化失败，不能再把“结构未升级”误判为“空项目可忽略”
7. 历史项目若曾把项目根路径误写入 `description`，迁移后允许通过兼容回填到 `ProjectRootPath`，但正式写入链路不得继续写回 `description`
8. 第十五批完成后的数据库基线应与客户端第十五批正式协议保持一致：项目 / 集正式路径字段可稳定回读，roots 同构结构可稳定回读，`fileKind` 不得缺失

### 3.20 新设备首次激活与本地 bootstrap 边界（已定稿）

1. 新设备首次激活协同项目时，若本地项目工作区不存在，客户端允许自动创建本地工作区骨架并执行 bootstrap
2. 自动创建范围仅限本地工作区骨架，例如本地 registry、manifest、SQLite 与必要工作区目录；不得把这些动作等同于伪造生产素材目录
3. `projectRootPath` 可作为本地工作区承载目录，在首次激活时允许自动创建
4. `lensFolderRootPath` 与 `layoutCheckPath` 属于文件检查上下文，不得由 `projectRootPath` 直接冒充或兜底替代
5. 远端激活时，客户端必须先基于服务端正式路径字段与 roots 明细解析出本机最终路径，再执行本地工作区创建
6. 项目根目录、镜头根目录、Layout 根目录属于不同语义；后续实现不得再把项目根目录错误回退为镜头根目录或 Layout 根目录
7. 若镜头根目录与 Layout 根目录在本机解析后重复、互相包含，客户端必须阻断激活并明确报错，不能继续 bootstrap
8. 若本地缺失的是工作区骨架，允许自动初始化；若缺失的是业务要求必须真实存在的素材目录，应明确提示素材目录缺失，而不是静默代建一套伪目录
9. 新设备首次激活失败时，错误信息必须优先暴露真实原因，例如服务端路径配置缺失、本机路径解析冲突、本地工作区未初始化或素材目录缺失，不能统一模糊成“初始化失败”
10. “空本地工作区的新设备首次激活”属于标准验收场景，producer 与 maker 都必须纳入实际验收
11. 不同设备的项目盘符允许不同，例如制片机器可使用 `X:`、制作人员机器可使用 `R:`；盘符不一致本身不应阻断协同激活
12. 服务端保存的是协同标准路径；客户端激活时必须基于本机目录情况做盘符重定位，而不是要求所有机器盘符完全一致
13. Windows 环境下，若服务端路径在本机不存在，但盘符后的路径结构一致，客户端应优先尝试自动将 `X:\foo\bar` 重定位到 `R:\foo\bar`
14. 盘符重定位必须分别作用于 `projectRootPath`、`lensFolderRootPath`、`layoutCheckPath` 以及对应 roots 明细，不能只处理项目根目录
15. 自动重定位的依据是“盘符后的路径结构一致且本机目标路径真实存在”；若本机不存在对应目录，客户端必须明确提示缺失目录，而不是盲目猜测或静默改写
16. 当前阶段自动重定位仅处理 Windows 本地盘符差异，不要求自动支持 UNC、NAS、目录改名或路径结构变化场景

1. 客户端左侧菜单统一称为“左侧主导航”
2. 顶层页面名称以导航标签为准，例如“仪表盘”“项目”“文件检查”“提取”“镜头”“审片”“工作台”“日志”“设置”“用户”
3. 页面内部的大区块名称以页面标题和分区标题为准，不再混用“那个页面”“左边那个模块”“镜头那块”之类口头描述
4. 制片账号的项目成员维护入口正式位于“项目”页中的“项目成员管理（制片）”，不再把它与“用户”页混称
5. 路径映射不作为一级导航名词，正式名称为“路径映射设置”，入口位于“设置”页中的高级功能区域
6. 各角色可见菜单与功能说明统一以 `平台功能与UI统一命名清单.md` 为准

### 3.23 导演内部审片二级状态基线（已定稿）

第十七批开始前，导演内部审片能力的状态语义正式固化如下：

1. 镜头状态正式拆分为两层：镜头主状态与镜头二级状态
2. 镜头主状态继续表示对客户的正式业务状态，不承担导演内部审片语义
3. 镜头二级状态专用于内部导演审片流程，不得与镜头主状态混用
4. 镜头二级状态稳定集合为：`未进入审片`、`待提审`、`审片中`、`待处理反馈`、`已按反馈修改`、`内部通过`
5. 协议稳定码分别为：`NOT_IN_REVIEW`、`READY_FOR_REVIEW`、`IN_DIRECTOR_REVIEW`、`PENDING_FEEDBACK_FIX`、`FIX_UPDATED`、`DIRECTOR_APPROVED`
6. 导演内部审片中的“返修 / 修改”语义，正式落在镜头二级状态与审片反馈链路中，不直接改写镜头主状态 `REWORK`

### 3.24 镜头二级状态与版本联动规则（已定稿）

1. 镜头二级状态只描述当前镜头版本的内部审片结果
2. 不单独维护“内部通过版本号”业务概念；`内部通过` 的含义是导演已手动确认该镜头当前处于通过态
3. 当前镜头 `VersionNum` 升级后，不以“自动清空二级状态”作为默认策略，而是按本轮反馈是否已确认处理来继续推进
4. 当前镜头处于 `待处理反馈` 时，新版本产生本身不自动把状态重置为 `未进入审片`
5. 制作人员确认“本轮反馈已处理完成”后，才允许把状态推进到 `已按反馈修改`
6. 已处于 `内部通过` 的镜头不会因为草稿任务或普通任务装配自动回退到 `审片中`
7. 导演反馈记录仍应保留对应版本号用于追溯，但二级状态本身只绑定当前版本理解

### 3.25 镜头二级状态流转职责边界（已定稿）

1. 制片负责镜头二级状态的组织性推进，包括：
   - `未进入审片 -> 待提审`
   - `已按反馈修改 -> 待提审`
2. `待提审 -> 审片中` 的主驱动链路不再定义为“制片手工点镜头状态”，而是“制片提交正式审片任务后由系统推进”
3. 导演通过审片动作驱动镜头二级状态，包括：
   - 保存反馈后进入 `待处理反馈`
   - 确认当前版本通过后进入 `内部通过`
4. 制作人员只负责反馈处理完成后的确认动作：`待处理反馈 -> 已按反馈修改`
5. 制作人员不得把镜头直接改为 `内部通过`
6. 导演不负责把镜头推进到 `待提审`
7. 制片不应手工伪造 `待处理反馈`，该状态优先由导演反馈动作驱动
8. 已处于 `内部通过` 的镜头，不会因为草稿任务或普通任务装配自动回退到 `审片中`；只有导演新的反馈动作或明确人工调整，才改变该状态

### 3.25.1 制片内部提审派工主链路（已定稿）

1. 制片内部提审的真相源从本阶段起正式收口为“审片任务”，不再以镜头单独改成 `审片中` 视为已派工
2. `待提审` 只表示镜头具备入任务条件，不表示导演已经收到任务
3. 导演能看到审片任务的前提是服务端已经存在正式 `ReviewTask`
4. 制片派工的标准动作是：创建任务草稿、添加多个镜头、调整顺序、指定提审版本、选择导演、提交任务
5. 一个审片任务可以包含多个镜头，导演应按制片编排顺序串联审片
6. 任务提交后，任务内处于 `待提审` / `已按反馈修改` 的镜头由系统统一推进到 `审片中` / `IN_DIRECTOR_REVIEW`
7. 制片镜头页中的二级状态按钮仍可用于“标记待提审 / 重新提审”等组织性动作，但不再等价于正式派工

### 3.25.2 审片任务与镜头二级状态关系（已定稿）

1. 审片任务状态、镜头二级状态、镜头主状态是三套不同对象，不得混用
2. 审片任务状态描述的是“一轮任务”的组织与执行进度
3. 镜头二级状态描述的是“当前镜头版本”的内部审片结果
4. 镜头主状态描述的是面对客户的正式业务状态
5. 导演反馈回流后，镜头进入 `待处理反馈`，但任务本身仍可保持 `审阅中`
6. 镜头进入 `内部通过` 只表示该镜头当前版本内部通过，不自动等于整个任务已完成
7. 只有当任务内镜头整体达到本轮结束条件时，任务才允许进入 `已完成` 或 `已关闭`

### 3.25.3 审片任务镜头分类规则（已定稿）

1. 审片任务内镜头从本阶段起固定分为两类：`正式审片镜头` 与 `上下文陪审镜头`
2. `正式审片镜头` 指当前已具备真实提审版本、需要导演给出反馈或通过结论的镜头
3. `上下文陪审镜头` 指当前尚未完成正式版本、但因前后衔接、节奏判断或镜头关系需要被带入任务播放链路的镜头
4. 制片在任务装配过程中新增镜头时，默认参与类型必须遵守素材匹配规则：当前版本视频已匹配时默认 `正式审片镜头`；若当前任务需要依赖 `layout` 视频做连播补位，则默认 `上下文陪审镜头`
5. `上下文陪审镜头` 可以使用已匹配的 `layout` 或其他参考视频参与连播，但不因此获得“已送审”或“待导演处理”的业务语义
6. `进任务播放列表` 与 `进入镜头二级状态机` 是两个不同事实；只有 `正式审片镜头` 参与内部审片状态推进
7. `上下文陪审镜头` 在任务中默认保持原镜头主状态与原镜头二级状态不变，通常仍为制作中 / 未进入审片，不因任务提交自动改成 `审片中`
8. `上下文陪审镜头` 不产生导演反馈待办，不要求导演给出 `内部通过` 或 `待处理反馈` 结论，也不计入“当前任务尚未处理完毕”的阻塞条件

### 3.25.4 基于镜头分类的任务提交与导演操作规则（已定稿）

1. 制片创建任务草稿时，可同时加入 `正式审片镜头` 与 `上下文陪审镜头`，两者都参与任务排序与连播
2. 制片正式提交任务后，系统只对任务内 `正式审片镜头` 自动推进二级状态：`待提审 -> 审片中`、`已按反馈修改 -> 审片中`
3. `上下文陪审镜头` 在任务提交、任务开始、任务关闭过程中都不自动推进到 `审片中`
4. 导演工作台对 `正式审片镜头` 展示正常操作能力：提交反馈、要求返修、镜头通过
5. 导演工作台对 `上下文陪审镜头` 只提供观看与顺序衔接能力，不展示“镜头通过”“返修”“提交反馈”等会写入正式审片结论的操作
6. 任务级统计、完成判定、待处理数量、通过数量只统计 `正式审片镜头`，不得把 `上下文陪审镜头` 误计为未完成审片对象
7. 若某个 `上下文陪审镜头` 后续已产出正式版本并需要纳入本轮正式审片，应先由制片把其任务内分类切换为 `正式审片镜头`，再参与后续二级状态推进与导演结论动作

### 3.25.5 审片任务完成后的收尾规则（已定稿）

1. 导演完成本轮审片后，任务应先进入 `已完成` / `completed`，而不是直接物理删除
2. `已完成` 任务的业务含义是“本轮派工已结束，默认不再作为导演待办继续展示”
3. 当前阶段推荐的收尾策略是“完成后从默认任务列表隐藏或归档”，而不是立即做数据库物理删除
4. 不推荐直接物理删除的原因包括：任务反馈卡片、绘制遮罩、任务镜头顺序、最近任务回挂、审片统计追溯都仍可能依赖该任务事实
5. 若产品层确实需要“删除”体验，当前阶段应优先实现为“仅删除默认列表中的可见性”或“归档后不再默认显示”，不应先实现不可恢复的硬删除
6. 只有在后续明确补齐历史追溯、反馈归档、镜头回挂迁移、审计保留策略后，才考虑增加真正的物理删除能力
7. 若本项目明确采用“内部评审不保留历史反馈”的业务口径，则制片执行 `关闭任务` 时可作为正式清场动作：物理删除该任务下全部导演反馈与绘制事实
8. 上述“关闭并清空”动作的执行权限属于制片账号，不属于导演账号
9. 制片关闭任务并清空反馈后，任务内 `正式审片镜头` 的二级状态统一回退到 `READY_FOR_REVIEW`，供后续重新组织下一轮审片
10. `上下文陪审镜头` 因不参与正式审片状态机，关闭任务时不要求额外状态回退

### 3.26 内部审片与正式提交关系（已定稿）

1. 当前镜头二级状态不是 `内部通过` / `DIRECTOR_APPROVED` 时，不允许执行正式提交
2. 镜头能否正式提交，必须同时满足镜头主状态链路要求与二级状态通过要求
3. “内部未通过不可提交”必须由服务端做最终拦截，客户端只负责前置提示和按钮收口
4. 当前镜头一旦因导演反馈进入 `待处理反馈` 或由制作人员推进到 `已按反馈修改`，正式提交能力必须继续保持锁定，直到导演再次手动确认通过

### 3.27 导演内部审片任务与反馈产物基线（已定稿）

1. 导演端以“审片任务”为核心工作对象，而不是零散镜头集合
2. 审片任务由制片创建、排序、指定提审版本并提交给导演
3. 导演在审片工作台中直接对视频画面定帧标注并填写反馈说明
4. 导演反馈的正式提交主体是“反馈卡片 + 镜头级导演绘制数据”的组合；文字与绘制保持弱绑定，任一存在有效修改都可构成正式提交
5. 每条反馈卡片都必须归属到具体镜头、具体任务、具体版本、具体帧
6. 导演反馈允许先以本地临时草稿方式保存；点击“提交反馈”后，再将本轮已保存卡片上传到服务器
7. 导演绘制内容在底层只按单帧存储；若视觉上需要持续到后续帧，由客户端按继承链解释，不新增区间型底层模型
8. 制作人员在镜头详情页中消费导演反馈卡片，并可按需打开导演反馈视频遮罩，不直接进入导演审片工作台
8. 反馈卡片落库时必须持久化版本号，`review_comments.VersionNum` 视为正式基线字段，避免导演登录或镜头反馈查询时出现版本字段缺失的结构错误
9. 允许只提交文字反馈，也允许只提交绘制反馈；但若镜头既没有有效文字反馈也没有有效绘制内容，则不能正式提交

### 3.32 反馈图片持久化与回读基线（已定稿）

1. 一级返修图片与导演反馈图片都必须进入服务端正式持久化流程
2. 客户端不得把本地绝对路径当作正式协同图片地址
3. 本地草稿态可以先显示缩略图，但提交后必须切换为可回读的服务端正式地址
4. 镜头详情与审片工作台中的图片展示必须共用同一套图片地址解析规则
5. 服务端返回的 `/uploads/...` 静态资源地址必须视为正式图片地址，不得再误转为 `file://`
6. 任何“仅存本地路径、不落服务端附件”的实现都不再视为完成
7. 导演反馈配图固定由“当前视频帧 + 导演当前绘制图层”合成，不再提供手动选图入口

### 3.35 镜头详情反馈分页与图片闭环（已定稿）

1. `记录` 分页固定承载一级返修记录、返修说明、返修图片与生命周期事件
2. `导演反馈` 分页固定承载导演内部审片反馈卡片、标注图、原图、缩略图与版本信息
3. 镜头详情页不得再把导演反馈混写进 `记录` 分页，也不得把一级返修记录改挂到导演反馈分页
4. 反馈编辑态必须先展示缩略图；提交成功后必须回读服务端正式图片地址
5. 重新登录、切换设备或重开镜头详情后，反馈图片仍必须保持可预览、可放大、可追溯
6. `导演反馈` 分页中的视频预览区必须预留“显示导演反馈遮罩”开关；开启后，视频按已提交绘制数据叠加播放遮罩
7. 遮罩开关默认消费正式已提交绘制数据，不显示导演工作台中的本地未提交绘制

### 3.36 本轮文档固化说明（已定稿）

本轮已完成以下文档收口：

1. `镜头详情页设计文档.md`：明确 `记录` / `导演反馈` 并列分页与图片回读边界
2. `导演账号功能设计文档.md`：明确反馈编辑态缩略图、回看态正式图片与放大预览
3. `项目固化文档.md`：新增反馈图片持久化与镜头详情分页闭环定稿
4. `通信协议文档.md`：明确反馈图片字段的草稿语义与正式回读语义

本轮最终口径：本地草稿可以短暂使用本机路径预览；一旦提交并回读，图片地址必须切回服务端可访问资源，且镜头详情中的 `记录` 与 `导演反馈` 必须长期保持并列阅读模型。

### 3.27.0.1 导演审片工作台交互基线（已定稿）

1. 导演工作台的视频体验必须是“任务级连播”，而不是手工逐镜头点开多个独立视频
2. 同一任务内镜头应按制片排定顺序连续播放，目标体验是导演感觉自己在看一条完整视频
3. 随播放推进，镜头队列标签必须自动激活当前播放镜头，并可回跳到任意镜头
4. 导演可在预览窗口直接暂停；只有暂停态才允许在画面上进行绘制标注
5. 暂停后绘制工具必须继续可见；播放器底部进度条在播放态与暂停态都必须可见
6. 画面绘制本身不会自动生成反馈卡片；只有在下方反馈编辑区生成文字反馈后，才形成可提交反馈列表
7. 每个反馈点卡片至少包含：文字反馈、所属镜头、所属任务、所属版本、帧号或时间码；如存在绘制，则附带原始帧图、标注图与标注原始数据
8. “保存为草稿”只在当前镜头反馈列表中生成本地草稿，不上传服务器
9. “提交反馈”动作只负责把当前镜头下已暂存的反馈列表与镜头级绘制数据上传服务器，不自动改变任务最终结论
10. 导演对任务的“通过 / 返修 / 关闭”仍属于独立任务动作，不与“提交反馈”按钮合并
11. 仅当当前镜头反馈列表为空且不存在待处理反馈时，才允许点击 `镜头通过`
12. 若当前镜头存在绘制内容但尚未形成任何文字反馈，只要该绘制本身构成有效修改，仍允许正式提交；只有文字与绘制都为空时才应阻止提交
13. 当前提交反馈与镜头通过的接口边界已固定：`提交反馈` 不收状态字段，服务端固定推进 `PENDING_FEEDBACK_FIX`；`镜头通过` 走独立通过接口，仅处理通过状态
14. 右侧 `草稿与已提交反馈` 本质上是“当前镜头反馈列表”；同一帧在同一镜头下只允许保留一条反馈项，再次编辑时应视为更新该项，而不是新增重复项
15. 已提交反馈在导演工作台中仍可再次编辑或删除；编辑后先形成本地待提交修改，删除后形成本地待提交删除
16. 只要当前镜头存在新增、修改、删除中的任一待提交变化，或存在仅绘制修改，`提交反馈` 按钮就必须可点击
17. 当前镜头反馈列表应放在播放器下方区域，不得挤占视频预览区与绘制操作区
18. 当前镜头反馈列表的正式服务端数据源必须严格按镜头 id 隔离；同一任务内其他镜头的正式反馈不得出现在当前镜头列表中
19. 客户端在渲染导演工作台反馈列表前，必须再按 `lensId`（必要时结合 `taskShotId`）执行一次镜头级过滤，作为防御性兜底

### 3.27.0.2 导演绘制遮罩与本地保存基线（已定稿）

1. 导演在审片工作台产生的绘制数据作用域属于当前镜头，而不是单条反馈卡片
2. 绘制数据底层最小单位是单帧绘制记录；同一帧允许多个绘制对象，共同组成该帧绘制集合
3. 绘制与文字反馈保持弱绑定：允许文字反馈无绘制，也允许仅凭有效绘制完成正式提交；但不允许空白文字加空白绘制构成正式提交
4. 导演在某一帧完成绘制后，若后续帧没有新绘制内容，则显示层继续沿用最近一次有效绘制
5. `清屏` 是当前帧动作，会从当前帧开始打断前序帧的绘制继承内容
6. 清屏后直到出现新的绘制帧之前，后续帧保持无绘制显示
7. 若导演回到历史帧修改旧绘制，则该帧之后直到下一绘制帧或清屏帧之前的显示结果必须自动更新
8. 绘制内容必须保存为本地数据；切镜头、刷新页面、关闭页面后都必须能恢复
9. 绘制内容只有在导演手动清屏或完成正式提交后，才允许被清理或替换为新的正式状态
10. 导演点击 `提交反馈` 时，绘制数据按当前镜头维度一次性提交该镜头当前轮次全部本地绘制数据，而不是只提交本次增量
11. 镜头详情页中的导演反馈遮罩只显示当前轮正式已提交的绘制内容，不显示本地未提交绘制
12. 当前轮反馈提交成功后，导演工作台中的本地反馈内容与本地绘制内容应清空，进入下一轮空白编辑态
13. 当前阶段 `AnnotationDataJson` 仍作为绘制聚合载荷使用，属于第三阶段待收口项；后续需要拆出独立绘制协议后再统一收口该字段
14. 同一轮提交的多条反馈卡片必须共享同一个 `feedbackRoundId`，镜头详情与回读接口只允许把该轮作为正式事实展示
15. `CLEAR` 帧在正式回读中必须稳定保持为清屏锚点，不允许被补成伪对象或丢失
16. 提交反馈不会改变任务最终结论；任务结论仍由独立通过 / 返修 / 关闭动作决定
17. 反馈回读接口统一以 `content` 作为正文事实字段，`commentText` 仅保留请求层命名；若 `frameNumber` 为空但 `timecode` 有值，视为历史兼容数据，客户端需允许降级展示
18. 当前帧若存在已提交绘制，则该帧的视频预览应优先显示该帧正式绘制；导演在该帧继续编辑后，本地最新绘制应实时覆盖预览，并在提交时作为该帧后的最新控制点写回

### 3.27.1 审片任务派工与导演入口基线（已定稿）

1. 导演的主工作入口必须是“仪表盘 -> 审片任务 -> 审片工作台”
2. 导演仪表盘中的任务列表必须来自正式审片任务，而不是根据镜头二级状态临时推导
3. 导演审片工作台必须按任务镜头队列组织，不得要求导演自行拼接镜头顺序
4. 从镜头详情定位任务属于辅助回跳能力，不得替代正式任务入口
5. 若只把镜头状态改为 `审片中` 但没有正式任务，则视为业务断链，不符合当前基线

### 3.28 第十七批产品范围补充（已定稿）

第十七批起，平台正式进入“制片 - 导演 - 制作人员”三角色内部审片闭环阶段，范围补充如下：

1. 制片可管理镜头二级状态并组织导演审片任务
2. 导演可在审片工作台联播、逐帧、画面标注、临时保存反馈卡片、提交反馈卡片与确认内部通过
3. 制作人员可在镜头列表/详情中查看二级状态，并执行“已按反馈修改”动作
4. 镜头列表、镜头详情、审片任务、导演反馈、正式提交权限都必须消费同一套二级状态事实
5. `通信协议文档.md` 中的镜头二级状态字段、状态写接口、审片任务接口、反馈卡片接口，从本批次起视为下一阶段实现基线

### 3.29 制片审片任务管理页基线（已定稿）

1. 制片端必须具备独立的审片任务管理能力，不能只停留在镜头列表点状态
2. 制片任务管理页至少支持：创建草稿、编辑任务、添加镜头、移除镜头、调整顺序、指定提审版本、提交任务、关闭任务
3. 制片镜头页承担“筛选与整理镜头”职责，任务管理页承担“正式派工”职责
4. 制片需要查看任务级进度，包括镜头数量、已反馈数量、内部通过数量、最近更新时间

### 3.30 当前内部审片闭环统一口径（已定稿）

1. 制作人员产出新版本后，先完成文件同步与版本回写，不因“新版本已产生”自动重置二级状态
2. 制片完成预检查并确认当前版本可进入内部审片后，推进到 `待提审`
3. 制片把镜头加入草稿任务时，镜头保持原二级状态
4. 制片正式提交任务后，处于 `待提审` / `已按反馈修改` 的镜头进入 `审片中`
5. 导演保存反馈后，镜头进入 `待处理反馈`
6. 制作人员处理并确认后，镜头进入 `已按反馈修改`
7. 制片决定是否重新纳入下一轮任务，并再次手动推进到 `待提审`
8. 导演确认当前版本通过后，镜头进入 `内部通过`
9. 已通过镜头保持 `内部通过`，除非导演后续又提交新的反馈
10. 当前镜头二级状态不是 `内部通过` 时，正式提交仍必须被拦截
11. 导演点击 `提交反馈` 时，同步上传的是当前镜头下的反馈列表、绘制遮罩数据与镜头状态
12. 导演点击 `镜头通过` 时，只上传镜头通过状态，不上传空反馈卡片

### 3.30.1 草稿任务与已通过镜头补充规则（已定稿）

1. 草稿任务只表示任务装配，不直接改写镜头二级状态
2. 草稿阶段镜头保持原状态，例如 `待提审`、`已按反馈修改`、`内部通过`
3. 正式提交任务时，只推动处于 `待提审` / `已按反馈修改` 的镜头进入 `审片中`
4. 已处于 `内部通过` 的镜头不会因为加入草稿任务、再次装配任务或普通任务提交而自动回退到 `审片中`
5. 已通过镜头只有在导演后续再次提交新反馈时，才重新进入 `待处理反馈`

### 3.31 第十七批后审片接口分层基线（已定稿）

当前内部审片链路的接口分层正式固化如下：

1. 导演主链路继续围绕正式 `ReviewTask`、评论、反馈卡片展开
2. 制片任务装配链路单独围绕草稿任务、制片任务列表、任务详情投影、镜头增删排序展开
3. `review-tasks` 的底层实体可以复用，但制片页与导演页允许消费不同 projection
4. 历史兼容入口如 `/api/reviews/*`、`/api/review-tasks/tasks/*` 不再作为新增功能的正式基线
5. 当前正式基线路由以 `通信协议文档.md` 第 6 章记载为准

### 3.32 第十七批后状态投影基线（已定稿）

为保证导演页、制片页、数据库状态三者口径一致，当前状态投影规则正式固化如下：

1. 服务端真实任务状态集合为：`draft`、`ready`、`pending`、`in-review`、`completed`、`closed`
2. 制片页展示文案 `待提交` 是 `ready` 的页面投影，不新增独立持久化状态
3. 导演页展示结果 `approved` 对应服务端 `completed`
4. 导演页展示结果 `rejected` 对应服务端 `closed`
5. 后续若继续沿用页面投影名，必须同步写入 `通信协议文档.md`，禁止只在代码里隐式映射
