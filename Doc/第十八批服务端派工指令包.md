## 2. 本批次目标

这一批完成后，服务端必须具备：

1. 新版本产出后，镜头若处于 `PENDING_FEEDBACK_FIX`，不再被自动重置为 `NOT_IN_REVIEW`
2. 制作人员只有在明确确认处理完成时，才把镜头推进到 `FIX_UPDATED`
3. 草稿任务阶段不改写镜头二级状态
4. 正式提交任务时，只把 `READY_FOR_REVIEW` / `FIX_UPDATED` 的镜头推进到 `IN_DIRECTOR_REVIEW`
5. 已处于 `DIRECTOR_APPROVED` 的镜头，在草稿装配和普通任务提交时保持通过态
6. 只有导演新的反馈动作，才把已通过镜头重新拉回 `PENDING_FEEDBACK_FIX`
7. 服务端接口、状态推进、校验逻辑全部与当前文档口径一致

这一批不做：

1. 不迁移客户端本地文件扫描职责
2. 不重构整套 review domain 模型为另一套新命名
3. 不允许继续保留“版本一变化就自动重置为 `NOT_IN_REVIEW`”的旧逻辑
4. 不允许普通任务提交把所有镜头一刀切推进到 `IN_DIRECTOR_REVIEW`

---

## 3. 当前代码状态（按现状理解）

当前服务端已具备：

1. 镜头二级状态字段与基础状态流转能力
2. 制片任务投影接口与导演任务接口并存的双投影结构
3. 审片任务、任务镜头、评论、反馈卡片等主对象的基础实现

当前仍存在的主要风险点：

1. 版本升级后的旧实现或残余逻辑，可能仍默认自动重置二级状态
2. 任务提交时的镜头状态推进规则可能仍过宽，没有严格限制在 `READY_FOR_REVIEW` / `FIX_UPDATED`
3. 已通过镜头在再次装配任务时，可能存在被误推进到 `IN_DIRECTOR_REVIEW` 的风险
4. 文档口径已经更新，但服务层、DTO 投影、校验逻辑未必完全同步

---

## 4. 已固化业务边界（必须遵守）

1. 当前镜头处于 `PENDING_FEEDBACK_FIX` 时，新版本产出本身不自动重置为 `NOT_IN_REVIEW`
2. 制作人员确认处理完成后，才执行 `PENDING_FEEDBACK_FIX -> FIX_UPDATED`
3. 草稿任务只表示任务装配动作，不直接改写镜头二级状态
4. 正式提交任务时，只推动 `READY_FOR_REVIEW` / `FIX_UPDATED` 的镜头进入 `IN_DIRECTOR_REVIEW`
5. 已处于 `DIRECTOR_APPROVED` 的镜头，不因草稿任务或普通任务提交自动进入 `IN_DIRECTOR_REVIEW`
6. 已通过镜头只有在导演后续再次提交新反馈时，才重新进入 `PENDING_FEEDBACK_FIX`
7. 当前二级状态不是 `DIRECTOR_APPROVED` 时，正式提交仍必须被服务端拒绝
