# 审片功能二次迭代前端派工指令包

## 1. 文档用途

本文件用于把“审片功能二次迭代”的客户端工作直接派发给其他 AI 或开发执行。

目标目录：`D:\project\MovtoolsEx\movtools-client\`

参考文档：

- `D:\project\MovtoolsEx\Doc\项目固化文档.md`
- `D:\project\MovtoolsEx\Doc\通信协议文档.md`
- `D:\project\MovtoolsEx\Doc\审片功能二次迭代后端派工指令包.md`

---

## 2. 本阶段目标

修复导演工作台对 `context` 陪审镜头的误识别问题，并把导演 `审片完成` 与制片 `关闭任务` 的前端调用彻底分开。

本阶段完成后必须满足：

1. 任务详情中的 `context` 镜头会被正确识别
2. `context` 镜头只可播放，不可通过、不可返修、不可编辑正式反馈
3. 导演工作台的 `审片完成` 调用正式 `complete` 接口
4. 制片任务页的 `关闭任务` 继续调用 `close`
5. 任务完成按钮、提示文案、统计口径全部只统计 `review` 镜头

---

## 3. 阶段拆分

### 阶段 1：仓储字段映射收口

1. 修正任务详情 shot 字段映射
2. 对 `reviewParticipationMode` / `participationMode` 做兼容读取
3. 确保播放列表、任务详情、当前镜头上下文使用同一参与类型真相源

### 阶段 2：导演工作台操作能力收口

1. `context` 镜头隐藏正式操作
2. `context` 镜头禁止进入正式反馈编辑链路
3. `layout` 播放提示与 `context` 身份解耦

### 阶段 3：任务动作与完成逻辑收口

1. 导演 `审片完成` 走 `complete`
2. 完成判定只看正式镜头
3. 已完成和已关闭的页面表现保持既有归档语义

### 阶段 4：制片页面与交互文案校对

1. 保持制片 `close` 不变
2. 检查任务详情、队列标识、按钮 title、底部说明文案
3. 确保用户能看懂 `context` 与 `layout` 的区别

## 5. 必做事项

1. 修正任务详情 shot 映射，确保参与类型不会因为字段名差异而回落成默认 `review`
2. 确保以下逻辑全部以 `reviewParticipationMode === 'context'` 为判断依据，而不是以 `layout` 为判断依据：
   - 是否显示 `镜头通过`
   - 是否显示 `重新评审`
   - 是否允许提交反馈
   - 是否允许编辑反馈和标注
   - 是否计入任务完成条件
3. 导演工作台 `handleCompleteTask` 改为调用正式 `complete` 仓储接口；不得再复用 `closeTask`
4. 若仓储层缺少 `completeTask`，补齐正式方法、类型和调用链
5. 保持制片任务页 `关闭任务` 继续走 producer close，不要被导演 complete 改坏
6. 页面中所有“当前素材来源：Layout 补位”文案只表达播放源，不得暗示正式审片能力
7. 队列、角标、底部提示、禁用原因要明确区分：
   - `正式审片镜头`
   - `上下文陪审镜头`
   - `layout` 补位视频
8. 自测 mixed task：5 个正式镜头 + 1 个 context layout 镜头场景

---

## 6. 禁止事项

1. 不要用“当前是 layout 视频”替代“当前是 context 镜头”
2. 不要只修按钮显示，不修调用链和完成判断
3. 不要改坏制片侧的 `close` 流程
4. 不要在没有协议依据的情况下临时新增一套页面状态名
5. 不要把导演完成任务重新接回旧的 `/approve` 或 `/close` 捷径

---

## 7. 验收标准

1. `context` 镜头在导演工作台只读
2. `context` 镜头不会显示 `镜头通过`、`重新评审`、`提交反馈`
3. `审片完成` 按钮只受正式镜头约束
4. 导演完成任务后走 `complete`，制片关闭任务仍走 `close`
5. `npm run typecheck` 与 `npm run build` 通过
6. 相关文案和禁用提示与协议一致
