## 4. 历史脏数据排查与修复方案

### 4.1 排查口径

历史 `ParticipationMode` 脏数据只做“候选识别”，不允许把 `layout` 或“当前播放器来源”为唯一真值。

优先排查以下候选：

1. `review_task_shots.ParticipationMode = 'review'`，但该镜头在历史任务中表现为只读陪审：
   - 任务中没有任何正式反馈写入
   - 镜头没有触发 `DIRECTOR_APPROVED` / `PENDING_FEEDBACK_FIX` 等正式导演状态
   - 任务内同时存在至少一个明确正式镜头，且该镜头已形成正常审片闭环
2. `review_task_shots.ParticipationMode` 缺失、空字符串或旧字段兼容数据异常
3. 任务流转记录显示 mixed task 已完成，但末尾陪审镜头长期阻塞 `complete`

建议先导出候选集，再人工核对：

```sql
select
  t.Id as TaskId,
  t.Status as TaskStatus,
  ts.Id as TaskShotId,
  ts.LensId,
  ts.ParticipationMode,
  ts.SortOrder,
  l.LensCode,
  l.InternalReviewStatusCode,
  coalesce(ff.FeedbackCount, 0) as FormalFeedbackCount
from ReviewTaskShots ts
join ReviewTasks t on t.Id = ts.TaskId
join Lenses l on l.Id = ts.LensId
left join (
  select ReviewTaskId, TaskShotId, count(*) as FeedbackCount
  from ReviewFeedbacks
  group by ReviewTaskId, TaskShotId
) ff on ff.ReviewTaskId = ts.TaskId and ff.TaskShotId = ts.Id
where ts.ParticipationMode in ('review', 'context')
order by t.CreatedAtUtc desc, t.Id, ts.SortOrder;
```

### 4.2 可自动修与需人工核对

可自动修的最低条件：

1. 任务创建时间落在已知脏数据窗口
2. 镜头为任务末尾陪审镜头，且没有任何正式反馈事实
3. 任务内其他正式镜头均已有明确反馈或内部通过事实
4. 镜头当前播放事实与任务内历史备注均指向上下文陪审，而非正式审片

需要人工核对的情况：

1. `layout` 补位镜头与 `context` 陪审镜头语义重叠，但任务内又存在正式反馈
2. 同一任务里多个镜头都没有反馈事实，无法仅凭播放源判断
3. 历史镜头已被二次修改，无法从单一字段还原真实业务语义

### 4.3 修复步骤

1. 先导出候选任务与任务镜头备份 CSV / SQL 快照
2. 在测试库先执行修复脚本，验证任务详情、完成统计与反馈回读不回退
3. 对可自动修的记录，仅修正明确属于陪审的镜头参与类型为 `context`
4. 对无法自动确认的记录，保留为 `review` 并进入人工核对清单
5. 修复后复跑 6 镜头 mixed task 回归，确认 `complete` / `close` / `context` 只读边界全部成立

## 5. 回归矩阵与联调验收

### 5.1 6 镜头 mixed task 真实复现场景

| 场景 | 输入条件 | 预期结果 |
| --- | --- | --- |
| S1 | 6 镜头任务：1 为正式镜头且已提交反馈，2/3/4/5 已内部通过，6 为 `context` | 导演可以执行 `complete` |
| S2 | 在 S1 基础上把 6 号镜头改成 `review` | 导演不可完成任务，必须显示正式镜头未完成原因 |
| S3 | 导演切到 `context` 镜头 | 反馈区、通过按钮、重新评审按钮全部隐藏，只显示播放与衔接信息 |
| S4 | 制片账号打开任务详情 | 只读查看任务详情，不误入导演审片页 |
| S5 | 制片关闭已完成任务 | 任务进入 `closed`，导演侧列表移除，历史详情仍可追溯 |

### 5.2 联调矩阵

| 链路 | 验证点 | 通过标准 |
| --- | --- | --- |
| 导演 complete | `review` 镜头均满足正式条件后可完成 | `POST /api/review-tasks/{id}/complete` 返回成功 |
| 制片 close | 仅已完成任务允许关闭 | `POST /api/review-tasks/{id}/close` 返回成功，未完成任务返回冲突 |
| context 只读 | `context` 不展示正式操作 | 不显示提交反馈、通过、重新评审 |
| context 统计 | `context` 不计入完成统计 | 完成数 / 待处理数不被 `context` 污染 |
| 参与类型一致性 | `participationMode` / `reviewParticipationMode` 输出一致 | 回读不再默默落回 `review` |

### 5.3 验收口径

1. 先在测试库清理或隔离历史脏数据，再执行回归
2. 真实 6 镜头 mixed task 回归通过后，再判定 `complete` / `close` / `context` 边界收口成功
3. 若存在候选脏数据但无法自动确认，必须保留人工核对清单，不能直接批量修正
4. `layout` 仅能作为播放源补位，不可作为 `context` 判定依据

## 5. 必做事项

1. 设计历史数据排查方案，至少说明如何找出以下异常任务：
   - 实际业务应为陪审镜头，但数据库 `ParticipationMode` 为 `review`
   - mixed task 中最后一个陪审镜头阻塞 `complete`
   - 制片侧创建或追加镜头后参与类型被覆盖
2. 若可以安全识别，提供数据修复脚本或后台修复步骤；若不能完全自动识别，给出人工校验字段组合和筛查口径
3. 增加或完善自动化测试，至少覆盖真实复现场景：
   - 6 镜头任务：1 已提交反馈，2/3/4/5 通过，6 为 `context`，导演可完成任务
   - 将镜头 6 改成 `review` 后，导演不可完成任务
   - 制片账号点击制片任务详情不报 403，但进入导演接口仍应被拒绝
4. 补前端 / 手工联调验证矩阵，至少包含：
   - 导演账号进入 mixed task
   - 切到 `context` 镜头
   - 检查反馈区、通过按钮、完成按钮状态
   - 导演完成任务
   - 制片关闭任务
5. 更新文档，记录：
   - 正式真值源
   - 历史数据风险
   - 回归步骤
   - 常见误判说明（`layout` 不等于 `context`）

## 6. 禁止事项

1. 不要只验证新建任务，不验证历史任务
2. 不要没有筛查条件就直接批量把旧数据改成 `context`

4. 不要跳过真实复现场景回归
5. 不要遗漏制片账号入口验证

## 7. 验收标准

1. 历史数据修复方案清楚可执行
2. 真实 6 镜头复现场景有自动化或严格手工回归结论
3. mixed task 完成、导演 complete、制片 close 三条链路验证通过
4. 文档中已留下排查口径、回归清单与剩余风险
5. 可以向后续执行者明确说明“为什么这类问题不应再复发”
