# 导演账号功能二次迭代客户端派工指令包

## 1. 文档用途

本文件用于把“导演账号功能二次迭代”的客户端修复任务直接派发给其他 AI 或开发执行。

本轮客户端目标不是单纯继续改切镜动画，而是在服务端修复镜头反馈回读后，进一步为导演审片工作台补上“镜头级反馈隔离”的防御性实现，防止脏数据或竞态再次把其他镜头反馈展示到当前镜头。

## 4. 问题背景

当前导演审片工作台在实际测试中出现：

1. 视频已经播放到镜头 B
2. 右侧“当前镜头反馈列表”仍显示镜头 A 的反馈
3. `镜头通过` 按钮因此被错误禁用

根因主链在服务端，但客户端也需要补强以下能力：

1. 反馈列表只展示当前镜头的反馈
2. 即使接口返回脏数据，也不能直接污染 UI
3. 切镜竞态不能让旧请求覆盖当前镜头反馈视图

## 5. 必做事项

1. 复核 `src/pages/ReviewPage.tsx` 中以下状态链路：
   - `currentIndex`
   - `currentItem`
   - `activeShotId`
   - `currentShotFeedbackView`
   - `feedbackListEntries`
   - `canApproveCurrentShot`
2. 在 `loadAllFeedbacksForShot` 消费后增加客户端二次过滤：
   - 只保留 `feedback.lensId === 当前镜头 shotId`
   - 若当前任务镜头 `taskShotId` 可用，再进一步要求 `feedback.taskShotId` 与当前镜头上下文一致或为空兼容
3. 反馈列表必须以“当前镜头 ID + 当前任务上下文”为最终真相源，而不是直接信任接口原数组。
4. `镜头通过` 按钮只能基于过滤后的当前镜头反馈结果判断。
5. 切镜时旧请求不得覆盖当前镜头：
   - 旧镜头反馈请求返回后，若当前已切到新镜头，应丢弃结果
6. 如有必要，把“当前镜头反馈视图”从页面大状态中抽成显式的 `activeShotFeedbacks` 派生结果，避免多个 memo 之间相互引用旧值。

## 6. 推荐修改方向

### 6.1 先做数据过滤再进入 UI

建议在 `src/lib/reviewFrameFeedback.ts` 或 `src/pages/ReviewPage.tsx` 入口处增加一个显式过滤函数，例如：

1. 输入：当前 `shotId`、可选 `taskShotId`、接口返回 feedbacks
2. 输出：只属于当前镜头的 feedbacks

过滤规则建议：

1. `feedback.lensId === shotId` 才允许进入列表
2. 若 `taskShotId` 存在，且 `feedback.taskShotId` 也存在，则两者必须相等
3. 对历史兼容数据，允许 `feedback.taskShotId` 为空，但不能允许 `lensId` 不匹配

### 6.2 审片页状态隔离

建议把以下概念彻底拆开：

1. `currentItem`：当前视频播放项
2. `activeShotId`：当前镜头反馈上下文
3. `shotFeedbackViewByShotId`：缓存
4. `activeShotFeedbacks`：已经过滤过的当前镜头正式反馈数组

其中 `feedbackListEntries`、`currentFrameFeedbacks`、`currentFrameSubmittedFeedback`、`canApproveCurrentShot` 都必须只基于 `activeShotFeedbacks` 推导。

### 6.3 通过按钮逻辑收口

`镜头通过` 只能在以下条件同时满足时可点击：

1. 当前镜头反馈已加载完成
2. 当前镜头正式反馈数量为 0
3. 当前镜头本地草稿数量为 0
4. 当前镜头待提交绘制修改为 0
5. 当前没有提交中 / 切镜中竞态

## 8. 禁止事项

1. 不要为了让按钮可点而直接放宽 `canApproveCurrentShot` 判定。
2. 不要把其他镜头反馈隐藏成纯视觉层“看不见”，但内部状态仍保留；内部状态也必须隔离。
3. 不要继续依赖“当前接口大概率是干净的”这种假设，客户端必须做镜头级防御性过滤。

## 9. 自测要求

至少完成以下手工回归：

1. 镜头 1 有正式反馈，镜头 2 无反馈
2. 自动播放切到镜头 2 后：
   - 反馈列表为空
   - 不显示镜头 1 的缩略图 / 文字 / 绘制
   - `镜头通过` 可点击
3. 切回镜头 1 后：
   - 恢复镜头 1 自己的反馈列表
4. 同一任务下镜头 1 / 2 / 3 各有不同反馈时，切换过程中不得串镜头
5. 保存草稿、提交反馈、删除反馈后，再切换镜头，列表仍只展示当前镜头数据
6. `npm run typecheck` 通过

## 10. 交付标准

1. 当前镜头反馈列表彻底按镜头隔离
2. 自动联播与手动切镜都不会再串反馈
3. `镜头通过` 只受当前镜头真实状态影响
4. 类型检查通过

## 11. 最终输出给上游的内容

1. 修改 / 新增文件清单
2. 客户端增加了哪些镜头级过滤与竞态防护
3. 还依赖哪些服务端修复前提
4. 手工验证结果
