## 2. 本批次目标

这一批完成后，客户端必须具备：

1. 制片账号在镜头返修记录中稳定支持选择图片、粘贴图片、图片排序、图片移除、图片上传到服务器与返修记录编辑
2. 上述一级返修图片能力被抽成可复用模块，而不是继续散落在 `LensPage.tsx` 私有逻辑中
3. 镜头详情页新增与“记录”并列的 `导演反馈` 分页签，用与记录区一致的图文卡片形式回看导演反馈
4. 导演工作台按任务镜头顺序无缝连播视频，给导演“一条完整视频”的体验
5. 随播放推进，镜头队列标签自动激活当前播放镜头
6. 导演可直接在预览窗口暂停视频，并在暂停态下进行画笔标注
7. 导演审片反馈优先复用一级返修图片能力模块来完成图片导入与本地临时图保存
8. 导演可对当前帧填写反馈说明，并保存为本地临时反馈卡片
9. 导演可点击“提交反馈”把当前任务下已暂存卡片逐条上传服务器
10. `提交反馈` 不自动改变任务最终结论；通过 / 返修 / 关闭仍为独立动作

这一批不做：

1. 不重做整套镜头页产品结构
2. 不引入另一套平行审片模型
3. 不把提交反馈和任务结论按钮合并
4. 不要求一次性把本地草稿同步成跨机器协同能力

## 3. 当前代码状态（按现状理解）

当前客户端已具备：

1. 导演任务列表、任务详情、镜头队列、评论、反馈卡片列表的基础页面
2. 镜头详情页已有一级返修图片选择 / 粘贴 / 编辑链路，以及导演反馈卡片保存链路与视频预览基础能力
3. `reviewService.createReviewFeedback(...)` 已可提交版本号、帧号、时间码、图片路径、标注 JSON
4. Electron 侧已有 `window.movtools.dialog.savePastedImage` 可保存图片到本地

当前仍存在的主要风险点：

1. 一级返修图片能力仍写死在镜头页返修弹窗与返修记录编辑弹窗中，尚未模块化
2. 镜头详情页当前导演反馈仍停留在详情主区临时编辑形态，尚未收口到与 `记录` 并列的 `导演反馈` 分页签
3. 一级返修图片当前仍偏本地路径语义，十九批后必须落成服务端真实保存，不能只停留在客户端本地文件
4. `ReviewPage.tsx` 当前仅播放单个视频，未实现任务级连播
5. 队列高亮只受点击切换影响，未绑定真实播放进度
6. 预览窗口没有暂停后可绘制的画布层
7. 导演页仍缺少“本地临时反馈卡片”状态与“提交反馈”主按钮
8. 仓储层 `createReviewFeedback` 类型可能仍缺 `decisionCode` / `taskShotId` 等字段

## 4. 已固化业务边界（必须遵守）

1. 导演工作台以正式审片任务镜头顺序为连播顺序真相源
2. 播放器必须服务“无缝连播”体验，不得退化为导演手工点多个独立视频
3. 只有暂停态允许画面标注；播放态应锁定画布避免误操作
4. 反馈卡片允许本地暂存，但服务端正式事实仍以提交成功后的卡片为准
5. “提交反馈”只上传卡片，不自动触发任务通过、返修、关闭
6. 镜头页只负责消费与查看导演反馈，不再作为导演侧主编辑入口
7. 一级镜头反馈与导演反馈是两条不同链路；镜头详情页中前者保留在 `记录` 分页，后者单独放入与之并列的 `导演反馈` 分页签，不得混成一个泛化反馈区
8. 导演审片新增图片能力时，优先复用一级返修图片能力模块，不允许复制出第二套分叉实现
9. 一级返修图片在十九批完成后必须成为服务端可持久化、跨角色可回读、可预览、可放大查看的正式协同资产
