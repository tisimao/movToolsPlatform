# 导演账号功能服务端派工指令包

## 1. 文档用途

本文件用于把“导演账号服务端协议、反馈轮次、绘制遮罩、镜头状态与镜头详情回放能力”拆成可连续落地的阶段性服务端任务。

目标目录：`D:\project\MovtoolsEx\movtools-server\`

- `D:\project\MovtoolsEx\Doc\导演账号功能设计文档.md`

---

## 2. 服务端总目标

全部阶段完成后，服务端必须具备：

1. 稳定的导演任务、反馈卡片、镜头级绘制数据与镜头状态协议
2. 以 `feedbackRoundId` 聚合同一轮反馈文字与绘制遮罩
3. `提交反馈` 只改变镜头反馈事实与镜头二级状态，不改变任务最终结论
4. 按镜头维度接收“当前轮全部本地绘制数据”并稳定回读
5. 镜头详情页可读取当前轮正式反馈与当前轮正式遮罩数据
6. 当前轮反馈提交成功后，后续客户端可进入下一轮空白编辑态，而服务端继续保留上一轮正式事实用于回看

---

## 3. 必须遵守的服务端边界

1. 绘制数据底层只按单帧记录，不定义区间型存储模型
2. 同一帧多个绘制对象以单帧绘制集合方式持久化
3. 绘制与文字反馈弱绑定，但正式提交必须至少包含一条文字反馈
4. `提交反馈` 时接收的是当前镜头当前轮全部文字反馈、全部绘制数据和镜头状态
5. `镜头通过` 时只接收镜头通过状态
6. 镜头详情遮罩只显示当前轮正式已提交内容
7. 不把客户端本地草稿视为服务端正式事实
8. 任务结论 `approve / reject / close` 仍保持独立动作边界

---

## 4. 阶段拆分

### 4.1 第一阶段：协议与领域模型收口

目标：先把反馈轮次、单帧绘制数据、镜头状态边界写进服务端 contracts、application、domain。

必须完成：

1. 为反馈卡片补齐 `feedbackRoundId`、`timestampSeconds` 等必要字段
2. 为镜头级绘制数据定义正式请求/响应模型或等价组合模型
3. 明确 `DRAWN / CLEAR` 两类单帧绘制状态
4. 明确 `feedbackRoundId` 用于聚合同一轮反馈卡片与绘制数据
5. 明确“只提绘制不提文字”在服务端的拒绝策略或前置校验约束

建议重点排查：

- `src/Movtools.Server.Api/Contracts/ReviewContracts.cs`
- `src/Movtools.Server.Application/**/*Review*`
- `src/Movtools.Server.Domain/Entities/*Review*`
- `src/Movtools.Server.Infrastructure/Services/ReviewService.cs`

完成定义：

1. 协议层、应用层、领域层对反馈轮次与绘制帧模型达成一致
2. 文档与代码字段语义一致

### 4.2 第二阶段：提交反馈、镜头通过与绘制落库链路

目标：打通正式提交动作与镜头级绘制数据持久化。

必须完成：

1. `提交反馈` 时可接收当前镜头当前轮全部文字反馈
2. `提交反馈` 时可接收当前镜头当前轮全部绘制数据
3. `镜头通过` 时只处理镜头通过状态
4. 存在文字反馈时，镜头状态推进到 `PENDING_FEEDBACK_FIX`
5. 镜头通过时，镜头状态推进到 `DIRECTOR_APPROVED` 或等价内部通过态
6. 若当前轮只有绘制没有文字反馈，正式提交必须拒绝
7. 绘制落库采用单帧记录，支持 `CLEAR` 作为清屏锚点

建议重点排查：

- `src/Movtools.Server.Api/Controllers/ReviewsController.cs`
- `src/Movtools.Server.Api/Controllers/*Review*`
- `src/Movtools.Server.Application/Interfaces/IReviewService.cs`
- `src/Movtools.Server.Infrastructure/Services/ReviewService.cs`

完成定义：

1. 服务端已形成正式提交与镜头通过两条稳定链路
2. 绘制数据可稳定写入并与反馈轮次关联

### 4.3 第三阶段：回读投影、镜头详情遮罩与当前轮选择

目标：让客户端能稳定回读“当前轮反馈 + 当前轮遮罩”。

必须完成：

1. `GET /api/review-feedbacks/lens/{lensId}` 或等价接口可按 `feedbackRoundId` 回读反馈卡片
2. 绘制数据回读接口可按镜头与当前轮回读单帧绘制集合
3. 镜头详情页回读或组合查询可明确给出当前镜头最近正式反馈轮次
4. 镜头详情遮罩只默认消费最近一轮正式 `feedbackRoundId` 对应的绘制数据
5. 不同轮反馈之间的数据不会混放成一套不可区分结果
6. 任务详情中的反馈计数、最近反馈时间、镜头状态统计保持一致

建议重点排查：

- `src/Movtools.Server.Infrastructure/Services/ReviewService.cs`
- `src/Movtools.Server.Infrastructure/Queries/*`
- `src/Movtools.Server.Api/Controllers/*`
- `tests/Movtools.Server.Tests/*Review*`

完成定义：

1. 客户端可独立消费“最新一轮正式反馈 + 最新一轮正式遮罩”
2. 统计、状态、回读结果三者口径一致

### 4.4 第四阶段：校验、回归测试与联调收口

目标：把所有关键边界做成可验的服务端保证点。

必须完成：

1. 测试覆盖：
   - 只有绘制无文字反馈时提交被拒绝
   - 同一轮多条反馈共享 `feedbackRoundId`
   - 同一轮全部绘制数据成功落库并可回读
   - `CLEAR` 帧可稳定回读
   - 提交反馈后镜头进入 `PENDING_FEEDBACK_FIX`
   - 镜头通过后镜头进入 `DIRECTOR_APPROVED`
   - 提交反馈不改变任务最终结论
2. 任务统计、镜头统计、镜头详情回读在新模型下无冲突
3. 服务端构建与测试通过
4. 对客户端给出当前正式接口组合、字段要求与错误语义说明

完成定义：

1. 服务端已具备可长期维护的导演反馈轮次与遮罩模型
2. 与客户端联调所需的正式保证点已齐备

## 6. 服务端最终完成定义

只有同时满足以下条件，才算服务端派工包完成：

1. 反馈卡片、绘制帧、反馈轮次、镜头通过动作边界已稳定
2. 服务端可按镜头接收当前轮全部文字反馈与全部绘制数据
3. 不允许纯绘制提交
4. 服务端可稳定回读当前轮正式反馈与当前轮正式绘制遮罩
5. 镜头详情只显示正式提交内容，不显示本地未提交态
6. 镜头状态、任务统计、反馈回读三者一致
7. 构建、测试与联调验证通过
