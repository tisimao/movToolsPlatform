# 通信协议文档

## 1. 文档目的

本文件用于固化客户端与服务端在当前联调范围内的请求、响应、错误与字段命名规范，作为双方实现接口时的统一对照基准。

本文件不再限定为“第五批”专用文档，而是后续各批次共同维护的**唯一协议基线**。

目标：

- 避免客户端 / 服务端重复改协议
- 避免 DTO 字段名、字段类型、数组结构不一致
- 避免“前端发了但后端没接”或“后端加了但前端没用”的偏差
- 避免多批次并行开发时因引用旧协议文件而再次产生阻塞

### 1.0 当前已完成协议范围

当前已经完成并落地的协议范围包括：

1. 登录与当前用户协议
2. 项目创建协议
3. 项目初始化状态协议
4. 第六批客户端主导初始化职责边界
5. 镜头批量同步协议
6. 项目成员协议

### 1.0.1 第七批待共同定义范围

第七批将围绕“镜头文件引用 / 文件绑定与旧项目对齐”继续扩充本文件。以下范围目前仍需客户端、服务端、验收三方共同定义后才能开始实现：

1. 文件绑定同步接口
2. relativePath / sourceRoot / bindingType / versionNum 等字段语义
3. 绑定覆盖、重复、删除、冲突规则
4. 镜头详情中的文件绑定回读协议

维护规则：

- 客户端派工、服务端派工、联调验收都必须共同维护本文件
- 任何一侧新增、删除、重命名、放宽或收紧协议字段，都必须先更新本文件，再同步代码实现
- 如果实现与本文件不一致，以本文件为准，并将不一致项列入阻塞

### 1.1 协议红线（必须遵守）

以下规则为硬性红线：

1. 不允许客户端和服务端私自扩字段后再“事后补文档”
2. 不允许只修改一侧实现，期待另一侧“自动兼容”
3. 不允许把页面静态文案、默认值、兼容分支当成协议
4. 不允许在未更新本文件前合并任何协议相关实现
5. 不允许在验收时跳过协议核对

凡违反上述任一条，均视为**协议阻塞**。

### 1.2 协议变更流程

所有协议变更必须按以下顺序执行：

1. 先修改本文件
2. 明确变更影响范围：客户端 / 服务端 / 验收
3. 再修改代码实现
4. 最后在交付说明中写清：
   - 改了哪些字段 / 路由 / 状态 / 错误语义
   - 哪些文件同步修改了
   - 哪些旧逻辑继续兼容，哪些不再兼容

若代码已改而本文件未改，则该实现不得视为完成。

### 1.3 联调与验收要求

联调和验收时必须额外检查：

1. 客户端实际请求是否与本文件一致
2. 服务端实际响应是否与本文件一致
3. 错误状态与错误码是否与本文件一致
4. 是否存在实现已变更但文档未更新的情况

只要存在“实现已变更但文档未更新”，则验收结论不得判定为通过。

### 1.4 交接速览（当前稳定范围）

当前协议已稳定落地的主线可以直接按角色理解：

1. 管理员：`/api/users`、`/api/roles`、`/api/path-mappings/*`、`/api/diagnostics/*`
2. 制片：`/api/projects`、`/api/project-members`、`/api/projects/{projectCode}/episodes`、`/api/episodes/{episodeId}/lenses`、`/api/review-tasks/*` 的任务装配侧、`/api/lenses/{lensId}/repair-attachments`
3. 导演：`/api/review-tasks`、`/api/review-feedbacks/*` 的审片侧
4. 制作人员：`/api/projects` 只读范围内的项目列表、`/api/episodes/{episodeId}/lenses` 的本人镜头只读范围、`/api/lenses/{id}/detail` 的反馈回看

当前协议治理建议优先记住三条：

1. 服务端决定协同主数据范围，客户端决定本机文件能力范围
2. `ReviewTask` 已拆成“导演审片主链路”和“制片任务装配链路”两种消费方式
3. `review` / `context` 是任务镜头参与类型，不能被播放源语义替代

## 2. 通用规则

### 2.1 基本约定

- 所有 JSON 字段使用 `camelCase`
- 需要多个值的字段统一使用数组，不要用逗号拼接字符串
- 标识类字段优先使用 `Guid` / `string`，不要混用数字 ID
- 可选字段在协议层应显式标注为可空

### 2.2 错误响应

服务端统一错误响应以 `ApiErrorResponse` 为准：

- `category`
- `code`
- `message`
- `traceId`
- `errors`

其中：

- 参数校验失败统一返回 `validation_error`
- 业务冲突按业务错误码返回
- 权限不足统一返回授权/拒绝类错误码

补充约定：

1. 审片任务草稿创建 / 编辑时，服务端必须先校验导演用户存在性与导演角色；若导演不存在，返回 `user_not_found`，若用户不具备导演角色，返回 `director_required`
2. 草稿任务保存若因数据库结构落后、镜头重复或外键冲突失败，服务端不得仅暴露原始 `DbUpdateException`，应映射为业务错误码：`review_task_schema_outdated`、`review_task_duplicate`、`review_task_fk_failed`、`review_task_save_failed`
3. 客户端创建审片任务时必须提交真实导演 `userId`，不得再把用户名、显示名或手工文本当作导演主键
4. 当前审片任务创建与编辑链路不再依赖截止时间输入；`deadlineUtc` 仅作为兼容字段保留，客户端默认不收集、不展示
5. 服务端启动时会自动执行数据库迁移；在开发环境下会优先使用业务账号连接 PostgreSQL 维护库来删除并重建缺失的开发数据库，若重建失败则自动回退到 `postgres` 维护库继续完成本地联调
6. 导演进入审片任务时，客户端应先调用开始任务动作，再进入任务详情，以保证任务状态从 `pending` 正常切换到 `in-review`
7. 导演工作台中的“提交反馈”只负责上传本轮反馈卡片与本轮绘制数据，不自动调用任务 `approve` / `reject` / `close` 等最终结论动作
8. 从导演反馈绘制时间线收口版本开始，正式协议语义固定为：`反馈条目` 与 `绘制时间线` 是两个对象层级；前者按帧绑定，后者按 `feedbackRoundId` 绑定
9. 任意页面若需要回放正式绘制，必须优先消费轮次级 `drawingTimeline` / `latestRound.drawingFrames`，不得再把单条 feedback 自带 `drawingFrames` 视为唯一正式真相源
10. 镜头详情页导演反馈回放若需要和导演审片页保持一致，客户端必须在“当前版本 + 当前轮”范围内选择正式绘制 timeline；不能直接把整镜头 `latestRound` 无条件当作当前版本真相源
11. 回放类页面消费正式绘制时，协议层只提供“轮次级 timeline / drawingFrames 事实”；客户端若要实现与视频同步的逐帧回放，必须自行按当前视频帧调用绘制解析逻辑，不得把“最终态整图”视作协议要求
12. 审片任务镜头参与类型正式字段统一为 `participationMode`；`reviewParticipationMode` 仅作为历史兼容回读别名，不再作为新增请求字段
13. 新增或写回任务镜头时，`participationMode` 必须显式传入 `review` 或 `context`，不允许缺省后由服务端默默补成 `review`
14. `review` 与 `context` 的判定必须以任务镜头参与类型为准；播放源、layout 源或前端展示状态不得反向决定正式参与类型
15. 导演 `complete` 与制片 `close` 必须保持独立动作：`complete` 仅允许导演侧完成任务，`close` 仅允许制片侧归档已完成任务
16. `context` 镜头不得参与正式反馈、正式通过、正式返修、完成统计与关闭后的正式状态回退

## 3. 认证协议

### 3.1 登录

- `POST /api/auth/login`
- Request: `LoginRequest`
  - `userName: string`
  - `password: string`
- Response: `LoginResponse`
  - `userId`
  - `userName`
  - `displayName`
  - `accessToken`
  - `expiresAtUtc`
  - `roles: string[]`

### 3.2 当前用户

- `GET /api/auth/me`
- Response: `CurrentUserResponse`

### 3.3 角色显示约定

认证接口中的 `roles: string[]` 仍用于协同权限判定，短期不改为中文。

补充约定：

1. `roles` 中的角色码仍使用稳定英文标识，例如：`admin`、`system-admin`、`producer`、`director`、`maker`、`viewer`
2. 任何用户可见的角色下拉框、角色标签、角色说明文案，不允许直接显示原始英文角色码
3. 若客户端通过 `GET /api/roles` 读取角色字典，则角色响应中应至少可提供：
   - `code: string`
   - `displayName: string`
4. 若当前阶段 `GET /api/roles` 尚未返回 `displayName`，客户端必须自行维护稳定的中文映射，并保证所有用户可见角色文案显示为中文

### 3.4 用户角色分配语义

`POST /api/users/{userId}/roles` 的请求体 `roleCodes` 以角色稳定码为主契约，但为了兼容历史数据与中文界面回填，服务端可接受以下等价输入：

1. 角色 `code`，如 `maker`
2. 角色 `name`，如 `制作人员`
3. 角色显示名 `displayName`，如 `制作人员`

补充约定：

1. 客户端提交时必须优先使用 `code`
2. 用户界面只显示中文，不得把中文展示值当成持久化主键
3. 服务端内部落库与权限判断仍以 `code` 为准
4. 若输入无法映射到已知角色，服务端继续返回 `role_not_found`

补充说明：本机文件检查能力仅影响当前机器上的版本状态 / Layout 状态 / 预览状态展示，不改变协同镜头主数据；因此 maker 可以执行本地检查，但不能执行会写回协同 binding 的操作。
Layout 引用检查同样属于本机诊断能力，只用于当前机器上的缺失排查与导出，不得触发协同绑定写回。

补充说明：当前版本文件状态的提示文案必须直接暴露缺失项类型；当 MA 或 MOV 未匹配时，客户端应优先显示“缺失 MA 文件”或“缺失 视频文件”，仅在无法细分时才回退为“当前版本未完全匹配”。

补充说明：当当前版本文件已全部匹配时，客户端应显示绿色完整态，并折叠列出已匹配文件名；当完全未匹配时，应显示红色缺失态并提示文件缺失。

补充说明：登录态切换后，客户端必须丢弃旧账号缓存的镜头列表并重新拉取当前账号可见镜头，避免跨账号残留数据造成“数量没变 / 状态错位”的假象。

### 3.5 制作人员本地文件检查语义

在协同模式下，`maker` 角色仍然允许执行纯本地文件检查与预览相关动作，用于补强当前机器的镜头状态展示。

补充约定：

1. `检查文件`、`刷新列表`、`刷新全部文件匹配`、`检查当前 Layout 引用` 属于本地能力，不是协同写操作
2. 这些动作可以刷新当前机器上的 `currentVersionReady`、`layoutReady`、`layoutVideoReady`、预览与引用检查结果
3. maker 仍然不得执行任何会回写协同主数据的绑定、状态流转、导入、删除操作
4. 客户端不得因为 maker 只读而屏蔽本地扫描结果合并逻辑

## 4. 项目协议

### 4.1 创建项目

- `POST /api/projects`
- Request: `ProjectCreateRequest`

字段：

- `code: string`
- `name: string`
- `projectRootPath?: string | null`
- `description?: string | null`
- `versionTag: string`
- `layoutTag: string`
- `projectDefaultFps?: number`
- `initialEpisodeCode?: string | null`
- `initialEpisodeName?: string | null`
- `initExcelPath?: string | null`
- `lensRoots?: ProjectScanRootRequest[]`
- `layoutRoots?: ProjectScanRootRequest[]`
- `members?: ProjectMemberCreateRequest[]`

`ProjectScanRootRequest` 目标结构：

- `rootId?: string | null`
- `label?: string | null`
- `fileKind?: "ma" | "mov" | "layout" | null`
- `absolutePath: string`
- `priority?: number`
- `isEnabled?: boolean`
- `initExcelPath?: string | null`

`ProjectScanRootResponse` 目标结构：

- `rootId?: string | null`
- `label?: string | null`
- `fileKind?: "ma" | "mov" | "layout" | null`
- `absolutePath: string`
- `priority?: number | null`
- `isEnabled?: boolean | null`
- `initExcelPath?: string | null`

补充约定：

1. `initialEpisodeCode` / `initialEpisodeName` 用于明确“创建项目时是否同步创建首集”
2. 若创建首集，则服务端必须按同一请求承接初始化上下文，而不是仅保存项目骨架
3. `lensRoots` / `layoutRoots` 不允许再退化为 `string[]`，否则会丢失 `initExcelPath`、优先级、启用状态等语义
4. `initExcelPath` 为兼容旧流程的全局默认值；若根目录对象同时存在自己的 `initExcelPath`，则以根目录对象为准
5. `members` 仍用于项目成员预置，不得退回 `memberUserIds`
6. 创建项目时不要求先把 Excel 中所有制作人员全部手工放入 `members`；第八批起允许“项目先创建，再按当前项目成员对 Excel 人名做自动匹配”
7. `members` 可为空数组或不传；这不应阻断项目创建，创建人仍需自动成为项目 producer
8. `projectRootPath` 是项目主目录正式字段；`description` 仅保留项目说明语义，不得再借用为路径传输字段
9. `fileKind` 用于声明该 roots 条目服务于哪类文件检查上下文；服务端与客户端不得再依赖数组名称或本地猜测补足该语义
10. `projectDefaultFps` 为项目默认帧率字段，客户端创建项目时可显式传入；若未传或非法，服务端必须按 `30` 归一化兜底，并在项目回读与列表回读中返回该值

- `deadlineUtc?: DateTimeOffset | null`（兼容字段，当前客户端默认不展示/不收集）

### 4.1.1 创建项目响应

- Response: `ProjectCreateResponse`

字段：

- `project: ProjectResponse`
- `initialEpisode?: EpisodeResponse | null`
- `initResult: ProjectInitializationResult`

`ProjectInitializationResult` 目标结构：

- `status: "not_requested" | "skipped" | "success" | "partial_success" | "failed"`
- `message: string`
- `excelImportAttempted: boolean`
- `excelImportSuccess: boolean`
- `createdLensCount?: number`
- `lensFoldersPlanned?: number`
- `lensFoldersCreated?: number`
- `pendingClientActions?: string[]`
- `errors?: string[]`

补充约定：

1. 不能再用静态“项目创建成功”充当初始化成功
2. `project` 创建成功但 `initResult.status = failed` 是合法状态，客户端必须明确展示
3. 若本地镜头文件夹创建仍由客户端承担，则服务端应通过 `pendingClientActions` 明示，例如：
   - `create_lens_folders`
   - `refresh_local_episode_workspace`
4. `not_requested` 表示根本未请求初始化
5. `skipped` 表示请求了项目创建，但因缺少 Excel 或根目录条件，未执行初始化
6. `partial_success` 表示项目和首集上下文已成功创建，但仍有后续客户端动作未完成
7. `failed` 表示项目创建后初始化失败或被中止
8. `success` 表示初始化已真实完成

`ProjectResponse` 在项目详情 / 列表 / 创建回包中统一返回以下协同主数据：

- `code: string`
- `name: string`
- `projectRootPath?: string | null`
- `description?: string | null`
- `versionTag: string`
- `layoutTag: string`
- `initExcelPath?: string | null`
- `lensFolderRootPath?: string | null`
- `layoutCheckPath?: string | null`
- `lensRoots: ProjectScanRootResponse[]`
- `layoutRoots: ProjectScanRootResponse[]`

`ProjectScanRootResponse` 目标结构：

- `rootId?: string | null`
- `label?: string | null`
- `fileKind?: "ma" | "mov" | "layout" | null`
- `absolutePath: string`
- `priority?: number | null`
- `isEnabled?: boolean | null`
- `initExcelPath?: string | null`

补充约定：

1. `lensRoots` / `layoutRoots` 必须返回真实根目录绝对路径 `absolutePath`，用于客户端协同激活时推导本机目录
2. 当前阶段允许服务端保存并回读该绝对路径，客户端再按 Windows 盘符差异做本机重定位
3. `initExcelPath` 仅作为协同上下文默认值，不代表本地文件能力
4. `projectRootPath`、`lensFolderRootPath`、`layoutCheckPath` 等主字段属于正式响应字段，不得再借用 `description` 或其他非路径字段承载
5. `fileKind` 必须稳定回读，客户端不得再把远端 roots 直接当成本地扫描 roots 类型而省略映射
6. 若后续需要扩展根目录语义，必须继续保持 `absolutePath` 可稳定回读

`EpisodeResponse` 在项目集列表 / 集详情 / 创建集回包中统一返回以下协同主数据：

- `id: Guid`
- `code: string`
- `name: string`
- `sequence: number`
- `description?: string | null`
- `projectId: Guid`
- `projectCode: string`
- `isArchived: boolean`
- `rowVersion: long`
- `createdAtUtc: DateTimeOffset`
- `updatedAtUtc: DateTimeOffset`
- `versionTag?: string | null`
- `layoutTag?: string | null`
- `initExcelPath?: string | null`
- `lensFolderRootPath?: string | null`
- `layoutCheckPath?: string | null`
- `lensRoots: ProjectRootResponse[]`

`ProjectRootResponse` 目标结构：

- `rootId?: string | null`

- `fileKind?: "ma" | "mov" | "layout" | null`
- `absolutePath: string`

补充约定：

1. 项目激活与集切换时，客户端可以优先使用 `projectRootPath`、`lensFolderRootPath`、`layoutCheckPath`
2. 若 `lensFolderRootPath` / `layoutCheckPath` 缺失，客户端必须可回退到 `lensRoots` / `layoutRoots` 中优先级最高且启用的 `absolutePath`
3. maker / producer 等有项目访问权限的账号，读取到的项目 / 集 roots 必须与服务端当前协同配置一致，不允许因角色差异丢失 `absolutePath` 或 `fileKind`
4. `ProjectResponse` 与 `EpisodeResponse` 中 roots 的结构必须保持同构，避免客户端维护两套映射规则

### 4.1.1.0 协同路径字段正式语义

自第十五批起，项目 / 集协同路径字段正式定稿如下：

1. `projectRootPath` 表示协同项目主目录，是项目激活与本地工作区恢复的首选主字段
2. `lensFolderRootPath` 表示当前项目或当前集的默认镜头文件根目录主字段，主要用于激活时快速建立文件检查上下文
3. `layoutCheckPath` 表示当前项目或当前集的默认 Layout 根目录主字段
4. `description` 仅承担业务说明语义，不再承担任何路径字段语义
5. `lensRoots` / `layoutRoots` 属于完整协同配置真相源；主字段缺失时，客户端必须从这些 roots 中回退推导
6. `fileKind` 属于正式协议字段；客户端不得继续依赖“lensRoots 一定等于 ma / mov、layoutRoots 一定等于 layout”的隐式假设
7. 历史数据若曾把路径写入 `description`，只允许在迁移或兼容阶段临时兜底读取，不得继续作为正式写入规则

### 4.1.1.0.1 第十五批数据库迁移与环境验收约束

自第十五批起，协议字段不仅要求接口层存在，也要求数据库结构与迁移状态同步满足以下约束：

1. 当前数据库必须已应用迁移 `20260511000000_S15_ProjectEpisodeFormalPaths`
2. `projects` 表必须存在以下正式列：
   - `ProjectRootPath`
   - `LensFolderRootPath`
   - `MaCheckPath`
   - `MovCheckPath`
   - `LayoutCheckPath`
3. `episodes` 表必须存在以下正式列：
   - `LensFolderRootPath`
   - `LayoutCheckPath`
4. 若运行环境缺失上述列，则 `GET /api/projects`、`GET /api/projects/{code}`、`GET /api/projects/{projectCode}/episodes`、`GET /api/episodes/{id}` 都可能在 ORM 查询阶段直接失败；该场景应判定为数据库结构阻塞
5. 清空业务数据脚本仅允许清数据，不允许视作协议升级动作；环境升级必须通过 EF migration 完成
6. 验收 producer / maker 空项目登录场景前，必须先确认数据库结构已满足本节要求，否则验收结论无效

### 4.1.1.0.2 新设备激活时的本地路径解析与 bootstrap 约束

1. 客户端激活协同项目时，必须先将服务端正式路径字段与 roots 明细解析为本机最终路径，再决定是否创建本地工作区
2. `projectRootPath`、`lensFolderRootPath`、`layoutCheckPath` 必须分别按各自语义解析；不得把 `projectRootPath` 直接作为镜头文件根目录或 Layout 根目录兜底
3. 若 `lensFolderRootPath` / `layoutCheckPath` 缺失，客户端只允许从对应语义的 roots 中回退推导，不得跨语义回退到项目根目录
4. 若本地项目工作区不存在，客户端可执行 bootstrap，但 bootstrap 仅用于创建本地工作区骨架，不代表允许伪造生产素材目录
5. 若本机最终镜头文件根目录与本机最终 Layout 根目录重复、互相包含，客户端必须在 bootstrap 前直接阻断激活，并返回明确路径冲突错误
6. 若缺失的是本地工作区骨架，可自动初始化；若缺失的是业务要求必须真实存在的素材目录，应返回素材目录缺失错误，不得静默代建
7. 远端激活失败时，错误语义应优先区分：服务端路径字段缺失、本机路径重定位失败、本机路径冲突、本地工作区未初始化、素材目录缺失
8. 不同设备之间允许盘符不同；服务端路径中的盘符不是协同强约束，客户端应以盘符后的路径结构作为 Windows 多机重定位的主要匹配依据
9. 当服务端路径在本机不存在时，客户端应按 `projectRootPath`、`lensFolderRootPath`、`layoutCheckPath` 以及对应 roots 明细分别尝试“同结构不同盘符”重定位
10. 只有当本机候选路径真实存在且盘符后的路径结构一致时，客户端才允许自动采用重定位结果；否则应返回本机缺少对应目录的明确错误
11. 自动盘符重定位完成后，客户端仍必须对重定位后的镜头文件根目录与 Layout 根目录执行冲突校验，避免因盘符变化掩盖路径语义错误
12. 当前阶段盘符重定位协议只覆盖 Windows 本地盘符差异，不要求自动兼容 UNC、NAS、目录改名或路径结构已变化的情况
13. 激活失败后的手动修复流程只允许用户重选 `projectRootPath`；镜头文件根目录与 Layout 根目录继续沿用服务端路径，并仅按新项目盘符做自动重映射
14. 项目页和仪表盘页的激活失败修复入口必须一致：只提供项目根目录重选，不提供镜头文件根目录或 Layout 根目录单独修复入口

### 4.1.1.1 第六批初始化职责边界定稿

第六批起，制片初始化主链路按以下职责边界执行：

客户端负责：

1. 读取并解析旧项目当前真实使用的 Excel 文件
2. 兼容旧 `.xls` 格式以及旧表头/旧字段映射规则
3. 将 Excel 解析结果转换为镜头初始化数据
4. 在本地完成初始化前校验
5. 在项目与首集创建成功后，调用服务端批量创建镜头接口
6. 在服务端镜头落库成功后，创建本地镜头文件夹
7. 回填并展示真实初始化结果

服务端负责：

1. 创建项目
2. 创建首集
3. 返回客户端继续初始化所需上下文
4. 接收客户端批量同步的镜头并真实落库
5. 返回结构化结果与错误语义

第六批明确不再要求：

1. 服务端直接兼容旧项目 `.xls` 解析
2. 服务端负责本地镜头文件夹创建

### 4.1.1.2 第六批初始化状态语义

在第六批客户端主导初始化方案下，`ProjectInitializationResult` 语义补充如下：

1. `not_requested`
   - 表示项目创建请求中根本未请求初始化
2. `skipped`
   - 表示项目和首集创建成功，但客户端侧因缺少 Excel / 根目录 / 必要初始化条件而未继续执行初始化
3. `partial_success`
   - 表示项目和首集上下文已创建成功，服务端已把后续初始化主导权交给客户端，但仍存在待执行的客户端动作
4. `failed`
   - 表示初始化链路已进入执行，但失败发生在以下任一层：
      - 客户端本地 Excel 解析失败
      - 客户端初始化数据校验失败
      - 服务端镜头批量同步失败
      - 其他导致初始化中止的错误
5. `success`
   - 表示项目创建、镜头同步、本地镜头文件夹创建都已真实完成

补充约定：

1. `excelImportAttempted = true` 不再等同于“服务端已解析 Excel”，而表示初始化链路已真实尝试执行
2. 在第六批方案下，旧 Excel 的解析主责任在客户端，不在服务端
3. 页面必须能区分“项目创建成功”和“初始化全部成功”
4. 当服务端已创建项目和首集上下文，但仍需客户端继续执行镜头同步 / 本地目录动作时，`initResult.status` 应返回 `partial_success`

### 4.1.2 客户端 / 服务端职责边界

服务端必须负责：

- 校验项目创建请求
- 创建项目
- 在协议允许的情况下创建首集
- 接收客户端批量同步的首集镜头并真实落库，或明确返回未执行原因
- 返回结构化初始化结果

客户端必须负责：

- 收集并透传完整初始化参数
- 解析旧项目 Excel 并生成初始化镜头数据
- 在项目 / 首集创建成功后发起镜头批量同步
- 展示真实初始化结果
- 若协议约定本地镜头文件夹创建由客户端承担，则在收到响应后执行并回填结果展示

双方都不得做：

- 用模糊文案代替真实初始化状态
- 在协议未更新前单方面新增字段
- 把系统角色和项目角色混用
- 把旧项目 Excel 解析责任在客户端和服务端之间来回漂移而不更新协议

### 4.1.3 第六批镜头批量同步协议

- `POST /api/projects/{projectCode}/episodes/{episodeId}/lenses/batch`
- Request: `LensBatchCreateRequest`
- Response: `LensResponse[]`

`LensBatchCreateRequest` 目标结构：

- `lenses: LensCreateRequest[]`

`LensCreateRequest` 目标结构：

- `code: string`
- `name: string`
- `sequence: number`
- `singleFrame?: number`
- `description?: string | null`
- `rootCode?: string | null`
- `logicalPath?: string | null`
- `versionTag?: string | null`
- `layoutTag?: string | null`
- `makerUserId?: Guid | null`
- `makerNameRaw?: string | null`
- `makerMatchStatus?: "matched" | "unmatched" | "unassigned"`

`LensResponse` 在保留既有镜头主字段基础上，目标上也必须返回：

- `singleFrame: number`
- `makerUserId?: Guid | null`
- `makerNameRaw?: string | null`
- `makerDisplayName?: string | null`
- `makerMatchStatus?: "matched" | "unmatched" | "unassigned"`

加入导演内部审片功能后，`LensResponse` 还必须补充以下二级状态字段：

- `versionNum?: string | null`
- `internalReviewStatusCode?: "NOT_IN_REVIEW" | "READY_FOR_REVIEW" | "IN_DIRECTOR_REVIEW" | "PENDING_FEEDBACK_FIX" | "FIX_UPDATED" | "DIRECTOR_APPROVED" | null`
- `internalReviewStatusName?: string | null`
- `internalReviewUpdatedAtUtc?: DateTimeOffset | null`
- `latestReviewTaskId?: Guid | null`
- `latestDirectorFeedbackAtUtc?: DateTimeOffset | null`
- `pendingDirectorFeedbackCount?: number`
- `submissionAllowed?: boolean`

第九批为了让镜头列表页能回读刷新后的绑定状态摘要，服务端可以在同一份 `LensResponse` 中额外返回以下只读字段：

- `fileBindingCount: number`
- `latestFileBindingUpdatedAtUtc?: DateTimeOffset | null`

同一份 `LensResponse` 还作为以下接口的统一回包协议：

- `POST /api/episodes/{episodeId}/lenses`
- `GET /api/episodes/{episodeId}/lenses`
- `GET /api/lenses/{id}`
- `PUT /api/lenses/{id}`
- `GET /api/lenses/{id}/detail`

`LensUpdateRequest` 目标结构与创建请求一致保留 maker 语义字段，更新时服务端必须按下列规则解释：

- `makerUserId?: Guid | null`
- `makerNameRaw?: string | null`
- `makerMatchStatus?: "matched" | "unmatched" | "unassigned"`
- 若未传入任一 maker 字段，则保留镜头现有 maker 状态
- 若显式传入 maker 字段，则必须按第八批规则校验并回写真实状态

补充说明：

1. `LensUpdateRequest` 仍不用于修改镜头二级状态
2. 镜头二级状态必须走专用内部审片状态协议，避免与镜头基础信息编辑混用

补充约定：

1. 本接口用于第六批客户端主导初始化链路中的镜头批量同步
2. 客户端必须先完成旧 Excel 解析，再调用本接口
3. 服务端不负责在本接口内再次解析旧项目 Excel
4. `code` 为镜头主标识，必须在同一集内保持唯一
5. `name` 为镜头名称，允许与旧项目命名保持一致
6. `sequence` 为镜头顺序号，必须为正整数
7. `logicalPath` 若传入，则在同一集内必须保持唯一；若与同批次或已存在镜头冲突，返回业务冲突
8. `rootCode` / `versionTag` / `layoutTag` 可按当前初始化数据真实情况传入；若未提供，服务端按可空字段处理
9. `singleFrame` 表示镜头初始设计帧数；若 Excel 未提供则由客户端按 `0` 传入
10. 第八批起，镜头正式制作人员以 `makerUserId` 为准，不能再以纯文本名字直接充当正式负责人主字段
11. `makerNameRaw` 用于保留 Excel 原始制作人员姓名，供自动匹配失败后人工补挂使用
12. `makerMatchStatus` 语义如下：
    - `matched`：已匹配到系统账号，且 `makerUserId` 有值
    - `unmatched`：Excel 中有原始姓名，但当前无法匹配账号，此时 `makerUserId = null`
    - `unassigned`：镜头当前未指定制作人员，且不存在有效原始姓名线索
13. 客户端在批量同步前，必须优先按“当前项目成员”做制作人员账号匹配，至少支持：
    - 按 `displayName` 精确匹配
    - 按 `userName` 精确匹配
    - 在不引入高误判风险前提下做基础规范化（去空格、全半角/大小写归一）
14. 若未匹配到账号，不得伪造 `makerUserId`，也不得把原始姓名直接冒充为系统账号标识
15. 当 `singleFrame = 0` 时，表示镜头帧数尚未初始化完成；后续允许客户端使用已匹配的 Layout 视频做一次性补偿初始化
16. 该补偿初始化必须先在客户端本地完成真实落库，再通过镜头更新链路同步到服务端；服务端不负责主动根据客户端本地 Layout 视频推导帧数
17. 一旦客户端已完成这次初始化并同步成功，服务端回读的 `singleFrame` 即视为正式镜头帧数；后续变更只能通过人工镜头编辑提交，不能再依赖自动 Layout 回填

成功语义：

1. 服务端返回实际创建成功的镜头列表
2. 客户端应以服务端返回结果或服务端最终镜头列表作为后续本地镜头文件夹创建依据
3. 若制作人员未匹配成功，接口仍可创建镜头，但必须在响应中返回 `makerMatchStatus = unmatched` 与对应 `makerNameRaw`

错误语义：

1. 空镜头列表：返回参数校验错误
2. 非法字段：返回参数校验错误
3. 重复镜头编号：返回业务冲突（409）
4. 重复 logicalPath：返回业务冲突（409）
5. 非项目成员调用：返回授权失败
6. 项目或首集不存在：返回未找到错误
7. 若传入了 `makerUserId` 但该用户不属于当前项目成员，返回参数校验错误或业务错误，不能静默接收

联调要求：

1. 客户端请求结构必须与本节保持一致
2. 服务端响应结构必须与本节保持一致
3. 若批量创建行为发生变化，必须先更新本文件，再改代码

### 4.1.3.1 第八批制作人员账号匹配规则

第八批起，镜头制作人员的协同语义补充如下：

1. 镜头正式负责人必须可映射到系统用户账号
2. Excel 制作人员列只作为匹配线索，不再直接等同于正式负责人字段
3. 客户端在项目 / 首集创建后的镜头批量同步前，应先拉取当前项目成员列表并尝试匹配
4. 若匹配成功：
   - 传入 `makerUserId`
   - 写入 `makerNameRaw`
   - `makerMatchStatus = matched`
5. 若匹配失败：
   - `makerUserId = null`
   - 仍需保留 `makerNameRaw`
   - `makerMatchStatus = unmatched`
6. 后续当 admin 创建了账号、制片把该用户加入项目后，镜头编辑接口应允许把未匹配镜头重新指定到合法项目成员账号
7. “未匹配”不视为接口失败，而视为可回补的协同状态

### 4.1.3.2 镜头编辑中的制作人员处理规则

第八批起，镜头编辑对制作人员的协议语义补充如下：

1. 镜头编辑页修改正式制作人员时，客户端提交的仍必须是账号标识，而不是纯文本姓名
2. 若用户在镜头编辑中选择了某个制作人员账号，则：
   - `makerUserId` 必须为该账号
   - `makerMatchStatus = matched`
   - `makerNameRaw` 可保留最后一次导入原始姓名，或在无意义时置空，但不得覆盖正式账号语义
3. 若用户在镜头编辑中主动清空制作人员，则：
   - `makerUserId = null`
   - `makerMatchStatus = unassigned`
   - `makerNameRaw` 仅在业务需要保留导入线索时继续保留；若本次属于纯手工清空且无导入线索，可为空
4. 服务端必须拒绝把非当前项目成员账号写入 `makerUserId`
5. 镜头编辑成功后，列表与详情接口必须返回更新后的真实状态

### 4.1.3.3 镜头二级状态协议补充

加入导演内部审片后，镜头状态协议正式拆分为两层：

1. 镜头主状态：面向客户的正式状态流转
2. 镜头二级状态：面向内部导演审片流程

镜头二级状态稳定码约定如下：

- `NOT_IN_REVIEW`：未进入审片
- `READY_FOR_REVIEW`：待提审
- `IN_DIRECTOR_REVIEW`：审片中
- `PENDING_FEEDBACK_FIX`：待处理反馈
- `FIX_UPDATED`：已按反馈修改
- `DIRECTOR_APPROVED`：内部通过

补充约定：

1. 二级状态只描述当前镜头版本的内部审片状态
2. 不单独设计“内部通过版本号”协议字段
3. 当前镜头 `versionNum` 变化后，不以“自动清空二级状态”作为默认策略，而是按本轮反馈是否已确认处理来继续推进
4. 当前镜头处于 `PENDING_FEEDBACK_FIX` 时，新版本产出本身不自动重置二级状态
5. 制作人员确认“本轮反馈已处理完成”后，才允许把状态推进到 `FIX_UPDATED`
6. `DIRECTOR_APPROVED` 的业务含义是“导演已手动确认该镜头当前处于通过态”
7. 已处于 `DIRECTOR_APPROVED` 的镜头，不会因为草稿任务或普通任务装配自动回退到 `IN_DIRECTOR_REVIEW`
8. 导演可主动撤销通过，将 `DIRECTOR_APPROVED` 回退到 `IN_DIRECTOR_REVIEW`，表示重新评审当前镜头
9. 若导演后续又对已通过镜头提交新反馈，该镜头重新进入 `PENDING_FEEDBACK_FIX`
10. 当前二级状态不是 `DIRECTOR_APPROVED` 时，镜头不允许正式提交
11. `submissionAllowed` 为服务端返回的只读判定结果，客户端不得自行放宽

### 4.1.3.4 镜头二级状态写接口

- `PUT /api/lenses/{id}/internal-review-status`
- Request: `LensInternalReviewStatusUpdateRequest`
- Response: `LensResponse`

`LensInternalReviewStatusUpdateRequest` 目标结构：

- `targetStatusCode: "NOT_IN_REVIEW" | "READY_FOR_REVIEW" | "IN_DIRECTOR_REVIEW" | "PENDING_FEEDBACK_FIX" | "FIX_UPDATED" | "DIRECTOR_APPROVED"`
- `reason?: string | null`
- `reviewTaskId?: Guid | null`

补充约定：

1. 本接口只用于镜头二级状态流转，不修改镜头主状态
2. 服务端必须校验目标状态是否合法，不能允许任意跳转
3. 服务端必须按角色限制可执行动作
4. 服务端返回更新后的完整 `LensResponse`，供列表与详情直接刷新

角色动作限制建议如下：

1. 制片可执行：
   - `NOT_IN_REVIEW -> READY_FOR_REVIEW`
   - `FIX_UPDATED -> READY_FOR_REVIEW`
2. 导演可执行：
   - `IN_DIRECTOR_REVIEW -> PENDING_FEEDBACK_FIX`
   - `IN_DIRECTOR_REVIEW -> DIRECTOR_APPROVED`
   - `DIRECTOR_APPROVED -> IN_DIRECTOR_REVIEW`
3. 制作人员可执行：
   - `PENDING_FEEDBACK_FIX -> FIX_UPDATED`

补充说明：

1. 制作人员不得通过本接口把镜头改成 `DIRECTOR_APPROVED`
2. 导演不负责把镜头推进到 `READY_FOR_REVIEW`
3. `READY_FOR_REVIEW -> IN_DIRECTOR_REVIEW` 的正式主驱动应是“任务提交成功后的系统联动”，而不是制片手工点状态主链路
4. `DIRECTOR_APPROVED -> IN_DIRECTOR_REVIEW` 的业务语义是“撤销通过并重新评审”，该动作不自动生成反馈记录
5. 制片不建议伪造 `PENDING_FEEDBACK_FIX`，该状态优先由导演反馈动作触发
6. 如服务端允许制片做纠偏回退，必须额外校验并写入操作日志

加入导演内部审片工作台后，审片协议建议扩展为任务、反馈卡片、图片附件三类接口：

- `POST /api/review-tasks`
- `GET /api/review-tasks`

- `GET /api/review-feedbacks/lens/{lensId}`
- `PUT /api/review-feedbacks/{id}`
- `DELETE /api/review-feedbacks/{id}`

补充约定：

1. `approve / reject / close` 是审片任务动作，不是镜头主状态值
2. 审片页“返修”按钮对应的协议动作仍为 `reject`
3. 镜头页“返修”必须走镜头状态流转协议，状态值统一为 `REWORK` / `返修`
4. 审片返修结果与镜头返修状态不可混用同一枚枚举
5. 镜头返修成功后，服务端必须回写并返回新的 `VersionNum`，客户端刷新时以服务端版本为准
6. 返修后的本地文件绑定同步必须跟随新的 `VersionNum`，禁止用旧版本号继续提交绑定
7. 导演内部审片反馈只驱动镜头二级状态，不直接改镜头主状态
8. 当前镜头处于 `PENDING_FEEDBACK_FIX` 时，新版本产出本身不自动清空为 `NOT_IN_REVIEW`；需待制作人员确认已处理后再推进到 `FIX_UPDATED`
9. 当前镜头二级状态不是 `DIRECTOR_APPROVED` 时，正式提交必须被服务端拒绝
10. `READY_FOR_REVIEW` 只表示镜头具备进入任务的条件，不表示导演已经收到任务
11. 导演任务列表的真相源必须是正式 `ReviewTask`
12. `IN_DIRECTOR_REVIEW` 的主驱动来源必须是“任务提交 / 任务开始”等正式派工动作，不允许仅把镜头单独改状态就视为已派工
13. 草稿任务只表示任务装配动作，不直接改写镜头二级状态
14. 正式提交任务时，只推动处于 `READY_FOR_REVIEW` / `FIX_UPDATED` 的镜头进入 `IN_DIRECTOR_REVIEW`
15. 已处于 `DIRECTOR_APPROVED` 的镜头不会因为加入草稿任务或普通任务提交而自动进入 `IN_DIRECTOR_REVIEW`
16. 审片任务中的任务镜头从本阶段起固定分为 `正式审片镜头` 与 `上下文陪审镜头` 两类
17. `上下文陪审镜头` 允许进入任务播放列表并使用 `layout` 或其他参考视频参与连播，但不因此进入镜头二级状态机
18. 正式提交任务或开始任务时，只允许对 `正式审片镜头` 执行 `READY_FOR_REVIEW` / `FIX_UPDATED` 到 `IN_DIRECTOR_REVIEW` 的自动推进
19. `上下文陪审镜头` 不允许通过导演工作台写入 `DIRECTOR_APPROVED`、`PENDING_FEEDBACK_FIX` 或正式反馈卡片
20. 任务级统计、完成判定、待处理数量、通过数量只统计 `正式审片镜头`，不得把 `上下文陪审镜头` 计入正式审片结论范围
21. 制片创建但尚未 `提交审片` 的任务，不得出现在导演任务列表中
22. 导演账号可见的任务前提是：该任务已经由制片正式提交，且尚未被制片关闭
23. 导演工作台必须提供显式 `审片完成` 任务动作；该动作与单镜头 `镜头通过`、`提交反馈` 分离
24. `审片完成` 的服务端放行前提是：当前任务下全部 `正式审片镜头` 都已获得正式结论；结论判定为“镜头已 `DIRECTOR_APPROVED`”或“镜头已存在正式反馈事实”
25. `上下文陪审镜头` 不参与 `审片完成` 的阻塞校验
26. 制片只允许关闭已完成的审片任务，不允许关闭尚未完成导演审片的任务
27. 任务关闭后，导演侧不得再返回该任务；即便该任务曾处于导演侧 `completed` 列表中，也必须同步移除
28. 任务关闭后，制片侧应将其归入 `已关闭审片任务列表`；该列表属于归档视图，不应继续占用默认主操作区
29. 任务关闭后默认保留反馈卡片、绘制时间线与镜头级反馈事实；如需物理删除，必须由制片执行额外手动清场动作
30. 制作人员不消费任务列表，但可在有权限的镜头详情页读取导演反馈内容，并对 `PENDING_FEEDBACK_FIX` 镜头执行 `FIX_UPDATED`
31. `layout` / `submitted-version` / `fallback-version` 属于播放源协议语义，`review` / `context` 属于任务镜头参与类型协议语义；两者不得互相代替
32. 客户端不得依据“当前播放源是 layout”直接推断该镜头必然可正式操作或必然是 `context`

### 6.0.1 第十九批导演审片工作台补充基线

本批次补充的正式协议口径如下：

1. 导演工作台的视频播放模型是“任务级播放列表”，以任务镜头顺序为真相源
2. 工作台必须支持镜头视频无缝连播；前端可通过预加载与自动切换实现，但不得要求导演手工逐镜头点播
3. 镜头队列激活状态必须绑定当前实际播放镜头，而不是只绑定点击选中项
4. 导演暂停后可在预览窗口进行绘制标注；绘制数据与文字反馈是同一镜头下的并行数据，不强制一一绑定
5. 绘制数据底层只按单帧存储；同一帧允许多个绘制对象，连续显示效果由客户端按继承链解释，不额外引入区间绘制协议字段
6. 反馈卡片允许先在客户端本地暂存，之后由“提交反馈”统一上传到服务器
7. 导演提交反馈时，除反馈卡片外，还应同步上传该镜头当前轮次的导演反馈绘制数据或等价遮罩产物
8. 单张反馈卡片上传后，只生成或更新反馈卡片实体，不自动改变任务最终结论
9. 任务最终结论仍由显式任务动作接口控制：`approve`、`reject`、`close`
10. 点击 `镜头通过` 时，只上传镜头通过状态，不上传空反馈卡片

### 6.0.2 导演反馈条目与绘制时间线正式语义

从本次协议版本开始，导演反馈链路正式拆分为两层：

1. `ReviewFeedback`：帧级反馈条目，用于表达某一帧的文字说明、决策、截图缩略图与附件
2. `DrawingTimeline`：轮次级绘制时间线，用于表达某一轮反馈在镜头时间轴上的正式绘制事件流

正式约定：

1. `ReviewFeedback.frameNumber` 只定义该条反馈条目的锚点帧，不定义其绘制是否向后续帧持续显示
2. 正式绘制是否显示，必须由该轮 `DrawingTimeline` 回放决定，而不是由“当前帧是否有 feedback 条目”决定
3. `feedbackRoundId` 是 feedback 条目与 timeline 的统一绑定键
4. 同一 `feedbackRoundId` 下必须只存在一条 canonical timeline 语义真相源
5. 客户端正式查看页必须优先消费轮次级 timeline；feedback 级 `drawingFrames` 仅作为历史兼容字段保留

补充约定：

1. 若当前回包中包含 `latestRound.drawingFrames`，则该字段正式等价于当前轮 canonical drawing timeline
2. 若查询指定历史轮次，也必须返回该轮完整 timeline，而不是要求客户端自行聚合该轮下每条 feedback 的旧字段
3. 若新版本服务端已返回 round 级 timeline，客户端不得再优先读取 `feedback.drawingFrames`
4. 历史兼容读取顺序固定为：`round timeline` -> `feedback.drawingFrames` -> `annotationDataJson`

### 6.0.3 导演反馈逐帧显示解释规则

正式绘制显示规则固定如下：

1. `DRAWN` 表示从该帧开始的当前可见绘制内容被替换为该帧绘制集合
2. `CLEAR` 表示从该帧开始清空当前可见绘制内容
3. 某一帧最终显示什么绘制，必须通过对当前轮 timeline 从头回放到该帧得出
4. 若某帧没有 feedback 条目，但 timeline 回放结果非空，则该帧仍应显示正式绘制
5. 若某帧有 feedback 条目，但 timeline 回放结果为空，则该帧只显示反馈条目，不显示绘制
6. 导演工作台草稿态允许存在本地未提交绘制保留链；但正式提交后，所有查看页都必须只认正式 timeline

### 6.1.1 审片任务列表与详情语义

`GET /api/review-tasks` 至少应返回：

1. 任务基础信息
2. 任务状态
3. 提交人
4. 目标导演
5. 镜头数量
6. 已反馈数量
7. 内部通过数量
8. 最近更新时间
9. 截止时间

`GET /api/review-tasks/{id}` 至少应返回：

1. 任务基础信息
2. 任务镜头队列
3. 每个镜头的顺序号
4. 每个镜头的提审版本
5. 每个镜头的实际播放版本
6. 每个镜头当前二级状态
7. 每个镜头反馈数量
8. 每个镜头最近反馈时间

补充约定：

1. 任务详情中的每个任务镜头都应返回其任务内参与类型，正式字段为 `reviewParticipationMode: "review" | "context"`
2. `review` 表示正式审片镜头，参与二级状态自动推进、导演反馈、导演通过、任务统计与完成判定
3. `context` 表示上下文陪审镜头，只参与任务排序、播放与前后镜头衔接，不参与正式审片结论
4. 历史兼容阶段允许服务端额外返回 `participationMode` 作为旧字段别名；但新增客户端与新增接口文档都必须以 `reviewParticipationMode` 作为正式字段名
5. `context` 镜头即使存在 `layout` 或其他参考素材，也不得被客户端当作“待导演操作镜头”
6. 官方客户端在制片新增任务镜头时，必须按素材匹配规则显式传入参与类型：当前版本视频已匹配时传 `review`；若当前镜头本轮仅依赖 `layout` 视频补位连播，则传 `context`
7. 该“默认参与类型”规则属于客户端装配规则，不属于服务端自动判定规则；服务端只消费客户端最终提交值
8. 若请求未显式传入参与类型，服务端兼容默认按 `reviewParticipationMode = review` 入库；该兜底仅用于兼容旧客户端，不视为正式业务默认规则
9. 导演工作台若收到 `reviewParticipationMode == context` 的任务镜头，必须视为只读播放项，不得展示正式反馈或正式结论操作

### 6.1.2 审片任务新增动作协议

除已有任务接口外，后续正式基线补充以下动作：

- `POST /api/review-tasks/{id}/start`
- `POST /api/review-tasks/{id}/complete`

补充约定：

1. `start` 用于导演正式开始任务，不能由非导演越权触发
2. `complete` 用于任务级完结动作，不等价于单镜头内部通过
3. 若实现阶段尚未完整落地上述动作接口，必须在代码和文档中明确兼容策略，不能静默缺失
4. `submit` / `start` 触发的镜头二级状态自动推进，只作用于 `reviewParticipationMode == review` 的任务镜头
5. 若任务中同时包含 `review` 与 `context` 镜头，服务端必须保证上下文镜头不会被误推进到 `IN_DIRECTOR_REVIEW`
6. `complete` 之后，任务默认进入“已完成但不再作为默认待办展示”的收尾态；当前阶段不要求立即提供物理删除路由
7. `close` 由制片执行时，若当前项目采用“关闭即清空内部反馈”的口径，服务端必须物理删除该任务下全部 `review_comments` 与关联绘制事实
8. `close` 完成后，任务内 `reviewParticipationMode == review` 的镜头二级状态统一回退到 `READY_FOR_REVIEW`
9. `close` 完成后，任务内 `reviewParticipationMode == context` 的镜头不要求额外状态回退
10. 正式客户端中，导演工作台的 `审片完成` 只能调用 `POST /api/review-tasks/{id}/complete`；`POST /api/review-tasks/{id}/close` 仅供制片关闭已完成任务使用
11. 服务端必须拒绝导演以 producer close 语义结束未完成任务，不能因空请求体或缺少 `rowVersion` 而静默放行错误分支

### 6.1.2.1 审片任务完成后的可见性与删除语义

1. `POST /api/review-tasks/{id}/complete` 的正式语义是“结束本轮任务”，不是“删除任务记录”
2. `completed` / `closed` 任务默认可以从导演与制片的主待办列表中隐藏，但历史详情、反馈回读、统计追溯仍应可访问
3. 当前阶段若页面需要“完成后消失”的体验，应通过列表过滤或归档可见性实现，而不是通过物理删除数据库记录实现
4. 当前阶段不建议新增 `DELETE /api/review-tasks/{id}` 作为正式基线路由
5. 若未来确需增加物理删除，必须先明确其对 `review_comments`、绘制数据、`latestReviewTaskId`、任务统计、审计日志的影响与迁移策略
6. 若当前项目明确采用“关闭任务即清空内部反馈”的业务规则，则 `POST /api/review-tasks/{id}/close` 可以承担任务级反馈物理清理职责，但任务实体本身仍保留 `closed` 状态记录
7. 执行上述清理后，按当前任务维度回读反馈必须返回空数组；同镜头后续重新进入新任务时，不得继续看到已关闭任务的旧反馈残留

### 6.2 导演反馈卡片协议

`ReviewFeedbackResponse` 建议结构：

- `id: Guid`
- `feedbackRoundId?: Guid | null`
- `projectCode: string`
- `reviewTaskId: Guid`
- `lensId: Guid`
- `lensCode: string`
- `versionNum?: string | null`
- `frameNumber?: number | null`
- `timestampSeconds?: number | null`
- `timecode?: string | null`
- `commentText?: string | null`
- `frameImagePath?: string | null`
- `annotatedImagePath?: string | null`
- `thumbnailPath?: string | null`
- `annotationDataJson?: string | null`
- `feedbackStatusCode?: "draft" | "submitted" | null`
- `createdByUserId: Guid`
- `createdByDisplayName?: string | null`
- `createdAtUtc: DateTimeOffset`
- `updatedAtUtc?: DateTimeOffset | null`

补充约定：

1. 响应体正文统一以 `content` 为准；`commentText` 仅作为创建/更新请求字段使用。
2. `frameNumber` 为空但 `timecode` 有值时，视为历史脏数据兼容场景；客户端应优先用 `frameNumber`，为空时再用 `timecode` 判断帧级定位。
3. 同一条反馈在列表接口与详情接口中应保持相同的 `reviewTaskId`、`taskShotId`、`feedbackRoundId` 和正文语义。

`CreateReviewFeedbackRequest` 建议结构：

- `feedbackRoundId?: Guid | null`
- `reviewTaskId: Guid`
- `lensId: Guid`
- `taskShotId?: Guid | null`
- `versionNum?: string | null`
- `frameNumber?: number | null`
- `timestampSeconds?: number | null`
- `timecode?: string | null`
- `commentText?: string | null`
- `frameImageFileToken?: string | null`
- `annotatedImageFileToken?: string | null`
- `thumbnailFileToken?: string | null`
- `annotationDataJson?: string | null`

补充约定：

1. 一条反馈卡片对应导演在某一时间点保存的一次文字反馈；可带图片与标注，但不强制必须有绘制
2. `versionNum` 表示这条反馈对应的镜头版本
3. 保存反馈成功后，服务端应同步刷新镜头的 `latestDirectorFeedbackAtUtc`、`pendingDirectorFeedbackCount`
4. 只要保存了有效反馈卡片或有效绘制反馈，服务端应将镜头二级状态推进为 `PENDING_FEEDBACK_FIX`
5. 当导演执行“镜头通过”动作时，不通过反馈创建接口隐式完成，仍建议走镜头二级状态写接口
6. 保存反馈成功后，服务端还应同步刷新任务镜头反馈数量、最近反馈时间与任务摘要统计
7. `taskShotId` 为可选增强字段；若传入，服务端应校验其确实属于 `reviewTaskId` 且与 `lensId` 一致
8. `frameNumber`、`timestampSeconds`、`timecode`、`annotationDataJson` 至少应允许部分缺省；只要请求中存在有效文字说明或有效绘制数据，就视为有效反馈提交
9. 当前批次允许客户端以本地绝对路径字段临时上传反馈资产：`frameImagePath`、`annotatedImagePath`、`thumbnailPath`；后续统一文件上传协议时再收口到 file token
10. 当前一次迭代口径允许文字反馈不带任何绘制，也允许仅提交有效绘制而不带文字；但不允许提交“无文字且无绘制”的空反馈
11. `feedbackStatusCode` 当前回读以 `submitted` 为正式事实；`draft` 只作为未来扩展保留，当前客户端本地草稿不要求服务端持久化
12. 同一次“提交反馈”动作产生的多条反馈卡片，应共享同一个 `feedbackRoundId`，用于把本轮文字反馈与本轮绘制遮罩聚合成同一轮正式反馈结果
13. `feedbackRoundId` 必须同时作为反馈卡片回读与绘制帧回读的聚合键；服务端回包不得把本地草稿轮次当成正式轮次
14. 当前接口层已显式固定边界：`POST /api/review-feedbacks` 只负责提交反馈卡片，不接收状态字段；`POST /api/review-tasks/{id}/approve` / `reject` / `close` 才是任务结论动作，其中 `approve` 只走通过语义，服务端固定把提交反馈后的镜头推进到 `PENDING_FEEDBACK_FIX`
15. 同一镜头同一帧默认只允许存在一条有效反馈事实；若导演对该帧已有已提交反馈再次编辑，客户端应优先调用 `PUT /api/review-feedbacks/{id}` 执行更新，而不是再次创建同帧重复反馈
16. 若导演删除已提交反馈，客户端应调用 `DELETE /api/review-feedbacks/{id}`；删除后的正式反馈列表回读结果中不再出现该项
17. `GET /api/review-feedbacks/lens/{lensId}` 的正式语义必须严格按镜头实体 id 隔离；返回数组中的每条反馈都必须满足其 `lensId == 路由中的 lensId`
18. 服务端不得使用“该反馈所属任务包含当前镜头”作为按镜头回读条件；否则会把同一任务内其他镜头反馈误并入当前镜头结果
19. 若为兼容历史数据需要放宽匹配，唯一允许的兼容方式是通过 `taskShotId -> ReviewTaskShot.LensId == lensId` 做一对一归属确认；不允许按任务范围模糊归并
20. 若目标任务镜头被标记为 `reviewParticipationMode == context`，服务端必须拒绝创建、更新或删除该镜头的正式导演反馈，并返回可读业务错误

补充说明：上述图片路径字段在草稿编辑态可指向本机文件，但正式回读态必须是服务端可访问地址；客户端不得把 `file://`、本机绝对路径与服务端 `/uploads/...` 视为同一种语义。

### 6.2.0.1 导演反馈批量提交语义

当前批次不强制新增服务端“批量提交反馈”接口，允许客户端逐条调用：

- `POST /api/review-feedbacks`

补充约定：

1. “提交反馈”是客户端任务级交互，不等价于服务端单独一条批量路由
2. 客户端可按顺序逐条创建反馈卡片；若中途失败，未成功项应保留本地草稿，供用户重试
3. 服务端在单条反馈创建成功后，应保证该反馈可立即从 `GET /api/review-feedbacks/lens/{lensId}` 回读
4. 客户端触发“提交反馈”前，当前镜头必须至少存在一条有效反馈事实；有效反馈事实可以是文字反馈、绘制反馈，或两者同时存在
5. “保存为草稿”当前仅是客户端本地行为，不单独占用正式协议动作
6. 当前阶段 `AnnotationDataJson` 仍承载绘制聚合载荷，属于第三阶段待收口项；后续应拆出独立绘制协议后再收束该字段语义
7. 同一轮正式提交的多条反馈必须共享同一个 `feedbackRoundId`，且 `GET /api/review-feedbacks/lens/{lensId}?feedbackRoundId=...` 需按该轮回读全部反馈与全部绘制帧；未显式传入时默认回读最近一轮
8. `CLEAR` 帧是稳定清屏锚点，回读时 `drawingObjectsJson` 必须保持空值，不能被服务端重写为占位对象
9. 若客户端提交时既无有效文字反馈，也无有效绘制数据，服务端必须拒绝并返回可读业务错误码，避免空反馈成为正式事实
10. 若客户端提交时仅包含有效绘制数据而无文字反馈，服务端应允许创建或更新反馈，并保持 `feedbackRoundId`、`drawingFrames`、`frameNumber`、`timecode` 等上下文可稳定回读
11. 客户端“提交反馈”前，可同时存在三类待提交变化：新增反馈、更新既有反馈、删除既有反馈；服务端需分别通过 `POST`、`PUT`、`DELETE` 提供等价能力
12. 多镜头同任务下，`GET /api/review-feedbacks/lens/{lensId}` 与 `GET /api/review-feedbacks/lens/{lensId}/drawings` 都必须只返回当前镜头数据，不能把同任务其他镜头的反馈或绘制一并回读

### 6.2.0.2 导演绘制遮罩协议语义

当前阶段，导演绘制遮罩作为镜头级时间轴数据，与反馈卡片一起提交并供镜头详情回放消费。

建议新增或等价提供以下能力：

- `PUT /api/review-feedbacks/lens/{lensId}/drawings`
- `GET /api/review-feedbacks/lens/{lensId}/drawings`

`ReviewDrawingFrameResponse` 建议结构：

- `drawingFrameId: Guid`
- `feedbackRoundId?: Guid | null`
- `reviewTaskId: Guid`
- `lensId: Guid`
- `taskShotId?: Guid | null`
- `versionNum?: string | null`
- `frameNumber?: number | null`
- `timestampSeconds?: number | null`
- `timecode?: string | null`
- `drawingStateCode: "DRAWN" | "CLEAR"`
- `drawingObjectsJson?: string | null`
- `createdByUserId: Guid`
- `createdByDisplayName?: string | null`
- `createdAtUtc: DateTimeOffset`
- `updatedAtUtc?: DateTimeOffset | null`

`UpsertReviewDrawingFramesRequest` 建议结构：

- `feedbackRoundId?: Guid | null`
- `reviewTaskId: Guid`
- `lensId: Guid`
- `taskShotId?: Guid | null`
- `versionNum?: string | null`
- `frames: ReviewDrawingFrameUpsertItem[]`

`ReviewDrawingFrameUpsertItem` 建议结构：

- `frameNumber?: number | null`
- `timestampSeconds?: number | null`
- `timecode?: string | null`
- `drawingStateCode: "DRAWN" | "CLEAR"`
- `drawingObjectsJson?: string | null`

补充约定：

1. 每条绘制帧记录只对应单帧，不直接定义多帧区间字段
2. 同一帧允许存在多个绘制对象，但服务端回包与存储以该帧绘制集合作为单条记录
3. `drawingStateCode = CLEAR` 表示当前帧为清屏锚点，用于打断此前绘制继承链
4. 客户端回放遮罩时，应按“向前查找最近有效控制点”的方式解释显示结果；若最近控制点是 `DRAWN` 则显示对应绘制，若是 `CLEAR` 则不显示
5. 客户端提交绘制遮罩时，应按当前镜头维度一次性上传“当前轮次全部本地绘制数据”，服务端不要求理解前端本地增量编辑过程
6. 当前阶段镜头详情页只回看正式已提交绘制遮罩，且只显示当前轮已提交反馈对应的遮罩内容，不回读客户端本地草稿绘制
7. 当前轮反馈提交成功后，客户端应清空该镜头本地反馈列表与本地绘制数据，开始下一轮空白编辑态
8. 若实现阶段不单独新增绘制路由，也必须提供等价请求体或组合动作，保证“反馈卡片提交 + 镜头级绘制提交 + 镜头状态提交”三者边界清晰
9. `feedbackRoundId` 用于标识同一轮反馈提交；镜头详情页默认只消费当前镜头最近一轮正式提交的 `feedbackRoundId` 对应反馈列表与绘制遮罩
10. 绘制落库必须按单帧记录，不定义区间型存储模型；`DRAWN` 表示单帧绘制，`CLEAR` 表示单帧清屏锚点
11. 若服务端收到仅包含绘制而没有文字反馈的正式提交，必须拒绝并返回可读业务错误码
12. 当前帧若命中已提交绘制记录，客户端视频预览层应先显示该帧对应正式绘制；若导演在该帧继续编辑，则本地未提交绘制应实时覆盖当前预览，提交后再成为新的正式事实
13. 若目标任务镜头被标记为 `reviewParticipationMode == context`，服务端不得为其创建或回写正式绘制遮罩事实

### 6.2.1 镜头详情中的任务回挂语义

镜头详情在回读二级状态字段之外，还应逐步具备以下任务回挂字段能力：

- `latestReviewTaskId?: Guid | null`
- `latestReviewTaskName?: string | null`
- `reviewTaskSummaries?: ReviewTaskSummaryResponse[] | null`

`ReviewTaskSummaryResponse` 建议结构：

- `taskId: Guid`
- `taskName: string`
- `taskStatusCode: string`
- `taskStatusName?: string | null`
- `submitVersionNum?: string | null`
- `sequence?: number | null`

补充约定：

1. 镜头详情中的任务回挂能力用于支持客户端从镜头定位最近任务或查看所属任务历史
2. `latestReviewTaskId` 仍表示当前镜头最近一次进入的审片任务
3. 若当前实现尚未补齐任务摘要数组，至少应保证 `latestReviewTaskId` 稳定可回读

### 6.2.2 镜头详情中的导演反馈分页语义

镜头详情中的导演反馈不再建议悬浮在详情主区单独编辑，而应作为与 `记录` 并列的独立分页回看区。

分页语义约定：

1. `记录` 分页承接镜头主状态链路中的返修记录、返修说明、返修图片与生命周期事件
2. `导演反馈` 分页承接导演内部审片链路中的反馈卡片
3. 两个分页是并列关系，不得混写成一个泛化反馈区

`导演反馈` 分页建议至少展示以下字段：

- `feedbackId`
- `reviewTaskId`
- `versionNum`
- `frameNumber`
- `timecode`
- `commentText`
- `frameImagePath`
- `annotatedImagePath`
- `thumbnailPath`
- `createdAtUtc`
- `createdByDisplayName`

`导演反馈` 分页的视频预览区建议同时支持以下遮罩回放能力：

- `hasDirectorDrawingMask?: boolean`
- `directorDrawingMaskVersionNum?: string | null`
- `directorDrawingFrames?: ReviewDrawingFrameResponse[] | null`

补充约定：

1. `导演反馈` 分页的阅读模型应与 `记录` 分页保持一致，允许以卡片流或时间线形式展示图文反馈
2. 协议层不要求镜头详情接口直接内嵌完整反馈数组；客户端可继续通过 `GET /api/review-feedbacks/lens/{lensId}` 拉取数据后挂入该分页
3. 只要客户端最终呈现为镜头详情内与 `记录` 并列的 `导演反馈` 分页，即视为符合当前页面语义
4. `导演反馈` 分页中的视频遮罩开关默认应针对“已提交导演绘制遮罩”生效，不显示未提交本地绘制

### 6.2.3 镜头详情中的返修图片持久化与预览语义

第十九批完成后，一级返修记录中的图片不允许再只依赖上传机器本地文件。

必须满足以下正式语义：

1. 制片在返修弹窗中上传或粘贴的图片，保存返修记录时必须上传到服务器
2. 服务端必须持久化保存返修图片文件与附件元数据
3. 导演、制作人员、制片重新登录后，都可在镜头详情 `记录` 分页中展开返修记录看到图片预览
4. 点击返修图片缩略图时，客户端必须能拿到可放大查看的正式图片地址

返修记录中的附件建议结构：

- `attachmentId: Guid`
- `fileName: string`
- `contentType?: string | null`
- `fileSize?: long | null`
- `sortOrder: int`
- `previewUrl: string`
- `fullImageUrl?: string | null`
- `createdAtUtc: DateTimeOffset`

补充约定：

1. `previewUrl` 用于记录详情折叠区直接展示缩略预览
2. `fullImageUrl` 用于点击后放大查看；若当前实现允许原图直出，可与 `previewUrl` 指向同一资源
3. 不允许要求导演或制作人员依赖制片机器本地路径才能查看返修图片
4. 返修记录编辑时，客户端允许保留既有附件、移除既有附件、追加新附件，服务端必须稳定处理排序与删除语义

补充约定：`previewUrl`、`fullImageUrl`、`frameImagePath`、`annotatedImagePath`、`thumbnailPath` 等图片字段在回读时都应解析为可访问地址；若服务端返回的是 `/uploads/...`，客户端必须按静态资源地址展示，不得再拼成本地 `file://`。

补充约定：镜头详情 `记录` 分页与 `导演反馈` 分页均应使用同一图片地址解析语义；本地草稿仅用于编辑态预览，正式读回必须是服务端可访问 URL。

### 6.3 文件上传协作约定

由于导演反馈需要保存原始帧图、标注图和缩略图，且一级返修记录需要保存返修图片附件，协议补充如下：

1. 客户端先通过既有或新增上传接口上传文件，拿到 `fileToken`
2. 再在 `CreateReviewFeedbackRequest` 中提交 `frameImageFileToken`、`annotatedImageFileToken`、`thumbnailFileToken`
3. 服务端负责把 `fileToken` 绑定为正式反馈附件并落库
4. 协议主契约不传客户端绝对路径
5. 若当前实现仍沿用 `frameImagePath / annotatedImagePath / thumbnailPath` 兼容字段，则它们在正式回读侧必须表现为服务端可访问地址

若当前阶段尚未统一上传接口，可先临时约定：

- `POST /api/uploads/review-feedback-assets`

若要同时覆盖一级返修记录图片，可补充或统一为：

- `POST /api/uploads/lens-record-assets`
- 或统一使用 `POST /api/uploads/assets` 并通过业务类型区分

返回结构建议：

- `fileToken: string`
- `fileName: string`
- `contentType?: string | null`
- `fileSize?: long | null`

补充约定：

1. 一级返修记录图片与导演反馈图片都可以共用统一上传底座，但服务端落库时必须分开绑定到各自业务实体

## 10. 第七批文件绑定同步协议（草案）

### 10.1 目标

第七批的目标不是把文件能力迁移到服务端，而是建立以下闭环：

1. 客户端本地完成镜头文件绑定
2. 客户端生成绑定元数据
3. 客户端将绑定结果同步到服务端
4. 服务端保存相对路径与绑定元数据
5. 镜头详情真实回读服务端绑定结果

### 10.2 职责边界

客户端负责：

1. 本地文件扫描
2. 本地绑定动作发起
3. 生成 `relativePath`
4. 从客户端路径映射 / 根目录配置中生成 `sourceRoot`（逻辑根标识，不得是绝对路径）
5. 决定 `bindingType`
6. 补充 `versionNum`（若适用）
7. 将绑定结果同步到服务端
8. 基于路径映射解析本地打开/预览能力

补充约定：项目激活时，客户端可将服务端记录的本地绝对路径按 Windows 盘符差异重定位到当前机器，只要盘符后的相对路径一致即可；若无法重定位，则提示用户补齐本机路径。

服务端负责：

1. 接收绑定同步请求
2. 保存绑定元数据
3. 返回镜头详情中的真实绑定结果
4. 做权限校验、冲突校验、覆盖更新语义控制

第七批明确不做：

1. 服务端不做客户端磁盘扫描
2. 服务端不保存客户端绝对路径作为主契约
3. 服务端不承担本地文件打开 / 预览能力

### 10.3 文件绑定同步接口

- `POST /api/lenses/{lensId}/bindings`
- Request: `LensFileBindingSyncRequest`
- Response: `LensFileBindingSyncResponse`

补充约定：

1. `lensId` 由路由提供，服务端在持久化时必须同时保存对应 `lensCode`
2. `lensCode` 由镜头主数据解析得到，不作为客户端主契约字段强制传入
3. 服务端返回的同步结果必须能直接用于详情回读

`LensFileBindingSyncRequest` 目标结构：

- `bindingType: "ma" | "mov" | "layout" | "layoutVideo"`
- `relativePath: string`
- `sourceRoot?: string | null`
- `versionNum?: string | null`
- `fileName?: string | null`

补充约定：

1. `relativePath` 为服务端长期保存的主路径字段
2. `relativePath` 必须是相对路径，不允许上传客户端绝对路径
3. `sourceRoot` 用于标识来源根目录，必须是客户端可识别的逻辑根标识（例如 `lens-root-main`），不得上传本地绝对路径
4. `versionNum` 对版本型绑定（如 `ma` / `mov`）可选但推荐传入
5. `fileName` 作为展示辅助字段，可由客户端传入，也可由服务端从 `relativePath` 推导
6. 绑定同步接口按 **upsert** 语义实现，而不是“盲插入”
7. `ma` / `mov` 这类版本型绑定，以 `lensId + bindingType + versionNum` 作为唯一业务键
8. `layout` / `layoutVideo` 这类非版本型绑定，以 `lensId + bindingType` 作为唯一业务键
9. 若上传请求与现有记录在唯一业务键上命中，且 `relativePath + sourceRoot + fileName` 完全一致，则视为**幂等成功**，返回既有绑定记录，不报错
10. 若上传请求与现有记录在唯一业务键上命中，但 `relativePath` 或其他核心绑定字段发生变化，则视为**覆盖更新**，服务端更新既有绑定记录并返回更新后的绑定结果
11. 第七批默认不支持同一唯一业务键下保留多条并行有效绑定；若后续要支持多绑定并存，必须先更新本文件再改实现

示例：

```json
{
  "bindingType": "mov",
  "relativePath": "EP01/SC001/ANI/SC001_V01.mov",
  "sourceRoot": "lens-root-main",
  "versionNum": "V01",
  "fileName": "SC001_V01.mov"
}
```

`LensFileBindingSyncResponse` 目标结构：

- `bindingId: string`
- `lensId: string`
- `lensCode: string`
- `bindingType: string`
- `relativePath: string`
- `sourceRoot?: string | null`
- `versionNum?: string | null`
- `fileName?: string | null`
- `bindTime: string`

### 10.4 镜头详情回读协议

镜头详情中的绑定信息应统一体现为：

- `bindingId: string`
- `lensId: string`
- `lensCode: string`
- `bindingType: string`
- `fileName?: string | null`
- `relativePath: string`
- `sourceRoot?: string | null`
- `versionNum?: string | null`
- `bindTime?: string | null`

补充约定：

1. 服务端详情接口返回的绑定字段应逐步从旧的 `filePath/localPath/foundFileName` 语义收敛到以 `relativePath` 为核心
2. 客户端在展示协同 binding 信息时，应优先消费服务端返回的 `relativePath`
3. 客户端本地打开/预览文件时，应通过 `sourceRoot + relativePath` 结合路径映射解析成本地绝对路径
4. `LayoutCandidates` / `CurrentLayout` / `LayoutReferenceCheck` 不应被当成服务端提供的本地预览结论；若服务端没有真实协同数据来源，可返回空数组或 `null`，由客户端本地能力补齐

### 10.5 覆盖 / 幂等 / 删除语义

1. **幂等成功**
   - 同一唯一业务键下重复提交完全相同的绑定请求
   - 服务端返回已有绑定记录
   - 不应报业务冲突
2. **覆盖更新**
   - 同一唯一业务键下提交新的 `relativePath` / `sourceRoot` / `fileName`
   - 服务端更新原有绑定并返回最新记录
   - `bindTime` 应更新为最近一次成功绑定时间
3. **删除能力**
   - 删除语义采用**显式删除接口**，不复用 binding sync 做 tombstone
   - 路由：`DELETE /api/lenses/{lensId}/bindings`
   - Query 参数：
     - `bindingType`（必填）
     - `versionNum`（版本型绑定在需要精确定位时可传）
   - 唯一业务键规则保持不变：
     - `ma` / `mov`：`lensId + bindingType + versionNum`
     - `layout` / `layoutVideo`：`lensId + bindingType`
   - 删除命中既有 binding：服务端删除该记录并返回成功
   - 删除未命中既有 binding：按**幂等成功**处理，不报错
   - 当前协议不采用 tombstone / removedAt 持久化语义，避免污染第七批既有 upsert 契约

### 10.6 错误语义

1. `relativePath` 为空：返回参数校验错误
2. `bindingType` 非法：返回参数校验错误
3. `lensId` 不存在：返回未找到错误
4. 非项目成员调用：返回授权失败
5. 唯一业务键冲突但无法完成幂等或覆盖更新：返回业务冲突
6. 绝对路径上传：返回参数校验错误
7. 删除接口缺少 `bindingType`：返回参数校验错误
8. 删除版本型 binding 时若业务要求精确定位但 `versionNum` 缺失：返回参数校验错误

### 10.7 联调要求

1. 客户端绑定请求必须与本节保持一致
2. 服务端持久化结果必须保留 `relativePath` 语义
3. 远程详情回读必须能反映服务端真实保存的绑定结果
4. 若第七批绑定协议发生变化，必须先更新本文件，再改代码

## 11. 第九批刷新全部文件匹配同步协议（草案）

### 11.1 目标

第九批的目标不是重写旧项目文件匹配算法，而是建立以下闭环：

1. 客户端点击“刷新全部文件匹配”按钮
2. 客户端按旧项目规则重算当前集镜头的文件匹配结果
3. 客户端将刷新后的真实绑定结果同步到服务端
4. 服务端保存并回读这些结果
5. 镜头详情页以服务端真实回读结果作为协同基线，再由客户端当前机器补齐本地文件存在性、路径解析与视频预览信息

### 11.2 职责边界

客户端负责：

1. 本地扫描 MA / MOV 文件
2. 批量刷新当前集镜头的自动匹配结果
3. 保留并消费当前已有的 Layout candidate / selected layout 信息
4. 将刷新后的文件绑定结果同步到服务端
5. 刷新完成后重新拉取镜头列表 / 详情，保证页面以服务端回读作为协同事实基线
6. 继续在当前机器执行路径映射、本地文件存在性判断、FFmpeg / FFprobe 探测与视频预览生成
7. 若发现历史镜头 `singleFrame = 0` 且已存在可用 Layout 视频绑定，客户端可在本地执行一次性帧数补偿初始化，并将结果同步到服务端

服务端负责：

1. 接收第九批刷新后同步过来的绑定结果
2. 按第七批 binding sync 契约保存或更新绑定记录
3. 在镜头列表 / 详情中回读真实绑定结果
4. 若第九批决定同步文件检查摘要，则按新增协议承接并返回摘要
5. 不负责客户端本地绝对路径解析、媒体探测、视频预览生成或本地文件存在性判定

第九批明确不做：

1. 服务端不做客户端磁盘扫描
2. 客户端“刷新全部文件匹配”不重扫 Layout candidate
3. 协议未定义前，不新增不透明的服务端摘要字段
4. 不把 FFmpeg、FFprobe、路径映射、视频预览迁移到服务端
5. 服务端不以自身逻辑根据 Layout candidate / LayoutVideo binding 自动生成或覆盖镜头帧数

### 11.3 “刷新全部文件匹配”旧项目对齐规则

第九批必须以下列旧项目真实行为作为唯一对齐基线：

1. 按钮入口位于镜头列表页
2. 批量刷新主要重算当前集镜头的：
   - `ma` 当前版本绑定
   - `mov` 当前版本绑定
3. 批量刷新会更新本地文件检查结果与绑定结果
4. 批量刷新不会重新扫描 Layout candidate
5. Layout 状态来自当前已有 candidate / 当前选中 candidate，而不是本次批量刷新新发现的候选

### 11.4 Layout 文件匹配规则（旧项目对齐基线）

客户端在本地执行 Layout 文件匹配时，必须按以下规则实现：

1. 输入前缀集合：
   - `lensCode`
   - `lensName`
2. 所有前缀匹配前统一做：
   - `trim`
   - 转大写
3. `layoutTag` 统一按命名标签规范化，默认值为 `LAY`
4. Layout 文件候选命中条件：满足以下任一条件即可进入候选集
   - 文件名主干 `baseName === prefix`
   - 文件名主干 `baseName === ${prefix}_${layoutTag}`
   - 文件名主干 `baseName.startsWith(${prefix}_)`
   - 文件名主干 `baseName.startsWith(${prefix}_${layoutTag}_)`
   - 文件路径分段中包含 `prefix`
5. 候选排序顺序必须为：
   - `scoreLayoutCandidate` 分数高者优先
   - 版本号高者优先
   - 命名优先级高者优先
   - `sourcePriority` 小者优先
   - 最后按文件名中文排序
6. Layout 命名优先级必须按以下顺序：
   - `${prefix}_${layoutTag}`
   - `${prefix}_${layoutTag}_*`
   - `${prefix}_*`
   - `${prefix}`

### 11.5 Layout 视频匹配规则（旧项目对齐基线）

客户端在本地执行 Layout 视频匹配时，必须按以下规则实现：

1. 先将 `lensCode` 规范化为大写作为主匹配键
2. 若当前镜头存在 `selectedLayoutFileName`，优先按已选 Layout 文件名进行视频匹配
3. `selectedLayoutFileName` 存在时，优先级如下：
   - 视频文件名主干与 selected layout 文件名主干完全一致：最高优先级
   - 去掉版本号后的视频主干与去掉版本号后的 selected layout 主干一致：次高优先级
4. 若未命中 selected layout 文件名，则候选视频必须同时满足：
   - 文件名中能独立命中 `lensCode`
   - 去掉版本号后的主干中包含 `LAY` 或 `LAYOUT` 关键字
5. Layout 视频基础命名优先级必须按以下顺序：
   - `${lensCode}_LAY`
   - `${lensCode}_LAYOUT`
   - `${lensCode}_LAY_*`
   - `${lensCode}_LAYOUT_*`
   - `${lensCode}_*`
   - `${lensCode}`
6. 视频候选排序顺序必须为：
   - 匹配分数高者优先
   - 版本号高者优先
   - 命名优先级高者优先
   - `sourcePriority` 小者优先
   - 最后按文件名中文排序
7. 版本号在视频匹配中仅作为加分项，不改变“必须先命中 lensCode + layout 关键字”的前提

### 11.6 第九批与第七批 binding sync 的关系

1. 第九批默认复用第七批 `POST /api/lenses/{lensId}/bindings` upsert 协议同步刷新后的绑定结果
2. 若客户端刷新后需要同步的仅是绑定结果，则不得额外新增新接口
3. 若第九批需要同步 `file_check` 摘要（例如 `overallStatus` / `layoutCandidateCount`），必须先在本文件新增对应接口与字段定义，再开始开发
4. 第九批不得破坏第七批既有唯一业务键、幂等成功、覆盖更新语义
5. 当前第九批服务端实现仅承接 binding 刷新结果，不新增 `file_check` 摘要持久化；列表页仅回读 binding 摘要，详情页回读完整 binding 记录
6. 第九批服务端回读的是 binding 协同事实，不包含“当前机器是否存在文件 / 是否可预览 / 预览 URL / 本地媒体元数据”的承诺

### 11.9 第九批本地刷新结果回传粒度

1. 客户端在本地完成 `refreshLensBindings` 后，必须读取最新本地状态，再筛选当前批次的镜头结果上行服务端
2. 当前第九批允许同步的最小结果集为：
   - `ma` 绑定
   - `mov` 绑定
   - 当前已选中的 `layout` 绑定（若存在）
   - 当前已选中的 `layoutVideo` 绑定（若存在）
3. 第九批当前不新增 `file_check` 摘要回传协议；`overallStatus` / `layoutCandidateCount` 仍以本地状态为准
4. 第九批若本地刷新后发现某个 binding 已消失，客户端必须调用显式删除接口同步服务端状态：
   - `DELETE /api/lenses/{lensId}/bindings?bindingType=...&versionNum=...`
5. 第九批删除 / 清空 binding 统一采用显式删除接口，不采用 tombstone / removedAt 语义

### 11.7 镜头详情页测试基线

镜头详情页在第九批后的测试基线必须满足：

1. MA / MOV / Layout / LayoutVideo 绑定信息优先读取服务端真实回读结果
2. 详情页展示的 Layout 文件与 Layout 视频必须符合本节 `11.4` 和 `11.5` 的旧项目对齐规则
3. 详情页必须在服务端 binding 回读基础上，再由客户端当前机器补齐本地文件存在性、绝对路径、预览地址与媒体元数据；本地 SQLite 只能补齐，不能覆盖服务端已回读的 binding 事实
4. 服务端有 binding，不等于当前机器一定可以打开或预览；当前机器是否可预览必须由本地扫描结果、路径映射结果与本地预览链路共同决定
5. 若客户端本地刷新后未完成服务端同步，则第九批验收不得判定详情页闭环通过
6. 详情页测试不得把“本地 SQLite 看起来有数据”视为服务端闭环成立，也不得把“服务端有 binding”视为本机一定可预览
7. 镜头详情页中的 `记录` 与 `导演反馈` 必须作为并列分页理解：前者承接返修记录与生命周期事件，后者承接导演反馈卡片
8. `导演反馈` 分页中的反馈卡片必须能看到文字说明与图片产物，不得退化成只有 ID 或纯文本摘要的列表
9. `记录` 分页中的返修记录在重新登录、换角色、换机器后，仍必须能看到服务端返回的图片预览，不得依赖上传机器本地文件路径
10. 点击返修图片缩略图时，客户端必须能通过服务端回读地址打开放大预览

### 11.10 协同链路下的本地文件能力补齐规则

1. 后续协议与派工不再把“远程模式 / 本地模式”作为正式业务概念；统一表述为“服务端协同主数据 + 客户端本地文件能力”
2. 服务端详情接口返回的 binding 字段用于表达协同事实，最小包含：
   - `bindingType`
   - `relativePath`
   - `sourceRoot`
   - `versionNum`
   - `fileName`
   - `bindTime`
3. 服务端协议当前不承诺也不负责返回以下当前机器本地字段：
   - 绝对路径
   - 文件存在性
   - `mediaPreviewUrl` / `layoutVideoPreviewUrl`
   - 本地 FFmpeg / FFprobe 探测得到的媒体元数据
4. 客户端在展示镜头详情时，必须执行“两段合并”流程：
   - 先读取服务端详情作为协同事实基线
   - 再读取当前机器本地扫描结果与路径映射结果，补齐本地文件能力字段
5. 需要由客户端本地补齐的字段包括但不限于：
    - `absolutePath`
    - `exists`
    - `mediaPreviewUrl`
    - `mediaPreviewMode`
    - `mediaDurationSeconds`
    - `mediaFrameCount`
    - `mediaFps`
    - `layoutVideoPreviewUrl`
    - 其他本地媒体信息字段
6. 若当前机器没有对应文件、没有路径映射，或本地预览链路失败，客户端必须显示“当前机器不可预览 / 缺路径映射 / 本机缺文件”等本地状态，不能让服务端代判
7. 同一账号在不同机器登录时，binding 协同事实可以一致，但本地预览状态允许不同；这是协议允许且预期的行为

### 11.12 镜头详情强制代理预览规则

1. 镜头详情页的最终预览源必须以客户端主进程返回结果为准，页面层不得自行拼接或猜测预览地址
2. 在协同主数据场景下，production / layout 的详情播放必须优先使用代理预览
3. 若代理预览生成失败，客户端不得静默回退为原始 `file://` 直连播放，也不得把失败结果伪装成可播放
4. 详情页仅在主进程已明确返回可用预览源时，才可进入播放态；否则应维持不可预览状态并提示原因
5. 当前客户端的代理预览 URL 采用 `movtools-preview://video?path=...` 自定义协议承载，本地文件由主进程协议处理器映射到实际路径
6. 自定义协议注册必须在 `app.whenReady()` 之后完成，避免 Electron 启动阶段触发 `Session can only be received when app is ready` 类错误

### 11.13 第十一批文件检查协作台与镜头模块共用 binding 事实

1. 第十一批明确：文件检查协作台不是独立业务域，它服务于镜头模块；其产出的匹配结果必须进入镜头共用 binding 协同事实。
2. 文件检查协作台在当前机器本地生成或刷新以下结果时，必须继续复用镜头共用 binding 协议，不得发明第二套文件检查专用同步口径：
   - `ma`
   - `mov`
   - 当前已选 `layout`
   - 当前已选 `layoutVideo`
3. 第十一批默认继续复用第七批 / 第九批既有接口：
   - `POST /api/lenses/{lensId}/bindings`
   - `DELETE /api/lenses/{lensId}/bindings?bindingType=...&versionNum=...`
4. 文件检查协作台、镜头列表、镜头详情页必须消费同一份服务端回读 binding 事实；不允许出现“文件检查页一套结果、镜头详情另一套结果、镜头列表第三套结果”的并行口径。
5. 客户端可以在文件检查协作台保留仅当前机器可见的本地补充信息，但这些信息只限于本地能力字段，不构成新的协同事实，包括但不限于：
   - `absolutePath`
   - `exists`
   - 本地预览 URL
   - 本地媒体元数据
6. 第十一批若需要让文件检查协作台回读更多镜头摘要字段，也必须遵守“binding 协同事实优先、本地能力字段补齐”的边界，不得把本地可预览状态反写为服务端协同事实。
7. 若后续发现现有镜头 binding 协议不足以表达文件检查协作台所需的镜头共用事实，必须先更新本文件，再新增或调整接口；不允许客户端和服务端各自隐式扩展。

### 11.14 制片工作台镜头池排序协议

1. 制片工作台的镜头池列表、筛选结果、可直接拼接镜头列表、待补视频镜头列表，默认展示顺序均以镜头号 `lensCode` 为主排序依据
2. 排序比较必须采用自然排序语义，数字段按数值比较，确保 `SHOT05` 排在 `SHOT05_1` 之前，`SHOT10` 不得排在 `SHOT2` 之前
3. `sceneNo / 场次` 只作为镜头辅助元数据展示，不作为制片工作台默认排序字段
4. 制片工作台的“按镜头号重排”仅影响当前草稿任务内的镜头顺序，不修改镜头基础数据，也不改变镜头池源数据
5. 服务端返回镜头列表时，也必须与本节保持一致的排序口径，避免客户端刷新后出现顺序倒退
6. 若后续要支持“按场次排序”之类其他排序方式，必须以独立开关或独立入口表达，不得与“按镜头号重排”共用同一文案或同一动作语义

### 11.8 第九批联调要求

1. 客户端必须证明“刷新全部文件匹配”后的真实同步请求已经发出
2. 服务端必须证明已保存或回读第九批约定结果
3. 镜头详情页必须证明其协同事实来自服务端真实回读，且文件存在性 / 预览能力来自客户端当前机器本地结果
4. 若第九批协议发生变化，必须先更新本文件，再改代码

## 12. 试运行提醒

正式试运行前，还应重点确认以下项目：

1. 生产账号是否能进入项目页、用户页、审片页
2. 项目创建时的成员预置是否能稳定保存
3. 项目成员增删是否在项目页与用户页保持一致
4. 删除成员接口是否按 `projectCode + userId` 调用
5. 数据源切换后，成员列表是否仍能正确加载
6. 权限不足、接口参数错误、空数据等场景是否给出可读提示
7. 创建项目时是否真实执行首集初始化，而不是只跳转到镜头页
8. 初始化结果是否能区分未执行 / 成功 / 失败 / 部分成功
9. 根目录级 `initExcelPath` 是否在客户端到服务端链路中完整保留
