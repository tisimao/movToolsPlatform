# 导演账号功能二次迭代服务器端派工指令包

## 1. 文档用途

本文件用于把“导演账号功能二次迭代”的服务器端修复任务直接派发给其他 AI 或开发执行。

本轮问题不是单纯前端切镜状态错误，而是服务端“按镜头回读反馈”接口返回了同任务其他镜头的反馈，导致导演在镜头 B 播放时仍看到镜头 A 的反馈列表，并且 `镜头通过` 按钮被错误禁用。

## 4. 问题背景

当前服务端 `GET /api/review-feedbacks/lens/{lensId}` 名义上是“按镜头回读反馈”，但实现中存在查询范围过宽的问题。

重点排查：

- `movtools-server/src/Movtools.Server.Infrastructure/Services/ReviewService.cs`
- `movtools-server/src/Movtools.Server.Api/Controllers/ReviewsController.cs`

当前高风险实现位于 `LoadFeedbacksByLensInternalAsync`，存在类似以下条件：

1. `x.LensId == lensId`
2. `x.ReviewTask.Shots.Any(s => s.LensId == lensId)`
3. `x.ReviewTask.LensId == lensId`

后两项会把“同一任务内其他镜头的反馈”一并带回，造成镜头反馈列表串味。

## 5. 本阶段目标

把服务端“按镜头回读反馈 / 绘制”的语义彻底收紧为“只返回当前镜头自身正式反馈事实”，不再把同任务其他镜头的反馈混入结果。

## 6. 必做事项

1. 修复 `LoadFeedbacksByLensInternalAsync` 的查询边界。
2. 核对 `GetFeedbacksByLensAsync` / `GetDrawingFramesByLensAsync` 是否存在相同口径污染。
3. 明确 `GET /api/review-feedbacks/lens/{lensId}` 的正式语义：
   - 只回读 `lensId` 对应镜头的反馈
   - 不得因为该反馈所属任务包含当前镜头就一并返回
4. 若需要兼容历史脏数据，只允许以下兼容方式：
   - 优先按 `ReviewComment.LensId == lensId`
   - 若历史记录 `LensId` 缺失，但 `TaskShotId` 有值，则显式通过 `TaskShotId -> ReviewTaskShot.LensId == lensId` 做一对一兼容映射
   - 不允许再使用“任务中包含该镜头”作为回读条件
5. 核对绘制回读链路是否与反馈列表共用同一批脏数据源；若是，同步修复。
6. 返回结果必须继续稳定携带：
   - `reviewTaskId`
   - `taskShotId`
   - `lensId`
   - `feedbackRoundId`
   - `frameNumber`
   - `timecode`
   - `drawingFrames`

## 7. 禁止事项

1. 不要把问题掩盖成“客户端自己过滤就行”。服务端必须先修正正式语义。
2. 不要把接口改成“按任务聚合反馈”但仍沿用 `lens/{lensId}` 路由名。
3. 不要破坏镜头详情页、导演反馈回看页对当前轮反馈的既有消费语义。

## 8. 推荐修改方向

### 8.1 服务端查询收口

优先把反馈查询收口为以下逻辑之一：

1. 正常数据：`ReviewComment.LensId == lensId`
2. 历史兼容数据：`ReviewComment.TaskShotId` 指向的任务镜头其 `LensId == lensId`

不允许再保留：

1. `ReviewTask.Shots.Any(s => s.LensId == lensId)`
2. `ReviewTask.LensId == lensId`

### 8.2 返回结果语义自检

对 `GET /api/review-feedbacks/lens/{lensId}` 的结果执行自检：

1. 返回数组中每条反馈都必须属于当前 `lensId`
2. 返回的 `drawingFrames` 也只能来自当前 `lensId`
3. `latestRound` 聚合不能跨镜头

## 9. 自测要求

至少补 1 组服务端测试，建议场景如下：

1. 创建一个多镜头审片任务，包含镜头 A / 镜头 B
2. 只给镜头 A 创建正式反馈
3. 调用 `GET /api/review-feedbacks/lens/{A}`，应返回 A 的反馈
4. 调用 `GET /api/review-feedbacks/lens/{B}`，应返回空列表
5. 若 A 与 B 属于同一个任务，也不能互相串反馈

如有绘制回读接口，再增加：

1. 只给镜头 A 提交绘制
2. 调用镜头 B 的 drawings 接口应为空

## 10. 交付标准

1. 多镜头同任务下，镜头反馈列表严格按镜头隔离
2. `GET /api/review-feedbacks/lens/{lensId}` 返回结果中不再出现其他镜头反馈
3. 绘制回读不再串镜头
4. 服务端测试通过
5. 服务端编译通过

## 11. 最终输出给上游的内容

1. 本次实际修复了哪些查询口径
2. 是否存在历史脏数据兼容逻辑
3. 新增或修改了哪些测试
4. 是否确认镜头级反馈回读不再串镜头
