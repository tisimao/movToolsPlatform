# 制片内部提审派工指令包

## 1. 文档用途

本文件用于把“制片内部提审、任务派工、导演按任务审片”这一批功能直接派发给其他 AI 或开发执行。

本批重点不是继续把“镜头二级状态”当成“导演任务入口”，而是把以下链路彻底打通：

1. 制片整理待提审镜头
2. 制片创建或编辑审片任务
3. 制片把多个镜头按顺序派发给导演
4. 导演按任务队列串联审片
5. 导演反馈回流镜头与任务
6. 制片跟踪本轮任务进度并决定是否重新提审

## 2. 本批次目标

这一批完成后，系统必须具备：

1. 制片可把 `待提审` / `已按反馈修改` 的镜头编入一个审片任务，而不是只能单镜头硬切到 `审片中`
2. 审片任务支持多镜头、有顺序、有提审版本、有目标导演、有任务说明
3. 制片可保存草稿、调整顺序、提交任务、关闭任务、查看反馈回流进度
4. 导演只通过“审片任务”进入审片，不再依赖镜头手工改二级状态来出现任务
5. 导演审片页顶部展示“任务镜头队列”，支持按制片编排顺序串联观看
6. 任务提交后，镜头二级状态由系统推进到 `审片中`，而不是靠前端按钮伪造
7. 导演保存反馈后，镜头进入 `待处理反馈`；导演确认通过后，镜头进入 `内部通过`
8. 制作人员改完后标记 `已按反馈修改`，制片可再次把镜头编入下一轮任务
9. 制片页和导演页对“任务 / 镜头 / 二级状态”的口径完全一致

这一批不做：

1. 不把镜头主状态和内部审片二级状态混成一个字段
2. 不让导演自己拼任务镜头顺序
3. 不把“进入审片”继续保留为单镜头手工按钮主链路
4. 不仅改前端文案而不改真实任务创建和提交流程

---

## 3. 已固化业务边界（必须遵守）

1. `待提审` 只表示镜头具备进入任务的条件，不表示导演已经收到任务
2. 导演看见任务的前提是存在正式 `ReviewTask`，而不是镜头二级状态被手工改成 `审片中`
3. 制片是任务组织者，导演是任务执行者，制作人员是反馈处理者
4. `审片中` 必须由“任务提交 / 任务开始 / 镜头进入任务上下文”驱动，不允许继续作为普通镜头按钮主入口
5. 一个审片任务可以包含多个镜头，导演应按制片编排顺序串起来审
6. 审片任务状态、镜头二级状态、镜头主状态是三套对象，不能混用
7. 当前镜头处于 `待处理反馈` 时，新版本产出本身不自动清空为 `未进入审片`；需待制作人员确认处理完成后再推进到 `已按反馈修改`
8. 当前镜头二级状态不是 `内部通过` 时，正式提交仍必须被拦截

---

## 4. 问题定性

当前实现需要按以下结论理解：

1. 现有镜头页里的 `进入审片` 只是在改镜头二级状态，不会自动创建导演任务
2. 导演仪表盘和导演审片页读取的是 `ReviewTask` 列表，不会因为镜头状态变成 `审片中` 自动出现内容
3. 当前前端更接近“一镜头一任务”，与“制片成组派工、导演串联审片”的业务目标不一致
4. 服务端虽然已经存在任务与任务镜头雏形，但制片创建、编辑、提交、排序链路没有完整前端落地
5. 因为任务流和镜头二级状态流没打通，才会出现“制片改成审核中，但导演没有任务”的现象

## 5. 目标业务模型（本批定稿）

### 5.1 三层状态

1. 镜头主状态：正式业务状态，面向客户提交流程
2. 镜头二级状态：当前镜头版本的内部审片状态
3. 审片任务状态：一轮任务的组织与执行状态

### 5.2 镜头二级状态职责

1. `未进入审片`：当前版本尚未纳入内部审片
2. `待提审`：可被制片加入任务，但导演尚未收到
3. `审片中`：镜头已进入某个已提交给导演的审片任务
4. `待处理反馈`：导演已给出反馈，等待制作处理
5. `已按反馈修改`：制作已改完，等待制片决定是否重新提审
6. `内部通过`：导演确认当前版本内部通过

### 5.3 审片任务状态建议

1. `draft`：草稿，制片正在装配镜头
2. `ready`：待提交，任务结构已完整但还未派发
3. `pending`：待审，已派发给导演但导演未开始或未整体进入
4. `in-review`：审阅中，导演已开始处理该任务
5. `completed`：已完成，任务中镜头均有明确结论或本轮结束
6. `closed`：已关闭，人为收尾或中止

说明：若当前服务端状态命名暂时不能一步到位，可做兼容映射，但页面语义必须按上面表达。

## 6. 标准业务闭环（必须按此实现）

1. 制作人员产出新版本
2. 制作人员完成当前版本文件同步与版本回写
3. 制片完成预检查并将镜头标记为 `待提审`
4. 制片在“审片任务管理页”中创建任务或编辑草稿
5. 制片把多个镜头加入任务，并指定顺序、提审版本、目标导演、说明、截止时间
6. 草稿阶段镜头保持原二级状态
7. 制片正式提交任务
8. 任务进入导演待审列表，处于 `待提审` / `已按反馈修改` 的镜头进入 `审片中`
9. 导演按任务镜头队列联播、切镜头、保存反馈
10. 导演保存反馈后，对应镜头二级状态进入 `待处理反馈`
11. 导演确认某镜头通过后，对应镜头二级状态进入 `内部通过`
12. 制作人员处理反馈并产出新版本
13. 新版本产出后镜头仍保持 `待处理反馈`
14. 制作人员确认已按反馈处理后，镜头进入 `已按反馈修改`
15. 制片决定是否重新纳入下一轮任务，并再次手动推进到 `待提审`

## 7. 制片页必须新增或调整的能力

### 7.1 制片镜头页改造

制片镜头页继续承担“筛选与整理”的职责，但不再把“进入审片”当作最终派工动作。

必须完成：

1. 保留镜头级二级状态操作：
   - `未进入审片 -> 待提审`
   - `已按反馈修改 -> 待提审`
2. 弱化或移除当前直接把镜头改到 `审片中` 的主按钮
3. 新增批量勾选能力，用于把多个镜头加入审片任务
4. 新增筛选项：
   - 按内部审片二级状态筛选
   - 按是否已有可播放视频筛选
   - 按是否已加入未关闭任务筛选
   - 按是否有导演反馈筛选
5. 列表中补充任务相关字段：
   - 最近提审版本
   - 所属审片任务
   - 最近导演反馈时间
   - 待处理反馈数量
6. 新增批量操作入口：
   - 加入新任务
   - 加入现有草稿任务
   - 从草稿任务移除

### 7.2 制片审片任务管理页

制片端必须有独立的“任务装配台 / 审片任务管理页”，不能只靠导演页代替。

1. 任务列表区
2. 任务编辑区或详情抽屉
3. 待提审镜头池

任务列表字段至少展示：

8. 镜头顺序拖拽排序
9. 每个镜头指定提审版本
10. 每个镜头显示素材可用性
11. 保存草稿
12. 提交任务

2. 查看每个镜头反馈数量
3. 查看每个镜头当前二级状态

## 8. 导演页必须新增或调整的能力

### 8.1 导演仪表盘口径对齐

导演仪表盘必须明确说明：导演处理的是“任务”，不是“所有镜头”。

必须完成：

1. 当前激活项目的任务列表按正式 `ReviewTask` 展示
2. 任务卡片展示镜头数量，而不是只展示单镜头号
3. 任务卡片支持“开始审片 / 继续审片”
4. 任务卡片支持显示目标导演、提交人、截止时间、最近更新时间
5. 最近反馈记录回跳时优先回到“任务上下文中的镜头”

### 8.2 导演审片页改造

导演审片页必须从“单镜头任务视角”升级为“任务镜头队列视角”。

页面结构必须对齐：

1. 顶部：任务信息 + 镜头队列
2. 中部：播放器 / 画面标注区
3. 底部左侧：反馈编辑区
4. 底部右侧：当前镜头反馈列表

顶部任务区必须支持：

1. 展示任务名称、项目、提审说明、提审人、目标导演、截止时间
2. 展示任务内镜头顺序列表
3. 每个镜头项展示：
   - 顺序号
   - 镜头号
   - 提审版本
   - 实际播放版本
   - 反馈数量
   - 当前二级状态
   - 任务内处理状态
4. 点击镜头项切换当前镜头
5. 显示任务整体进度，如 `已完成 8 / 24`

导演审片动作必须支持：

1. 开始任务时把任务状态推进为 `审阅中`
2. 保存反馈卡片后刷新当前镜头反馈与任务进度
3. 对当前镜头执行“内部通过”
4. 对当前镜头执行“要求修改”时进入 `待处理反馈`
5. 当任务内所有镜头均完成本轮处理后，允许关闭或完成任务

### 8.3 导演镜头页定位

导演镜头页仍保留为辅助查询入口，但必须与任务模型口径一致。

必须补充：

1. 镜头详情中展示所属审片任务列表
2. 支持从镜头详情定位回当前或最近任务
3. 导演反馈区显示反馈卡片与所属任务
4. 明确文案：镜头页不是主审片入口，主入口在任务页

## 9. 服务端必须对齐的能力

### 9.1 核心原则

1. 创建任务和提交任务必须是独立动作
2. 任务可以包含多个镜头
3. 镜头进入 `审片中` 必须由任务提交或任务开始触发
4. 导演任务列表必须基于正式任务查询，而不是基于镜头状态反推

### 9.2 任务接口要求

服务端至少要稳定支持以下能力：

1. 创建任务草稿
2. 更新任务基础信息
3. 给任务添加镜头
4. 移除任务镜头
5. 调整镜头顺序
6. 为每个镜头保存提审版本
7. 提交任务给导演
8. 导演开始任务
9. 导演完成或关闭任务
10. 查询任务列表
11. 查询任务详情

建议接口语义：

1. `POST /api/review-tasks/draft`：创建草稿任务
2. `GET /api/review-tasks/producer`：查询制片任务列表
3. `GET /api/review-tasks/{id}/detail`：查询制片任务详情
4. `PUT /api/review-tasks/{id}`：更新任务基础信息
5. `POST /api/review-tasks/{id}/shots`：添加镜头到任务
6. `DELETE /api/review-tasks/{id}/shots/{taskShotId}`：移除任务镜头
7. `PUT /api/review-tasks/{id}/shots/reorder`：任务镜头排序
8. `POST /api/review-tasks/{id}/submit`：正式提交给导演
9. `POST /api/review-tasks/{id}/start`：导演开始任务
10. `POST /api/review-tasks/{id}/complete`：任务完成
11. `POST /api/review-tasks/{id}/close`：任务关闭

补充说明：当前正式协议已收口到 `通信协议文档.md`，本文件只保留派工说明，不再单独演化另一套接口命名。

### 9.3 任务详情回包要求

任务详情至少返回：

1. 任务基本信息
2. 目标导演
3. 提审说明
4. 状态
5. 镜头清单
6. 每个镜头顺序
7. 每个镜头提审版本
8. 每个镜头实际播放版本
9. 每个镜头当前二级状态
10. 每个镜头反馈数量
11. 每个镜头最近反馈时间
12. 每个镜头是否内部通过

## 12. 推荐实施顺序

1. 先统一业务口径：`待提审 != 已派工`
2. 先补服务端任务草稿、任务镜头、任务提交、任务详情能力
3. 再补制片任务管理页与镜头批量入任务链路
4. 再改导演仪表盘，让任务真正来自正式派发
5. 再改导演审片页顶部任务镜头队列
6. 最后回归镜头详情回挂、统计口径和状态联动验证

## 13. 本批次完成定义

只有同时满足以下条件，才算“制片内部提审派工链路”完成：

1. 制片已能按任务组织多个镜头并提交给导演
2. 导演任务来源已从“镜头手工改状态”切换为“正式任务派工”
3. 导演可在一个任务内按制片编排顺序串联审片
4. 镜头二级状态与任务状态职责清晰且联动正确
5. 制片页和导演页对任务、镜头、反馈回流的口径一致
6. 不再出现“制片改成审核中但导演没有任务”的业务断链
