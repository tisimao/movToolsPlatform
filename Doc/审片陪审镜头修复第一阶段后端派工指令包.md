# 审片陪审镜头修复第一阶段后端派工指令包

## 1. 文档用途

本文件用于把“陪审镜头参与类型真值统一、导演 complete / 制片 close 边界收口、服务端硬保护补齐”这一阶段的服务端工作直接派发给其他 AI 或开发执行。

参考文档：

- `D:\project\MovtoolsEx\Doc\项目固化文档.md`

---

## 2. 本阶段目标

本阶段只解决“服务端把陪审镜头当成正式审片镜头”以及“任务动作边界不够硬”的问题。

完成后必须满足：

1. 任务镜头参与类型在创建、追加、编辑、回读链路中语义统一
2. `context` 陪审镜头不会被默认写成 `review`
3. 导演 `complete` 与制片 `close` 的权限和动作边界稳定
4. `context` 镜头不参与正式反馈、通过、返修和完成统计
5. mixed task 场景下陪审镜头不会再阻塞任务流转

---

## 3. 工作边界

1. 本阶段只改服务端协议、控制器、服务、必要测试与必要文档
2. 本阶段不负责导演页按钮显隐与前端文案收口
3. 本阶段允许兼容旧字段，但必须确定一个正式真值源
4. 本阶段若发现历史接口仍把镜头默认设为 `review`，必须修掉，不得保留隐患

## 5. 必做事项

1. 清查所有任务镜头创建来源，确保以下链路都能保留真实 `ParticipationMode`：
   - 创建草稿任务
   - 更新任务镜头
   - 向已有任务追加镜头
   - 仅传 `shotIds` 的兼容入口
2. 重点修正 `ReviewsController.cs` 中 `BuildShotRequestsAsync` 的旧逻辑；当前它会把镜头统一写成 `Review`，这是必须消除的根因
3. 统一任务详情、任务列表、制片详情、导演详情里的镜头参与类型输出字段，至少保证：
   - 正式字段稳定
   - 兼容字段仍可读
   - 不会因为字段缺失而静默回落成 `review`
4. 补齐服务端硬保护，拒绝对 `context` 镜头执行以下正式写操作：
   - 创建正式反馈
   - 更新正式反馈
   - 删除正式反馈
   - 正式绘制事实写入
   - 写入 `DIRECTOR_APPROVED`
   - 写入任务内返修结论
5. 统一任务完成判定和统计口径，只统计 `review` 镜头；确保 mixed task 中 `context` 镜头不阻塞 `complete`
6. 保持导演完成任务走 `complete`，制片关闭任务走 `close`；如果发现某条服务端兼容路径把两者混用，必须收口
7. 补充或修正服务端测试，至少覆盖：
   - 5 个 `review` + 1 个 `context` 的 mixed task 完成场景
   - `context` 镜头被拒绝正式反馈/通过/返修场景
   - producer 不能执行 `complete`，director 不能越权走 producer close 语义场景
8. 更新必要协议说明，写清正式字段和兼容策略

---

## 6. 禁止事项

1. 不要只修一个接口，必须把所有能写任务镜头参与类型的入口都清掉旧默认值
2. 不要只靠前端按钮限制，服务端必须做最终拒绝
3. 不要把 `layout` 补位来源当成 `context` 镜头真相源
4. 不要为了兼容旧接口继续默认把未知值吞成 `review`
5. 不要改坏现有导演可见 completed、制片可见 closed 的归档语义

---

## 7. 验收标准

1. mixed task 中 `context` 镜头不会再被写成 `review`
2. 任务完成判定只统计正式镜头
3. `context` 镜头不能进入正式写链路
4. director `complete` 与 producer `close` 边界清晰且测试覆盖
5. 服务端相关测试通过
6. 协议字段说明与代码输出一致
