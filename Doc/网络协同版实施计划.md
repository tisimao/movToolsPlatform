# movTools 网络协同版实施计划

## 1. 总体策略

实施原则：

1. 先跑通协同主链路，再补外围能力
2. 尽量复用现有 Electron 单机版
3. 先同步元数据，文件能力始终留在客户端
4. 不集成 SVN，完全改为人工外部流程

---

## 2. 阶段划分

### 阶段 0：冻结边界

目标：明确新系统做什么、不做什么。

#### 输出物

- 架构说明文档
- 角色权限清单
- 数据边界清单
- 迁移范围清单

#### 必须确认

- 不集成 SVN
- 服务端不做文件扫描/预览/转码
- 客户端继续 Electron
- 服务端使用 C# + ASP.NET Core + PostgreSQL
- 路径使用“逻辑路径 + 客户端映射”模型

---

### 阶段 1：梳理旧项目可复用资产

目标：识别直接复用与必须改造的模块。

#### 重点盘点

##### 可直接复用

- React 页面
- TypeScript 类型定义
- Electron IPC 结构
- 本地文件扫描逻辑
- 预览/FFmpeg/任务队列
- 镜头业务规则

##### 需要改造

- 直接依赖 SQLite 的业务服务
- 本地唯一真相源假设
- 旧项目注册表与激活逻辑

#### 输出物

- 旧模块映射表
- 改造优先级表
- 风险点清单

---

### 阶段 2：设计服务端领域模型

目标：先把协同数据模型定清楚。

#### 优先建模模块

- 用户
- 角色
- 项目
- 集
- 镜头
- 镜头版本
- 文件绑定
- 审片任务
- 审片评论
- 路径映射
- 操作日志
- 同步变更日志

#### 输出物

- ER 图
- 表结构草案
- 状态流转图
- 角色权限矩阵

---

### 阶段 3：搭服务端最小骨架

目标：让服务端工程先跑起来。

#### 技术落地

- ASP.NET Core Web API
- EF Core
- PostgreSQL
- SignalR
- Docker Compose

#### 第一批模块

- Auth
- Users
- Projects
- Episodes
- Lenses
- Reviews
- Sync Changes

#### 验收标准

- 本机可运行
- Docker 可运行
- 可连接 PostgreSQL
- 可登录并拿到 token

---

### 阶段 4：定义客户端-服务端协议

目标：固定前后端接口。

#### 需要确定

- 登录/当前用户
- 项目列表
- 集列表
- 镜头列表/详情
- 镜头编辑
- 状态流转
- 审片提交/评论/通过/返修
- 路径映射读写
- 同步 push/pull

补充说明：审片“返修”与镜头“返修”属于不同协议域，前者是审片任务动作，后者是镜头主状态流转。

补充说明：镜头“返修”完成后，服务端必须同步递增并持久化 `VersionNum`，客户端刷新/同步绑定时必须以服务端返回的最新版本号为准。

#### 输出物

- API 文档
- DTO 结构
- 错误码约定
- 冲突返回规范

---

### 阶段 5：客户端抽象数据访问层

目标：把当前直接写 SQLite 的服务改成仓储接口模式。

#### 建议拆分

- `ProjectRepository`
- `LensRepository`
- `ReviewRepository`
- `PathMappingRepository`

并分别实现：

- `Local*Repository`
- `Remote*Repository`

#### 目标结果

- 页面层尽量不改
- 业务层可在本地模式与协同模式间切换
- 为后续同步能力打基础

---

### 阶段 6：打通协同主链路（MVP）

目标：先做最小可用的多人协作版本。

#### MVP 范围

1. 用户登录
2. 项目列表
3. 集切换
4. 镜头列表与详情
5. 镜头编辑
6. 镜头状态流转
7. 审片提交
8. 导演审片
9. 制作人员查看反馈

#### 暂不做

- 复杂冲突自动合并
- 服务端文件处理
- 高级报表
- Web 管理后台

#### 验收标准

- 两台客户端可协同查看同一项目
- 一台修改，另一台能收到更新
- 导演与制作之间形成审片闭环

---

### 阶段 7：接入路径映射机制

目标：解决不同机器挂载路径不同的问题。

#### 实施内容

- 服务端保存逻辑根与逻辑路径
- 客户端维护本机路径映射
- 客户端本地解析真实路径后执行预览和文件检查
- 项目激活时允许按 Windows 盘符差异自动重定位本地路径，要求盘符后的路径结构一致

#### 验收标准

- 多客户端可访问同一逻辑资源
- 服务端数据库中不出现本机绝对路径
- 同一服务端路径可在不同机器上落到不同盘符，但后缀路径保持一致

---

### 阶段 8：迁移绑定与检查摘要同步

目标：让协同端看到镜头文件准备情况。

#### 同步内容

- 当前版本绑定结果
- Layout 候选
- 当前采用 Layout
- 引用检查摘要
- 问题数量

#### 不同步的内容

- 本地扫描中间过程
- 本地预览缓存
- 大量临时日志

---

### 阶段 9：保留离线工作能力

目标：服务端异常时客户端仍尽量可工作。

#### 建议

- 继续保留本地 SQLite
- 引入本地 outbox 记录待同步操作
- 网络恢复后自动同步
- 本地扫描/预览/提取不受影响

---

### 阶段 10：历史数据迁移

目标：把旧项目数据带入新系统。

#### 迁移步骤

1. 从旧 SQLite 导出主数据
2. 导入 PostgreSQL
3. 客户端首次登录后拉取主数据
4. 客户端重新扫描本地文件并建立可达状态

#### 注意事项

- 旧绝对路径不能直接入服务端
- 必须先转换为逻辑路径模型

---

### 阶段 11：补齐用户管理与权限

目标：从可用提升到可控。

#### 第一版权限建议

- 管理员
- 制片
- 导演
- 制作人员
- 只读

#### 权限控制点

- 谁能改项目配置
- 谁能改镜头状态
- 谁能提交审片
- 谁能审片
- 谁能查看全部项目

---

### 阶段 12：测试与试运行

目标：小范围验证后再推广。

#### 建议方式

- 先选 1 个项目试点
- 先选 3~5 个真实用户
- 在 NAS 环境下跑完整流程

#### 重点验证

- 路径映射可用性
- 镜头状态实时同步
- 审片闭环
- 检查摘要同步
- 服务端异常下的客户端表现

## 4. 风险最高的点

### 4.1 路径模型

这是整个系统的关键，如果路径模型没定准，后续会反复返工。

### 4.2 旧服务层直接绑定 SQLite

这是客户端改造成本最高的部分，应尽早抽象 Repository。

### 4.3 多人编辑冲突

第一版建议使用乐观锁，不要一开始做复杂自动合并。

### 4.4 本地绝对路径污染服务端

这是必须严防的问题，服务端只允许逻辑路径。

### 4.5 第一版范围过大

如果第一版同时做太多功能，交付风险会明显升高。

## 8. 结论

这套计划的核心不是推倒重写，而是：

- 保留现有桌面端价值
- 抽离协同能力到服务端
- 通过路径映射解决 NAS / 多机器差异
- 用最小代价把单机工具升级成可协同系统
