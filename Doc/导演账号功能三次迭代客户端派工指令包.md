# 导演功能三次迭代客户端派工指令包

## 1. 文档用途

本文件用于把“导演审片反馈条目与绘制时间线彻底解耦、并保证导演端与制作协同端显示一致”这一组客户端任务，按三次迭代直接派发给其他 AI 或开发执行。

目标目录：`D:\project\MovtoolsEx\movtools-client\`

参考文档：

- `D:\project\MovtoolsEx\Doc\导演账号功能设计文档.md`
- `D:\project\MovtoolsEx\Doc\通信协议文档.md`
- `D:\project\MovtoolsEx\Doc\项目固化文档.md`
- `D:\project\MovtoolsEx\Doc\导演功能三次迭代服务器端派工指令包.md`

---

## 2. 目标问题

当前导演审片链路存在根本性模型错位：

1. 草稿阶段的绘制显示依赖本地“绘制事件流”推导
2. 提交后的绘制显示却依赖“某一条反馈卡片附带的 drawingFrames”
3. 非反馈帧上的继承绘制在提交后丢失
4. 导演工作台、镜头详情页、制作人员查看页可能各自按不同口径回放绘制

本批次必须把客户端统一到同一条真相源：

1. 反馈条目只表达“哪一帧有文字/决策反馈”
2. 绘制内容统一表达为“某一轮反馈的整段绘制时间线”
3. 任意页面只要查看同一 `feedbackRoundId`，就必须得到同一绘制结果

---

## 3. 三次迭代总目标

### 第一次迭代：统一前端显示模型

目标：先消除“提交前后显示不一致”。

必须完成：

1. 导演工作台提交后回放绘制，改为消费“当前轮完整 drawing timeline”
2. `resolveReviewVisibleAnnotationPaths` 改为真正的时间线回放器
3. 提交后第 4-9 帧、第 15-19 帧这类“无反馈条目但有继承绘制”的帧必须稳定显示
4. 导演端不再用“当前帧是否有 feedback”来决定是否能显示该帧绘制

### 第二次迭代：统一导演端与协同查看端

目标：导演工作台、镜头详情页、制作人员回看页全部消费同一套轮次级绘制数据。

必须完成：

1. `ReviewPage`、`LensPage`、导演反馈回放组件统一使用轮次级 timeline
2. 反馈列表和绘制预览不再各自从不同字段倒推
3. 制作人员看到的正式反馈条目与遮罩绘制，必须与导演提交后的正式结果一致

### 第三次迭代：收口兼容、缓存、验证与文档

目标：把旧字段兼容、历史数据解析、缓存刷新、测试基线全部收口。

必须完成：

1. 历史 `annotationDataJson` / 单条 feedback `drawingFrames` 兼容读取
2. 新任务、新轮次切换、关闭任务、重新进入任务时缓存不串轮
3. 核心用例与跨页面回归用例固化

---

## 4. 客户端必须遵守的业务口径

1. 反馈条目和绘制时间线不是同一个概念
2. 反馈条目按帧绑定；绘制时间线按轮次绑定
3. 当前帧是否显示文字，只看该帧是否存在反馈条目
4. 当前帧是否显示绘制，只看当前轮 timeline 回放结果
5. 镜头详情页与导演工作台必须共享同一绘制解释规则
6. 点击历史反馈卡片只允许定位与展示该帧结果，不得注入新的本轮保留链

---

## 5. 第一次迭代派工内容

### 5.1 目标

让导演工作台在“草稿态 / 提交后态”都按同一时间线语义显示绘制。

### 5.2 必改文件

- `src/pages/ReviewPage.tsx`
- `src/lib/reviewDrawingResolver.ts`
- `src/lib/reviewFrameFeedback.ts`
- `src/types/review.ts`

### 5.3 必做事项

1. 在 `ReviewPage.tsx` 中，把绘制读取源从“按 frame 找最后一条 submitted feedback”改为“按当前轮取完整 drawing timeline”
2. 保留 `getSubmittedFeedbackByFrame()` 仅用于文字反馈卡片，不再用于绘制显示
3. 新增或重构 `getCurrentRoundDrawingFrames()` 之类的统一入口，优先消费 `currentShotFeedbackView.latestRoundDrawingFrames`
4. 重写 `resolveReviewVisibleAnnotationPaths()`，支持：
   - `DRAWN`
   - `CLEAR`
   - 按 `frameNumber + order` 排序
   - 从头回放到目标帧
   - 返回目标帧最终可见路径
5. 提交成功后不要再把“清空本地草稿状态”误等价成“正式绘制为空”；提交后页面应立即切回服务端轮次数据并保持一致

### 5.4 禁止事项

1. 不要继续用 `submittedFeedback?.drawingFrames` 作为唯一正式绘制来源
2. 不要把“没有反馈条目”解释成“没有绘制结果”
3. 不要在页面层复制一套新的绘制继承规则，必须统一收口到 resolver / helper

### 5.5 第一次迭代验收标准

以下用例必须通过：

1. 第 3 帧建反馈并绘制 A
2. 第 10 帧清屏
3. 第 15 帧绘制 B
4. 第 20 帧清屏
5. 提交前后都满足：
   - 3 帧显示反馈条目 + A
   - 4-9 帧显示 A、无反馈条目
   - 10-14 帧空白
   - 15-19 帧显示 B、无反馈条目
   - 20 帧及之后空白

## 6. 第二次迭代派工内容

### 6.1 目标

让导演工作台、镜头详情页、制作查看页共享同一正式绘制真相源。

### 6.2 必改文件

- `src/components/lens/DirectorFeedbackPlayback.tsx`
- `src/pages/LensPage.tsx`
- `src/pages/DirectorDashboardPage.tsx`
- `src/lib/directorFeedback.ts`
- `src/lib/reviewFrameFeedback.ts`

### 6.3 必做事项

1. `DirectorFeedbackPlayback.tsx` 不再从 `currentRoundFeedbacks` 聚合单条 feedback 的 drawingFrames 来推导遮罩
2. 为镜头详情页和其他查看页提供“当前轮正式 timeline”统一输入模型
3. 若当前页面只需要显示遮罩，不需要显示逐帧编辑控件，也必须复用同一 timeline 解析器，而不是再写一套 `collectDirectorFeedbackMaskPaths()` 的拼装逻辑
4. 反馈列表区继续显示 feedback 条目；遮罩/预览区改为显示 round timeline
5. 页面切换轮次、切镜头、切任务时，必须同步切换 timeline 数据源，不能残留上一个镜头或上一个任务的绘制缓存

### 6.4 禁止事项

1. 不要让导演工作台和镜头详情页各自维护不同的 drawingFrames 解释方式
2. 不要把“遮罩预览”和“反馈列表”耦合成必须来自同一条 feedback 记录
3. 不要把查看历史反馈动作做成新的本轮绘制输入

### 6.5 第二次迭代验收标准

1. 导演工作台回看结果与镜头详情页导演反馈遮罩逐帧一致
2. 制作人员查看页看到的正式遮罩与导演提交后的正式结果一致
3. 同一轮反馈在不同页面下不出现“某页有 A 某页没 A”的情况

## 7. 第三次迭代派工内容

### 7.1 目标

完成兼容、缓存、旧数据解析、测试与代码清理收口。

### 7.2 必改文件

- `src/repositories/remote/reviewRepository.ts`
- `src/repositories/local/reviewRepository.ts`
- `src/services/repositoryService.ts`
- `src/types/review.ts`
- 与导演反馈缓存、本地草稿恢复相关的页面和 helper

### 7.3 必做事项

1. 明确前端正式主读取字段：
   - feedback 列表：`feedbacks`
   - 当前轮 timeline：`latestRound.drawingFrames` 或等价轮次字段
2. 历史兼容读取顺序明确化：
   - 先读 round 级 timeline
   - 若缺失，再兼容旧 `drawingFrames`
   - 再缺失，最后兼容 `annotationDataJson`
3. 草稿提交后、本地缓存清空后、重新拉取后，页面必须立刻回到正式 round timeline
4. 新建任务、关闭任务、切换任务、重新进入镜头时，必须清理旧 shot/round 的绘制缓存
5. 清理散落的旧 helper，避免同名但不同语义的 drawing resolver 并存
6. 补齐类型注释和文档说明，明确哪些字段属于历史兼容字段，哪些是正式主字段

### 7.4 禁止事项

1. 不要继续让 `annotationDataJson` 同时承担“本地草稿编辑态”和“正式轮次时间线真相源”两个职责
2. 不要只修 ReviewPage 而不修 LensPage / 详情页 / 回放组件
3. 不要遗漏关闭任务后缓存清理，否则旧 feedback/timeline 会复活

### 7.5 第三次迭代验收标准

1. 历史数据能继续回看
2. 新提交数据全部按 round timeline 正常显示
3. 关闭任务后旧正式绘制不会在新任务中残留
4. `npm run typecheck` 与 `npm run build` 通过
