# 导演账号功能三次迭代客户端派工指令包

## 1. 文档用途

本文件用于把“导演账号功能三次迭代”的客户端任务直接派发给其他 AI 或开发执行。

本轮目标是让导演工作台和制片任务装配页一起落地以下规则：

1. 正式审片镜头 / 上下文陪审镜头的任务内区分
2. 制片新增镜头入任务时默认参与类型必须遵守素材匹配规则：版本视频已匹配默认 `review`，依赖 layout 视频补位默认 `context`
3. 制片关闭任务后，该任务反馈被清空，重新组织下一轮时不再残留旧反馈

## 4. 问题背景

当前客户端在任务装配与导演审片链路上同时存在以下问题：

1. 导演工作台默认把任务内每个镜头都当成“导演需要给反馈或给通过”的对象
2. 但实际业务里存在 `context` 镜头：

   - 只有 `layout` 或参考视频
   - 用于帮助导演看前后镜头衔接
   - 不需要提交反馈
   - 不需要点击通过
3. 同一镜头重新加入新任务时，页面仍可能显示已关闭旧任务遗留的反馈
4. 制片任务装配页新增镜头时，默认参与类型必须按素材匹配规则稳定落位

如果客户端不收口，会出现：

1. 队列颜色和状态含义错误
2. 上下文镜头出现 `镜头通过`、`返修`、`提交反馈` 按钮
3. 任务进度、处理数、完成提示被上下文镜头污染
4. 导演误以为还有未处理镜头
5. 新任务中继续出现旧任务反馈残留
6. 制片误把正式镜头或补位镜头以错误默认逻辑加入成错误参与类型

## 5. 本阶段目标

客户端正式接入任务镜头参与类型语义，并保证：

1. `review` 镜头是正式待审对象
2. `context` 镜头是只读陪审对象
3. 两类镜头都可参与任务连播
4. 但只有 `review` 镜头参与反馈编辑、通过/返修操作和任务进度统计
5. 制片新增镜头入任务时默认参与类型稳定遵守素材匹配规则，这条规则由客户端实现
6. 制片关闭任务并清空反馈后，同镜头重新进入新任务时默认反馈为空

## 6. 必做事项

1. 接入任务镜头字段，建议消费：
   - `reviewParticipationMode: "review" | "context"`
2. 导演工作台镜头队列必须显式区分两类镜头
3. 对 `context` 镜头隐藏或禁用以下能力：
   - `提交反馈`
   - `镜头通过`
   - `返修`
   - 反馈编辑区输入
4. `context` 镜头允许：
   - 视频播放
   - 联播切换
   - 定位前后镜头关系
5. 任务统计、任务完成提示、处理进度条只基于 `review` 镜头计算
6. 制片创建 / 编辑任务时，支持为任务镜头指定参与类型
7. 旧数据兼容：若服务端暂未返回该字段，客户端默认按 `review` 处理，但代码结构必须为后续正式字段留出口
8. 导演完成任务后，任务应从默认列表移出；当前阶段按“列表不再展示已完成任务”处理，不做删除任务按钮主链路
9. 制片关闭任务后，相关任务反馈在导演工作台与镜头详情中的默认展示都必须清空

## 8. 推荐修改方向

### 8.1 任务镜头视图模型补充

建议在前端任务镜头 view model 中补充：

1. `reviewParticipationMode`
2. `isContextShot`

并统一派生：

1. 是否可提交反馈
2. 是否可点击通过
3. 是否计入任务进度
4. 队列标签文案与颜色

### 8.2 导演工作台交互收口

对 `context` 镜头建议采用以下交互：

1. 保留播放区
2. 显示“上下文陪审镜头”标签
3. 显示“仅供衔接参考，不参与本轮正式审片结论”提示
4. 隐藏反馈编辑区或切换为只读说明态
5. 隐藏 `镜头通过` 与 `返修` 按钮

### 8.3 制片任务装配页补充

制片页需要允许在加镜头入任务时标记：

1. `正式审片镜头`
2. `上下文陪审镜头`

并在任务详情中回显：

1. 镜头参与类型
2. 上下文镜头不进入正式审片统计
3. 新增镜头时若用户未手工切换类型，默认值必须按素材匹配规则自动计算：当前版本视频已匹配时选中 `正式审片镜头`；仅依赖 layout 视频补位时选中 `上下文陪审镜头`

### 8.3.1 关闭任务后的前端反馈清理表现

客户端需配合以下体验：

1. 制片关闭任务成功后，任务详情中的反馈统计与最近反馈时间刷新为清空后的状态
2. 同一镜头重新加入新任务后，导演工作台默认反馈视图必须为空
3. 若本地仍缓存旧任务反馈视图，切入新任务时必须丢弃旧缓存，不能继续展示
4. 镜头详情页若回看当前已关闭任务，不应继续把已清空反馈显示为正式事实

### 8.4 任务统计与完成提示收口

以下前端展示必须只基于 `review` 镜头：

1. 已处理数
2. 未处理数
3. 可完成提示
4. 当前镜头是否需要处理

`context` 镜头可单独显示为：

1. 参考镜头
2. 陪审镜头
3. 不计入审结

## 9. 禁止事项

1. 不要仅靠 tooltip 说明，而仍然保留可点击写动作
2. 不要让 `context` 镜头继续进入“黄色待处理”或“红色未完成”统计分母
3. 不要在客户端把 `context` 镜头伪装成 `review` 镜头，只是文案改一下
4. 不要破坏现有正式镜头的反馈隔离与通过按钮收口逻辑
5. 不要把“完成后消失”做成前端伪删除并假装服务端已删除
6. 不要让新增镜头脱离素材匹配规则而固定默认成 `review` 或 `context`
7. 不要在关闭任务后继续展示已被服务端清空的旧反馈缓存

## 10. 自测要求

至少完成以下手工回归：

1. 创建混合任务：
   - 镜头 A 为 `review`
   - 镜头 B 为 `context`
2. 导演进入任务后：
   - A 显示正常反馈与通过能力
   - B 只可播放，不可提交反馈、不可通过、不可返修
3. 自动联播切到 B 时：
   - 不出现“当前镜头待处理”的错误提示
   - 不把 B 算入未完成统计
4. 任务进度只基于 A 计算
5. 制片页查看任务详情时，能看出 A/B 的类型差异
6. `npm run typecheck` 通过
7. 任务完成后，导演默认任务列表中不再显示该任务；若有历史筛选页或已完成筛选，仍可查看
8. 制片关闭任务后，再用同一镜头新建任务并进入导演工作台，默认反馈视图为空
9. 制片新增镜头时，未手工切换类型的镜头默认类型与素材匹配结果一致

## 11. 交付标准

1. 客户端正式支持 `review / context` 两类任务镜头
2. `context` 镜头只参与播放，不参与正式写动作
3. 队列、统计、完成提示不再被上下文镜头污染
4. 制片任务装配页可指定并回显镜头参与类型
5. 类型检查通过
6. `completed` 任务从默认列表移出，但不伪造物理删除
7. 新增镜头默认参与类型稳定遵守素材匹配规则
8. 关闭任务后不再残留旧反馈展示
