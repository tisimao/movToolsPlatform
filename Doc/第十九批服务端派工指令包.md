## 2. 本批次目标

这一批完成后，服务端必须具备：

1. 镜头详情、任务详情、反馈回读对“记录”分页承接的一级返修记录与“导演反馈”分页承接的导演反馈卡片保持清晰边界，不把两类反馈混成同一语义
2. 导演工作台按任务镜头顺序回读可直接用于连播的镜头队列数据
3. 反馈卡片创建协议稳定支持任务、镜头、版本、帧号、时间码、标注数据、图片路径
4. `提交反馈` 只上传反馈卡片，不自动改变任务最终结论
5. 反馈创建成功后，任务镜头反馈计数、最近反馈时间、镜头二级状态与任务摘要统计保持一致
6. 如客户端补充传入 `taskShotId`，服务端可做一致性校验并用于更精确回挂
7. 一级返修记录图片在十九批后必须由服务端真实保存，并能稳定回读预览地址给导演、制作人员、制片查看

这一批不做：

1. 不把客户端本地截图/画布能力迁移到服务端
2. 不强制重做整套 review domain 命名
3. 不把“提交反馈”混同为 `approve` / `reject` / `close`
4. 不要求本批必须新增批量创建反馈接口，允许继续单条创建

## 3. 当前代码状态（按现状理解）

当前服务端已具备：

1. 正式 `ReviewTask`、任务镜头、评论、反馈卡片实体与基础接口
2. `CreateFeedbackAsync` 基础链路，可写入版本号、帧号、时间码、图片路径、标注 JSON
3. 反馈创建后同步刷新镜头二级状态、任务镜头反馈计数等基础逻辑

当前仍存在的主要风险点：

1. 任务详情回包未必已显式给出足够清晰的“当前任务镜头可播放与排序基线”字段语义
2. `taskShotId` 若前端补入，当前服务端协议与实现尚未完全固化
3. 反馈创建、任务统计、镜头反馈回读之间可能存在字段不齐或投影不一致
4. “提交反馈仅上传卡片，不改任务结论”的语义需要继续在服务层和控制器层明确收口
5. 一级返修图片当前并未明确形成“服务端接收文件、持久化保存、返回预览地址”的正式闭环

---

## 4. 已固化业务边界（必须遵守）

1. 导演工作台以正式审片任务为真相源，任务镜头顺序就是连播顺序
2. 导演暂停后在客户端画面上完成标注；服务端只接收其产物与元数据
3. 一张反馈卡片最少绑定到：任务、镜头、版本、帧号或时间码、文字说明、标注数据、图片产物
4. “提交反馈”只表示上传当前任务下的反馈卡片，不自动等价为任务通过、返修或关闭
5. 任务最终结论仍只能由显式任务动作接口推进
6. 反馈创建成功后，服务端必须同步刷新镜头与任务的统计和状态事实
7. 镜头主状态链路中的返修记录与导演内部审片反馈卡片是两条不同事实链路，投影字段和文档语义必须保持区分
8. 一级返修图片在十九批后不得再依赖上传机器本地绝对路径作为唯一可读来源，必须形成服务端正式附件资产
