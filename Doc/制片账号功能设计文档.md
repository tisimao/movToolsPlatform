# 制片账号功能设计文档

## 1. 文档目的

本文件用于固化制片账号在导演审片链路中的职责、页面能力、任务组织规则与协同边界，确保后续新增导演功能时，制片功能不会跑偏。

制片账号在这一轮里的核心职责不是替导演审片，而是组织好导演要审的内容。

补充说明：

1. 路由、字段、状态码以 `Doc/通信协议文档.md` 为唯一协议基线
2. 页面命名、按钮命名、状态中文文案以 `Doc/平台功能与UI统一命名清单.md` 为唯一显示基线
3. 本文档重点回答“制片为什么做、做什么、先后顺序是什么”，不单独维护另一套接口命名

## 2. 角色定位

制片账号是导演审片流程的组织者和调度者，核心职责如下：

1. 管理项目与项目成员
2. 查看项目镜头整体情况
3. 整理待提审镜头
4. 指定提审版本
5. 处理镜头二级状态
6. 按顺序创建审片任务
7. 提交任务给导演
8. 跟踪导演审片结果与返修闭环

## 3. 制片与导演流程关系

制片端和导演端的关系建议固化如下：

1. 制片负责组织任务
2. 导演负责审片与反馈
3. 制作人员负责接收反馈并更新版本

所以制片端必须成为“任务装配台”，而不是只看镜头列表。

## 4. 左侧主导航关注点

制片账号仍沿用当前较完整的左侧主导航，但与导演审片强相关的重点页面为：

1. 仪表盘
2. 项目
3. 镜头
4. 审片

其中：

1. 仪表盘看项目总体与待处理事项
2. 镜头页做筛选与整理
3. 审片页做任务创建、任务跟踪、结果回收

## 5. 制片账号核心目标

围绕导演功能扩展后，制片账号第一优先级目标应变成：

1. 快速找到需要提审的镜头
2. 为每个镜头选定提审版本
3. 推进镜头内部审片二级状态
4. 把镜头按叙事或审片顺序编成任务
5. 提交给指定导演
6. 跟踪任务状态与导演反馈进度

## 5.1 制片对二级状态的职责

加入导演内部审片后，制片账号是镜头二级状态的主要组织者。

制片需要负责：

1. 将新版本镜头从 `未进入审片` 推进到 `待提审`
2. 将已完成文件同步、通过预检查且处于 `待提审` / `已按反馈修改` 的镜头加入导演审片任务，并在正式提交任务后由系统统一推进到 `审片中`
3. 在制作人员完成反馈并手动确认为 `已按反馈修改` 后，决定是否重新进入提审
4. 确保未进入 `内部通过` 的镜头不能走正式提交链路

## 6. 关键新增业务对象

制片账号必须显式操作“审片任务”。

### 6.1 当前正式任务协议口径

当前制片页正式围绕两类任务对象工作：

1. 底层实体：正式 `ReviewTask`
2. 页面投影：制片任务摘要 / 制片任务详情

当前正式路由口径：

1. `POST /api/review-tasks/draft`
2. `GET /api/review-tasks/producer`
3. `GET /api/review-tasks/{id}/detail`
4. `PUT /api/review-tasks/{id}`
5. `POST /api/review-tasks/{id}/shots`
6. `DELETE /api/review-tasks/{id}/shots/{taskShotId}`
7. `PUT /api/review-tasks/{id}/shots/reorder`
8. `POST /api/review-tasks/{id}/submit`
9. `POST /api/review-tasks/{id}/close`

补充说明：

1. 草稿创建与正式提交是两个动作，不能再混成一次请求
2. 制片页当前展示的 `待提交` 是服务端 `ready` 的页面投影名
3. 若页面继续显示“草稿 / 待提交 / 待审 / 审阅中 / 已完成 / 已关闭”，实现层必须按协议文档中的状态映射消费

一个审片任务建议包含：

1. 任务名称
2. 所属项目
3. 目标导演
4. 提审说明
5. 镜头集合
6. 镜头顺序
7. 每个镜头的提审版本
8. 截止时间
9. 状态

## 7. 制片仪表盘设计关注点

制片仪表盘建议补强以下区块：

1. 当前项目概览
2. 待创建审片任务镜头统计
3. 已提交审片任务统计
4. 待跟进返修统计

建议展示的数据：

1. 可提审镜头数
2. 待导演处理任务数
3. 导演已反馈镜头数
4. 待制作人员处理反馈数

## 8. 镜头页面对制片的要求

制片在镜头页面需要比导演更多的整理能力。

### 8.1 筛选能力

建议支持：

1. 按项目 / 集筛选
2. 按制作人员筛选
3. 按镜头主状态筛选
4. 按内部审片二级状态筛选
4. 按版本号筛选
5. 按是否已有可播放视频筛选
6. 按是否已加入审片任务筛选
7. 按是否已有导演反馈筛选

补充约定：

1. 镜头池列表与筛选结果的默认展示顺序以镜头号 `lensCode` 自然排序为准
2. `sceneNo / 场次` 仅作为辅助字段展示，不作为镜头池主排序字段
3. 镜头号包含数字后缀时，排序必须满足自然排序语义，例如 `SHOT05` 应排在 `SHOT05_1` 之前

### 8.2 列表字段

建议展示：

1. 镜头号
2. 当前版本号
3. 当前素材完整度
4. 当前负责人
5. 镜头主状态
6. 内部审片二级状态
7. 最近提审版本
8. 最近审片结论
9. 最近导演反馈时间
10. 所属审片任务

补充说明：

1. 列表中展示的“顺序”语义应与镜头号一致，避免用户误把场次当成镜头排序依据
2. 若页面存在“按镜头顺序重排”或“按镜头号重排”入口，其行为必须与镜头池默认排序一致

### 8.3 关键操作

1. 选择镜头加入审片任务
2. 为镜头指定提审版本
3. 标记为待提审
4. 重新提审
5. 按镜头号重排
6. 查看历史审片任务归属
7. 查看导演反馈

补充说明：

1. “按镜头号重排”仅调整当前草稿任务的镜头顺序，不改变镜头池源列表
2. 该动作完成后，任务草稿必须可通过本地草稿或等价机制保持顺序，刷新页面后不应丢失
3. 若后续需要支持按其他维度排序，必须新增独立入口，不得复用同一个按钮混淆语义

补充说明：

1. 制片不应手工直接把镜头改为 `内部通过`
2. `待处理反馈` 应由导演反馈动作触发，不建议由制片伪造
3. 镜头详情页中的 `镜头反馈` 与 `导演反馈` 必须按同级分页阅读：前者通过 `记录` 分页查看一级返修记录，后者通过 `导演反馈` 分页查看导演反馈卡片
4. 制片上传到返修记录中的图片在保存后必须由服务端真实持久化，后续导演与制作人员查看镜头详情时也能稳定预览和放大查看

## 9. 审片任务管理页设计

制片端建议在“审片”页下重点提供“审片任务管理”。

### 9.1 页面目标

1. 创建草稿任务
2. 编辑草稿或待提交任务
3. 提交任务给导演
4. 查看任务状态
5. 查看导演反馈回流情况

### 9.2 页面区块

1. 审片任务列表
2. 新建审片任务
3. 任务详情抽屉或详情页
4. 任务状态跟踪区

### 9.3 审片任务列表字段

1. 任务名称
2. 项目
3. 导演
4. 镜头数量
5. 已反馈数量
6. 当前状态
7. 提交时间
8. 截止时间

### 9.4 新建审片任务

创建时建议填写：

1. 任务名称
2. 目标导演
3. 提审说明
4. 截止时间
5. 镜头清单
6. 镜头顺序
7. 每个镜头的提审版本

当前阶段补充说明：

1. 当前客户端创建草稿时主契约上传的是 `shotIds`
2. 当前阶段每个镜头的 `submitVersionNum` 由服务端依据镜头当前 `VersionNum` 自动补齐
3. 若后续要支持制片在页面中手工覆盖单镜头提审版本，必须先升级协议，再改实现

4. 任务中镜头的加入顺序、拖拽顺序与“按镜头号重排”结果必须遵循统一排序口径，不能与镜头池默认顺序冲突

### 9.5 任务状态建议

1. 草稿
2. 待提交
3. 待审
4. 审阅中
5. 已完成
6. 已关闭

服务端真实状态映射：

1. `草稿` -> `draft`
2. `待提交` -> `ready`
3. `待审` -> `pending`
4. `审阅中` -> `in-review`
5. `已完成` -> `completed`
6. `已关闭` -> `closed`

## 10. 制片创建审片任务的标准链路

建议固化如下：

1. 制片进入镜头页筛选待提审镜头
2. 勾选镜头加入审片任务
3. 创建草稿任务或打开已有草稿任务
4. 调整镜头顺序
5. 填写任务说明与截止时间
6. 选择目标导演
7. 保存草稿
8. 确认任务结构完整后再正式提交
9. 提交后任务进入导演待审列表

补充说明：

1. “加入任务”与“提交任务给导演”是两个不同阶段，不能混用
2. 提交成功后，任务内镜头二级状态由系统统一推进到 `审片中`
3. 制片侧“正式提交”的前置条件，仍然包含镜头内部审片是否达到允许正式提交的全局规则

## 11. 视频版本选择规则要求

制片端在创建任务时，应尽量显式指定提审版本，但导演端播放仍遵循统一兜底规则：

1. 优先使用提审版本号对应视频
2. 若无对应视频则使用最高版本号视频
3. 若无版本视频则使用 Layout 视频
4. 若无任何可播放素材则标记缺失

制片端页面应提前提示素材可用性，避免提交后导演才发现无法联播。

## 12. 制片查看导演反馈的页面要求

制片需要查看导演反馈，但重点不是画面标注本身，而是推进后续流转。

建议在任务详情中提供：

1. 当前任务反馈进度
2. 每个镜头的反馈数量
3. 每个镜头的镜头结论
4. 最新反馈卡片预览
5. 是否已同步给制作人员

## 13. 与制作人员的协同关系

制片是导演反馈回流给制作人员的中间管理者，重点职责包括：

1. 确认导演反馈已经形成结构化记录
2. 跟踪哪些镜头需要返修
3. 跟踪对应制作人员是否已处理
4. 必要时重新发起下一轮审片任务

补充约定：

1. 制片在“项目”页与“文件检查”页维护的正式路径配置口径统一为“镜头文件根目录”和“Layout 根目录”
2. 镜头版本 MA 与拍屏/视频文件共享同一组镜头根目录配置，不再单独维护默认 MA 根目录或默认视频根目录概念
3. Layout 根目录仅服务于 Layout Maya、Layout 视频扫描与引用排查，不参与镜头版本文件匹配

## 14. 权限边界

制片账号建议具备以下与审片任务相关的能力：

1. 创建审片任务
2. 编辑草稿任务
3. 调整镜头顺序
4. 指定提审版本
5. 提交任务给导演
6. 查看全部导演反馈
7. 关闭任务

制片账号不负责：

1. 替导演创建反馈卡片
2. 修改导演已保存的标注内容
3. 代替制作人员提交新版本

## 15. 制片端第一版范围建议

第一版建议完成：

1. 审片任务创建与编辑
2. 镜头加入任务与排序
3. 提审版本指定
4. 镜头二级状态管理
5. 任务提交给导演
6. 任务状态跟踪
7. 任务内查看导演反馈结果

当前已落地的一版基线补充：

1. 草稿任务创建、任务列表、任务详情、任务更新、加镜头、移除镜头、提交任务、关闭任务已进入正式文档基线
2. 任务镜头顺序重排属于正式协议基线，但页面交互仍可继续增强
3. “单镜头直接进入审片”不再视为制片正式派工能力

第二期可增强：

1. 审片任务模板
2. 常用镜头排序方案复用
3. 按导演维度统计负载
4. 自动生成返修清单

## 16. 与导演账号文档的对应关系

后续讨论时，制片端应始终围绕以下问题核对：

1. 这个功能是否帮助制片更高效地组织审片任务
2. 这个功能是否会把导演端任务组织责任错误转移给导演
3. 这个功能是否能确保导演看到的是已经整理好的镜头顺序
4. 这个功能是否能保障反馈回流后可继续推进返修闭环

## 17. 制片与提交权限的关系

接入内部审片后，制片端必须承担提交前校验责任。

规则如下：

1. 当前镜头二级状态不是 `内部通过` 时，不允许正式提交
2. 制片在镜头列表和详情中都需要看到是否“允许提交”的明确提示
3. 若当前版本升级导致二级状态清空，制片端应立即重新显示为不可提交
