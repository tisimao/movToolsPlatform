## 2. 本批次目标

这一批完成后，客户端必须具备：

1. 制作人员产出新版本后，镜头在 `待处理反馈` 阶段不被页面误判或自动回退为 `未进入审片`
2. 制作人员只有在明确确认“本轮反馈已处理完成”后，才执行 `待处理反馈 -> 已按反馈修改`
3. 制片完成文件同步核对与预检查后，再手动推进镜头到 `待提审`
4. 制片把镜头加入草稿任务时，镜头保持原二级状态，不因草稿装配提前进入 `审片中`
5. 任务正式提交后，只把 `待提审` / `已按反馈修改` 的镜头视为进入 `审片中`
6. 已处于 `内部通过` 的镜头在任务装配和普通任务提交时继续保持通过态
7. 只有导演后续再次提交新反馈时，已通过镜头才重新进入 `待处理反馈`
8. 创建 / 编辑审片任务时不再展示或收集截止时间

这一批不做：

1. 不重写整套播放器或标注系统
2. 不重做客户端本地文件扫描职责边界
3. 不新增另一套平行审片任务模型
4. 不允许页面层继续保留“版本一变化就自动回 `未进入审片`”的旧口径

---

## 3. 当前代码状态（按现状理解）

当前客户端已具备：

1. 制片草稿创建、制片任务列表、任务详情、更新、加镜头、删镜头、提交、关闭等基础消费链路
2. 导演任务列表、任务详情、评论、反馈卡片等主链路基础
3. 制片页 `ready -> 待提交`、导演页 `completed/closed -> approved/rejected` 的状态投影

当前仍存在的主要风险点：

1. 页面层可能仍残留“新版本产出后自动回 `未进入审片`”的旧假设
2. 草稿任务装配、正式提交、镜头二级状态推进之间的页面口径仍需继续收口
3. 已通过镜头在再次装配任务、查看任务详情、刷新列表时，可能存在被误显示为普通待处理镜头的风险
4. 制作人员、制片、导演三角色对二级状态动作的文案、按钮显示和前置条件仍需再统一

---

## 4. 已固化业务边界（必须遵守）

1. 新版本产出后，不自动把 `待处理反馈` 打回 `未进入审片`
2. 制作人员确认处理完成前，镜头保持 `待处理反馈`
3. 草稿任务只表示任务装配，不直接改写镜头二级状态
4. 正式提交任务后，只推动 `待提审` / `已按反馈修改` 的镜头进入 `审片中`
5. 已处于 `内部通过` 的镜头，不因草稿任务或普通任务提交自动进入 `审片中`
6. 已通过镜头只有在导演再次提交新反馈时，才重新进入 `待处理反馈`
7. 当前镜头二级状态不是 `内部通过` 时，正式提交入口必须继续被收口
