## 2. 本批次目标

这一批完成后，客户端必须具备：

1. 新设备首次激活协同项目时，若本地工作区不存在，可自动 bootstrap 本地工作区骨架
2. `projectRootPath`、`lensFolderRootPath`、`layoutCheckPath` 的语义彻底分离，不能再互相冒充
3. 跨设备盘符不同但路径结构一致时，可自动完成 Windows 盘符重定位
4. 盘符重定位以项目根目录为手动入口；镜头根目录和 Layout 根目录继续沿用服务端路径，只按新盘符自动重映射
5. 在本地 `project.create()` / bootstrap 前，先对“本机最终镜头根目录”与“本机最终 Layout 根目录”做冲突预校验
6. 若缺失的是工作区骨架，允许自动初始化；若缺失的是必须真实存在的素材目录，必须给出明确错误
7. producer / maker 在“空本地工作区的新设备首次激活”场景下行为一致且可验收

这一批不做：

- 不改服务端目录
- 不把盘符重定位逻辑迁回服务端
- 不回退到要求制作人员手工重新选择镜头根目录或 Layout 根目录的主流程
- 不允许把 `projectRootPath` 继续当成镜头根目录或 Layout 根目录的兜底值

---

## 3. 已固化业务边界（必须遵守）

1. 服务端保存的是协同标准路径，不要求所有设备盘符一致
2. 客户端负责将服务端路径解析为本机最终路径，并负责 Windows 盘符差异重定位
3. `projectRootPath` 只表示项目工作区承载目录；`lensFolderRootPath` 与 `layoutCheckPath` 是文件检查上下文，三者语义不同，手动重定位只允许重选 `projectRootPath`
4. 若本机最终镜头根目录与本机最终 Layout 根目录重复、互相包含，客户端必须直接阻断激活
5. bootstrap 仅用于创建本地工作区骨架，不代表允许伪造业务素材目录
6. 只有当盘符后的路径结构一致且本机候选目录真实存在时，才允许采用自动重定位结果
7. 当前阶段只要求支持 Windows 本地盘符差异，不要求自动兼容 UNC、NAS、目录改名或路径结构变化场景

---

## 4. 问题定性

当前问题必须按以下结论理解：

1. 新设备首次激活失败的核心，不是服务端没有项目，而是本地工作区骨架不存在、且当前激活路径解析可能混淆了项目根与镜头根
2. 现有激活链路在本地 bootstrap 前缺少“解析后的镜头根 / Layout 根冲突预校验”，导致错误在本地 `createProject()` 阶段才暴露
3. 不同设备盘符不同是正常协同场景，不应被视为异常；真正需要校验的是“盘符后的路径结构”和“本机目录是否真实存在”
4. 若继续允许 `projectRootPath` 跨语义兜底为镜头根目录或 Layout 根目录，会在新设备激活时持续制造误判与冲突
