# 镜头状态与内部审片状态设计文档

## 1. 文档目的

本文件用于明确“对客户的镜头状态”和“内部导演审片状态”之间的边界，避免在后续导演功能接入后把两套语义混在一起。

本版已补充定稿以下关键结论：

1. 内部审片状态是镜头二级状态
2. 二级状态与当前镜头版本强绑定
3. 当前版本升级不会自动清空二级状态，必须由角色确认动作继续推进
4. 二级状态未进入“内部通过”前，镜头不允许正式提交

核心结论只有一句话：

1. 镜头主状态继续表示对客户的正式交付状态
2. 导演审片新增一套独立的内部审片状态，不直接覆盖镜头主状态

## 2. 为什么不能直接复用现有镜头状态

当前镜头状态本质上是面对客户交付与项目外部协作的状态。

如果把导演内部审片直接写进镜头主状态，会产生 4 个问题：

1. 客户状态和内部状态语义混淆
2. 一个镜头可能还没给客户，但已经历多轮导演审片，单一状态无法表达
3. 导演“返修”和镜头正式“返修”不是同一层级动作
4. 后续统计时无法区分“内部未过”还是“对客户未交”

所以必须拆成两层状态。

## 3. 建议的双层状态模型

一个镜头以后至少有两套状态：

1. 镜头主状态：面向客户 / 正式流程
2. 内部审片状态：面向导演 / 内部流程

建议再补一个任务维度：

3. 审片任务状态：面向一轮具体提审任务

## 4. 第一层：镜头主状态

镜头主状态继续保持“正式业务状态”定位，不直接承担导演审片细节。

建议镜头主状态继续使用现有主链路，例如：

1. 未开始
2. 制作中
3. 待提交
4. 已提交
5. 返修
6. 通过
7. 关闭

这里的含义不变：

1. 它描述镜头在正式制作/交付链路里的位置
2. 它是列表主状态、统计主状态、对客户口径主状态
3. 它不用于表达导演某一轮具体反馈是否完成

## 5. 第二层：内部审片状态（镜头二级状态）

内部审片状态建议新增到镜头维度，作为镜头二级状态存在。

它只表示“当前镜头版本在内部导演审片链路中的最新情况”。

这里要特别定稿：

1. 二级状态不是镜头长期历史状态
2. 二级状态只服务当前版本
3. 当前版本变化后，不再简单采用“自动清空状态”策略，而是按本轮反馈是否已处理完成继续推进

建议状态如下：

1. 未进入审片
2. 待提审
3. 审片中
4. 待处理反馈
5. 已按反馈修改
6. 内部通过

### 5.1 状态解释

`未进入审片`

1. 镜头尚未纳入任何导演审片流程
2. 可能仍在制作中

`待提审`

1. 镜头已具备一版可供内部审片的内容
2. 但还没被制片放进具体审片任务

`审片中`

1. 镜头已经进入某个导演审片任务
2. 导演正在看，或任务正在处理中

`待处理反馈`

1. 导演已经给出反馈
2. 制作人员尚未完成本轮修改

`已按反馈修改`

1. 制作人员已根据导演反馈完成修改并产出新版本
2. 等待制片决定是否进入下一轮提审

`内部通过`

1. 导演对当前镜头版本的内部审片结果认可
2. 但不自动等于客户正式通过

### 5.2 二级状态与版本联动规则（定稿）

内部审片状态与当前镜头版本强绑定，必须遵守以下规则：

1. 二级状态默认描述的是“当前镜头当前轮内部审片处理状态”
2. 不单独设计“内部通过版本”字段
3. 当前镜头版本升级后，不以“自动清空状态”作为默认策略，而是按角色确认动作继续推进
4. 当前镜头处于 `待处理反馈` 时，新版本产出本身不自动把状态改回 `未进入审片`
5. 制作人员确认“本轮反馈已经处理完成”后，才允许把状态推进到 `已按反馈修改`
6. `内部通过` 状态不因创建草稿任务或重新纳入任务装配而自动丢失；只有导演新的反馈动作或明确人工调整，才改变该状态

补充说明：

1. `内部通过` 的含义是“导演已手动确认该镜头当前处于通过态”
2. `待处理反馈` 表示这轮导演反馈尚未被制作人员确认处理完成，因此新版本出现后仍可保持该状态
3. 草稿任务只表示任务装配动作，不直接改写镜头二级状态

## 6. 第三层：审片任务状态

审片任务状态只描述“一次任务”的进度，不描述镜头整体生命周期。

补充说明：

1. 页面中文文案以 `草稿 / 待提交 / 待审 / 审阅中 / 已完成 / 已关闭` 表达
2. 协议真实状态以 `draft / ready / pending / in-review / completed / closed` 为当前正式基线
3. 制片页显示的 `待提交` 是 `ready` 的页面投影
4. 导演页显示的 `通过 / 返修` 是 `completed / closed` 的结果投影

建议状态如下：

1. 草稿
2. 待提交
3. 待审
4. 审阅中
5. 已完成
6. 已关闭

说明：

1. 任务状态属于任务对象
2. 内部审片状态属于镜头对象
3. 镜头主状态属于正式业务对象

三者不能混用。

## 7. 推荐的状态关系

最稳妥的做法不是让三套状态互相覆盖，而是定义“驱动关系”。

建议关系如下：

1. 制片可将镜头从“未进入审片”推进到“待提审”
2. 制片创建草稿任务后，镜头仍保持原二级状态，不因为草稿存在而自动进入“审片中”
3. 制片正式提交审片任务后，处于 `待提审` / `已按反馈修改` 的镜头进入“审片中”
4. 导演保存反馈后，镜头内部审片状态变成“待处理反馈”
5. 制作人员完成反馈修改并确认后，可手动把镜头内部审片状态变成“已按反馈修改”
6. 制片再次把镜头推进到“待提审”并纳入下一轮任务，正式提交后才重新进入“审片中”
7. 导演确认本轮无问题后，镜头内部审片状态变成“内部通过”
8. 已处于 `内部通过` 的镜头不会因为草稿任务或普通任务装配自动回退到“审片中”
9. 若导演后续又对一个已通过镜头提交新反馈，则该镜头再进入“待处理反馈”

而镜头主状态只在正式业务节点变化，例如：

1. 提交给客户时改“已提交”
2. 客户要求返修时改“返修”
3. 客户确认后改“通过”或“关闭”

## 8. 导演返修和镜头返修如何区分

这一点必须严格分开。

### 8.1 导演返修

导演在审片中说“这个镜头要改”，本质上是：

1. 一次内部审片结论
2. 审片任务动作
3. 对应镜头内部审片状态进入“待处理反馈”

它不应该直接把镜头主状态改成正式 `返修`。

### 8.2 镜头返修

镜头正式 `返修` 仍表示：

1. 进入正式状态流转
2. 属于镜头主状态变化
3. 对应现有 `REWORK` 语义

只有当你们业务明确规定“导演内部打回就等同正式返修”时，才允许由制片在确认后把内部结论同步推动到镜头主状态。

建议不要自动联动，建议由制片做一次确认动作。

## 9. 推荐的实际流转方式

建议按下面的闭环处理：

1. 制作人员产出一个当前镜头版本
2. 制作人员完成当前版本文件同步，保证版本文件与绑定信息可回读
3. 制片执行预检查，确认当前版本具备进入内部审片的条件
4. 制片将镜头状态推进到“待提审”
5. 制片把镜头加入草稿任务
6. 草稿阶段镜头仍保持“待提审”或“已按反馈修改”等原二级状态，不自动进入“审片中”
7. 制片正式提交任务给导演
8. 镜头内部审片状态变为“审片中”
9. 导演给出图文反馈
10. 镜头内部审片状态变为“待处理反馈”
11. 制作人员完成修改并产出新版本
12. 新版本产出后，镜头仍保持“待处理反馈”
13. 制作人员确认本轮反馈已处理后，手动将状态改为“已按反馈修改”
14. 制片重新同步文件并做预检查
15. 制片将镜头重新推进到“待提审”
16. 制片把镜头加入下一轮草稿任务
17. 草稿阶段镜头保持原状态，正式提交后才再次进入“审片中”
18. 导演确认通过
19. 镜头内部审片状态变为“内部通过”
20. 已通过镜头保持“内部通过”，除非导演后续又提交新的反馈

在这个过程中，镜头主状态可以仍然保持“制作中”或“已提交”，不被内部审片频繁扰动。

## 9.1 正式提交限制（定稿）

镜头接入内部审片后，正式提交必须受二级状态约束。

规则如下：

1. 当前镜头二级状态不是 `内部通过` 时，不允许正式提交
2. 只有当前镜头版本已经进入 `内部通过`，才允许进入对客户的正式提交链路
3. 若当前镜头进入 `待处理反馈` 或 `已按反馈修改`，正式提交能力仍保持锁定，直到导演再次手动确认通过

这条规则用于确保“内部未通过的当前版本，不能提交给客户”。

## 9.2 角色改状态边界（定稿）

### 制片

制片可处理二级状态的组织性动作：

1. `未进入审片 -> 待提审`
2. `待提审 -> 审片中`（通过创建或提交审片任务触发）
3. `已按反馈修改 -> 待提审`
4. 必要时做人工纠正或回退

### 导演

导演不以“手工切状态”为主，而是通过审片动作驱动状态：

1. 保存反馈后进入 `待处理反馈`
2. 确认通过后进入 `内部通过`

### 制作人员

制作人员只负责反馈处理完成后的状态确认：

1. 在收到反馈并完成修改后，将镜头标记为 `已按反馈修改`
2. 制作人员不能把镜头直接改成 `内部通过`
3. 制作人员不能负责把镜头送入 `审片中`

## 10. 页面展示建议

接入导演功能后，镜头列表和镜头详情建议同时展示两类状态。

### 10.1 镜头列表

建议字段：

1. 镜头主状态
2. 内部审片状态
3. 最近审片任务
4. 最近导演反馈时间
5. 当前版本号

展示原则：

1. 主状态作为主标签
2. 内部审片状态作为次标签
3. 不要把两者合成一个枚举字段

### 10.2 镜头详情

建议单独展示两个区块：

1. 正式状态
2. 内部审片状态

并在 `导演反馈` 分页中或其分页头部附近显示：

1. 最近审片任务
2. 最近反馈结论
3. 是否存在待处理反馈
4. 当前版本是否已内部通过

## 11. 数据模型建议

建议在现有镜头对象上新增内部审片字段，而不是替换原字段。

镜头对象建议新增：

1. `internalReviewStatus`
2. `internalReviewUpdatedAt`
3. `latestReviewTaskId`
4. `latestDirectorFeedbackAt`
5. `pendingDirectorFeedbackCount`
6. `currentVersionSubmittedAllowed`

补充说明：

1. 不新增“内部通过版本号”字段
2. 二级状态直接绑定当前版本理解即可
3. 是否允许正式提交，可由当前二级状态实时推导，也可冗余缓存为只读字段
4. 反馈卡片对应版本号必须落库到评论/反馈表，作为服务端可追溯事实，而不是只依赖任务镜头上下文推导

审片任务对象继续保存：

1. `taskStatus`
2. `directorId`
3. `submittedAt`
4. `completedAt`

镜头与任务关联表可保存：

1. `taskId`
2. `shotId`
3. `submitVersionId`
4. `submitVersionNum`
5. `taskShotStatus`
6. `feedbackCount`
7. `lastFeedbackAt`

## 12. 推荐的状态命名

为了避免以后协议混乱，建议命名从一开始就分层。

### 12.1 镜头主状态

保留现有 `status` / `statusCode` 语义。

### 12.2 内部审片状态

新增：

1. `internalReviewStatusCode`
2. `internalReviewStatusName`

建议稳定码示例：

1. `NOT_IN_REVIEW`
2. `READY_FOR_REVIEW`
3. `IN_DIRECTOR_REVIEW`
4. `PENDING_FEEDBACK_FIX`
5. `FIX_UPDATED`
6. `DIRECTOR_APPROVED`

### 12.3 审片任务状态

建议稳定码示例：

1. `draft`
2. `ready`
3. `pending`
4. `in-review`
5. `completed`
6. `closed`

## 13. 最关键的定稿建议

如果你现在要把导演审片功能接进现有系统，我最建议的做法是：

1. 不改掉现有镜头主状态的业务定位
2. 新增一套“内部审片状态”字段
3. 新增“审片任务状态”作为任务层状态
4. 内部审片状态作为镜头二级状态，并且与当前版本强绑定
5. 当前版本一旦升级，二级状态立即清空并重新开始
6. 导演反馈只驱动内部审片状态，不直接改镜头主状态
7. 当前二级状态未进入“内部通过”前，镜头不允许正式提交
8. 如需把内部结论同步到正式镜头状态，必须通过制片确认动作触发

## 14. 适合作为后续讨论的统一口径

以后讨论状态时，建议固定使用这 3 个名词：

1. 镜头主状态
2. 内部审片状态
3. 审片任务状态

不要再笼统说“镜头状态”，否则后面 UI、接口、统计、权限都会乱。
