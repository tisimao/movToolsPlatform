# 镜头详情页设计文档

## 1. 文档目的

本文件用于单独固化镜头详情页的页面定位、当前真实实现状态、区块层级与后续角色化描述口径。

本文件优先解决以下问题：

1. 当前文档体系里缺少对镜头详情页本身的完整描述
2. 制片、导演、制作人员都在引用镜头详情页，但对页面结构的理解并不完全一致
3. 当前实现里“导演反馈”已经有独立区块，“镜头反馈/返修记录”仍主要挂在记录流中，层级表达不够清晰
4. 一级返修图片能力与导演反馈图片回看能力已完成闭环，当前需要明确的是页面层的稳定语义与展示边界

## 2. 页面定位

镜头详情页是围绕“单个镜头”的综合信息页，用来承接以下事实：

1. 镜头主状态与内部审片二级状态
2. 当前版本、文件绑定、Layout 候选与本机预览能力
3. 审片任务回挂信息
4. 一级镜头反馈（返修记录、返修图片）
5. 导演反馈卡片回看
6. 生命周期记录与排查信息

当前基线下，镜头详情页不是导演主审片入口，但它仍然是三类角色共享的镜头级事实查看入口。

## 3. 当前实现真实状态

当前客户端真实实现位于 `movtools-client/src/pages/LensPage.tsx`，镜头详情以弹窗形式打开，不是独立路由页。

### 3.1 弹窗形态

当前镜头详情是一个可最大化、可拖拽调尺寸的详情弹窗，包含：

1. 标题区
2. 顶部摘要卡区
3. 下一步建议与快捷操作区
4. 双视频预览区
5. `版本 / Layout / 导演反馈 / 记录` 四标签内容区

### 3.2 顶部标题区当前显示项

当前标题区会展示：

1. 镜头号
2. 镜头主状态
3. 当前版本号
4. 生命周期记录数
5. 当前制作人
6. 制作人匹配状态

### 3.3 摘要卡区当前显示项

当前摘要卡已包含以下信息：

1. 当前版本
2. 内部审片二级状态
3. 最近导演反馈数量与时间
4. 最近所属审片任务
5. 制作信息
6. 版本文件完整度
7. Layout 状态
8. Layout 引用排查状态
9. 最近状态动作
10. 最近更新时间

### 3.4 快捷操作区当前真实状态

当前快捷操作区已混合承载：

1. 刷新详情
2. 文件检查
3. 跳转到制片任务页 / 导演审片页
4. 编辑基础信息
5. 绑定当前版 MA / MOV
6. 补充 Layout / 检查 Layout 引用
7. 正式提交
8. 内部通过
9. 要求修改
10. 一级返修
11. 关闭

这意味着当前镜头详情已经同时承接“正式镜头状态流转”和“内部审片状态流转”。

### 3.5 反馈相关区当前真实状态

当前实现已调整为并列分页表达：

1. 一级镜头反馈仍主要沉淀在 `记录` 标签内的生命周期时间线中
2. 导演反馈改为单独的 `导演反馈` 标签页
3. `导演反馈` 标签页的界面形式与 `记录` 区保持同一阅读模型
4. 一级返修图片的新增/编辑能力通过返修弹窗与返修记录编辑弹窗承载
5. 反馈编辑态与回看态均已支持缩略图展示与放大预览

也就是说，当前页面不再把导演反馈悬浮在页面主区，而是让它和记录区并列存在。

## 4. 当前已实现能力拆分

### 4.1 一级镜头反馈链路（当前已实现）

这里的“一级镜头反馈”指镜头主状态链路中的返修记录，而不是导演内部审片反馈卡片。

当前已具备：

1. 制片或有权限角色可对镜头执行 `返修`
2. 返修时可填写返修说明
3. 返修时可选择本地图片
4. 返修时支持直接粘贴剪贴板图片
5. 返修图片支持排序、删除
6. 已保存返修记录支持再次编辑
7. 已保存返修图片支持保留/移除、排序、预览

这套能力已经具备“图片选择 + 粘贴板导入 + 本地路径收集 + 顺序管理 + 已保存记录编辑”的完整雏形。

### 4.2 导演反馈链路（当前已实现）

当前镜头详情页更适合作为导演反馈回看入口，已具备：

1. 按卡片查看导演反馈的文字说明
2. 查看帧号、时间码、版本号、反馈编号
3. 查看标注图、原图、缩略图等图片产物

补充约定：`导演反馈` 分页中的视频回放若需要展示正式绘制内容，必须和导演审片页保持同一正式反馈口径，即基于“当前版本 + 当前轮”反馈集合进行回放，不允许混用镜头详情接口自带的简化反馈摘要与正式回放 timeline。

当前仍不应在镜头详情里主做的能力：

1. 画面暂停定帧
2. 画布标注
3. 从视频直接产出原图/标注图/缩略图
4. 本地临时草稿态
5. 任务级统一提交反馈

因此当前镜头详情中的导演反馈标签页，应视为“回看入口”，而不是正式主编辑入口。

补充说明：当前反馈图片展示已统一区分本地草稿预览与服务端正式回读；本地草稿可先用 `file://` 预览，但服务端返回的 `/uploads/...`、`http://`、`https://` 与完整 URL 都必须按可访问地址渲染，不得再误转为本地 `file://`。

## 5. 镜头详情页区块层级基线

为避免后续继续混淆，本页区块层级正式建议收口如下：

### 5.1 一级层级

镜头详情页建议固定为以下一级区块：

1. 基础摘要
2. 快捷操作
3. 视频预览
4. 版本资产
5. Layout 候选与映射
6. 镜头反馈（记录标签）
7. 导演反馈（导演反馈标签）

### 5.2 反馈区层级要求

`镜头反馈` 与 `导演反馈` 必须是同一级别区块，不允许再出现以下混乱表达：

1. 把导演反馈放在镜头反馈之下，误导为一级返修的子类型
2. 把镜头反馈放在记录标签、导演反馈放在导演反馈标签，但两者阅读层级必须并列
3. 让不同角色看到的“反馈”实际上引用了不同层级概念

统一口径应为：

1. `镜头反馈`：镜头主状态链路中的返修记录、返修说明、返修图片
2. `导演反馈`：导演内部审片链路中的反馈卡片、标注图、说明、帧信息、任务回挂信息

补充要求：

1. `记录` 标签中的返修图片在保存后必须成为服务端正式附件资产
2. 导演、制作人员、制片重新登录后都必须还能看到返修图片预览
3. 点击返修图片缩略图时必须支持放大查看

## 6. 角色化阅读口径

### 6.1 制片账号

制片查看镜头详情时，重点关注：

1. 当前镜头是否具备提审条件
2. 当前二级状态处于哪一步
3. 最近属于哪个审片任务
4. 是否已有导演反馈
5. 一级返修记录与图片是否完整可追溯

制片不是导演反馈的主编辑者，但需要在镜头详情中同时看到：

1. 镜头反馈
2. 导演反馈
3. 两者之间的区别与先后关系

### 6.2 导演账号

导演查看镜头详情时，重点是辅助回看，而不是主编辑。

当前口径应为：

1. 导演主编辑入口在审片工作台
2. 镜头详情中的导演反馈标签只保留回看或辅助入口能力
3. 若保留临时编辑能力，也必须明确弱化，避免与审片工作台形成双入口竞争

补充说明：后续若对镜头详情页导演反馈回放做重构，只允许改回看链路，不允许反向改动导演审片工作台中的绘制、暂停标注、草稿、提交主流程。

### 6.3 制作人员账号

制作人员查看镜头详情时，重点关注：

1. 导演反馈卡片
2. 一级返修记录
3. 当前版本本机预览
4. 文件绑定与版本素材完整度
5. 是否可以执行“已按反馈修改”

制作人员对镜头详情的核心需求，是把“导演反馈”作为可消费事实，而不是作为可编辑事实。

## 7. 当前结论与稳定口径

当前镜头详情页的图片与反馈能力已经形成稳定闭环，后续文档不再以“待补齐”描述该部分，而应按以下口径固定：

1. `记录` 分页承载一级返修记录、返修说明、返修图片与生命周期事件
2. `导演反馈` 分页承载导演内部审片反馈卡片、标注图、原图、缩略图与版本信息
3. 反馈编辑态的本地草稿必须先展示缩略图；提交后必须转为可回读的服务端正式图片
4. 服务端正式图片地址必须能在重新登录、重开镜头详情后继续预览与放大
5. 页面层不得再把服务端图片路径误判成本地文件路径

补充说明：镜头详情页是导演审片页的素材真相源之一。导演审片页进入播放前，必须先复用本页已经完成的版本视频匹配结果，再按版本优先、Layout 兜底规则消费最终可播放 URL；审片页不得自行重新匹配并覆盖本页结果。

补充约定：当导演审片页引入播放器快捷键后，快捷键说明应固定在页面顶部说明区，不得作为视频遮罩叠加在画面上，以免干扰镜头查看。

## 8. 对后续文档编写的约束

后续再写制片、导演、制作人员相关文档时，凡提到镜头详情页，必须遵守：

1. 先引用本文件的页面结构
2. 再描述角色可见区块与可执行动作
3. 不允许跳过页面结构，直接写角色能力，避免再次把不同区块混成一团
4. 不允许把“镜头反馈”和“导演反馈”混写成一个泛化反馈概念

## 9. 当前建议结论

当前镜头详情页的真实状态可以概括为：

1. 基础信息、版本、Layout、预览、记录区已经较完整
2. 一级返修图片能力已经存在，并继续通过 `记录` 标签中的返修记录详情承载
3. 导演反馈已改为与 `记录` 并列的独立标签页，用相同的图文卡片阅读模型回看
4. 后续应先收口一级返修图片能力模块，再复用到导演审片标注能力
5. 页面信息架构上，`镜头反馈` 与 `导演反馈` 必须明确为同级区块

6. `导演反馈` 分页中的视频回放需要与导演审片页正式绘制结果一致，这一一致性应通过镜头详情页自身回放链路收口实现，而不是通过修改导演主审片编辑链路实现
