# 导演功能三次迭代验收派工指令包

## 1. 文档用途

本文件用于把“导演反馈条目与绘制时间线三次迭代”的联调验收任务直接派发给其他 AI 或测试执行。

本批验收覆盖：

1. 客户端三次迭代实现结果
2. 服务端三次迭代实现结果
3. 双方与 `Doc/通信协议文档.md`、`Doc/项目固化文档.md`、`Doc/导演账号功能设计文档.md` 的一致性

客户端目录：`D:\project\MovtoolsEx\movtools-client\`

服务端目录：`D:\project\MovtoolsEx\movtools-server\`

---

## 2. 本批次验收目标

只有当以下目标同时成立，才可判定本批通过：

1. 导演反馈条目与绘制时间线已在产品、协议、实现三层彻底分层
2. 导演工作台提交前后显示一致，不再出现“提交后继承帧绘制丢失”
3. 镜头详情页、导演反馈回放区、制作查看页对同一轮反馈的绘制回放完全一致
4. 服务端已经提供稳定的轮次级正式 timeline，客户端正式查看页优先消费该字段
5. 旧字段兼容、任务切换、关闭任务、重新进入任务后的缓存行为正确，无旧绘制残留

---

## 3. 已固化业务边界（必须遵守）

1. `反馈条目` 与 `绘制时间线` 是两个不同对象层级
2. 反馈条目按帧绑定，绘制时间线按 `feedbackRoundId` 绑定
3. 正式绘制显示必须通过轮次级 timeline 回放决定，不能靠“当前帧是否有 feedback 条目”决定
4. 导演工作台草稿态允许本地未提交绘制保留链，但正式提交后所有查看页都必须只认正式 timeline
5. 历史 `annotationDataJson` 与 feedback 级 `drawingFrames` 只保留兼容读取意义，不再作为正式主真相源

---

## 4. 验收总原则

1. 先核对协议文档，再核对实现；若实现与文档不一致，以文档为准
2. 先验服务端正式回包与时间线真相源，再验客户端页面回放结果
3. 不接受“导演工作台修好了，但镜头详情页仍按旧逻辑显示”的半通过
4. 不接受“前端本地草稿预览正常，但提交后正式回看仍丢帧”的假通过
5. 不接受“某个页面还在按单条 feedback 读取正式绘制”的残留实现

## 5. 直接可复制的总验收指令

```md
你现在执行“导演反馈绘制时间线三次迭代联调验收”任务。

本任务目标：

1. 验证反馈条目与绘制时间线是否已正式分层
2. 验证服务端是否稳定返回轮次级正式 timeline
3. 验证导演工作台提交前后是否保持一致显示
4. 验证镜头详情页、导演反馈回放区、制作查看页是否与导演工作台正式结果一致
5. 验证旧字段兼容、任务切换、关闭任务后的缓存与历史数据行为是否正确

你必须直接进入“核对文档 -> 跑接口 / 跑页面 -> 记录结果 -> 判定通过/阻塞”闭环，不能只停留在阅读。

参考文档：
- D:\project\MovtoolsEx\Doc\项目固化文档.md
- D:\project\MovtoolsEx\Doc\通信协议文档.md
- D:\project\MovtoolsEx\Doc\导演账号功能设计文档.md
- D:\project\MovtoolsEx\Doc\导演账号功能三次迭代客户端派工指令包.md
- D:\project\MovtoolsEx\Doc\导演账号功能三次迭代服务器端派工指令包.md

工作目录：
- D:\project\MovtoolsEx\movtools-client\
- D:\project\MovtoolsEx\movtools-server\

必须完成的验收项：

一、协议层验收
1. 核对 `GET /api/review-feedbacks/lens/{lensId}` 的返回是否稳定包含：
   - `feedbacks`
   - `latestRound`
   - `latestRound.feedbackRoundId`
   - `latestRound.drawingFrames`
2. 核对协议是否明确：
   - feedback 条目按帧绑定
   - drawing timeline 按轮次绑定
   - 正式读取优先级为 `round timeline` -> `feedback.drawingFrames` -> `annotationDataJson`
3. 核对 `feedbacks[]` 中若仍返回 `drawingFrames`，是否仅作为兼容字段，而不是正式主字段

二、服务端正式 timeline 验收
4. 创建一轮反馈后，验证服务端可回读该轮完整 timeline
5. 只改反馈文字时，验证 timeline 不被误清空或误覆盖
6. 只改绘制时，验证 timeline 被正确更新
7. 关闭任务后，验证该任务相关 feedback 与 timeline 都被清理
8. 历史数据若只有 `annotationDataJson` 或 feedback 级 `drawingFrames`，验证仍可兼容回读

三、导演工作台提交前后显示一致性验收
9. 使用导演账号进入审片工作台，构造以下用例：
   - 第 3 帧创建反馈并绘制内容 A
   - 第 10 帧清屏
   - 第 15 帧绘制内容 B
   - 第 20 帧清屏
10. 在未提交前逐帧播放，验证：
   - 第 3 帧显示反馈条目和 A
   - 第 4-9 帧显示 A，无反馈条目
   - 第 10-14 帧空白
   - 第 15-19 帧显示 B，无反馈条目
   - 第 20 帧及之后空白
11. 点击提交后再次逐帧播放，验证结果与提交前完全一致

四、跨页面一致性验收
12. 在导演工作台提交后，进入镜头详情页 `导演反馈` 分页，验证正式遮罩逐帧结果与导演工作台一致
13. 在导演反馈回放组件中验证同一轮遮罩回放结果一致
14. 使用制作人员账号进入对应镜头查看页，验证看到的正式反馈条目与正式绘制结果与导演提交结果一致
15. 验证不存在“导演端能看到 A，制作端看不到 A”或“详情页只显示第 3 帧、其余继承帧为空”的问题

五、任务切换与缓存验收
16. 提交成功后刷新页面重新进入任务，验证正式 timeline 正常回读
17. 切换到其他镜头再切回，验证不会串镜头绘制缓存
18. 切换到其他任务再切回，验证不会串任务/串轮次绘制缓存
19. 关闭任务后重新以同镜头创建新任务，验证新任务默认不残留旧 feedback 和旧正式绘制

六、构建与回归
20. 客户端 `npm run typecheck` 与 `npm run build` 通过
21. 服务端构建、测试与数据库迁移通过

验收输出必须包含：
1. 通过项清单
2. 阻塞项清单
3. 协议与实现不一致项
4. 服务端 timeline 真相源异常项
5. 客户端页面显示不一致项
6. 缓存/轮次串用异常项
7. 最终结论：通过 / 有条件通过 / 阻塞
```

## 6. 推荐验收顺序

1. 先读 `Doc/通信协议文档.md`，核对 feedback 与 timeline 的正式语义
2. 再验服务端回包结构、正式 timeline、关闭任务清理与历史兼容

4. 再验镜头详情页、导演反馈回放区、制作查看页的一致性
5. 最后做任务切换、关闭任务、重新建任务、刷新重进回归
