- `D:\project\MovtoolsEx\Doc\项目固化文档.md`

---

## 2. 本阶段目标

修复导演审片链路中 `context` 陪审镜头、任务完成动作、任务关闭动作三者混线的问题，确保服务端先具备硬约束。

本阶段完成后必须满足：

1. 任务详情返回的镜头参与类型字段语义统一
2. `context` 镜头不会被自动推进到 `IN_DIRECTOR_REVIEW`
3. `context` 镜头不能通过任务内正式操作写入 `DIRECTOR_APPROVED`、正式反馈或正式绘制事实
4. 导演的 `审片完成` 与制片的 `关闭任务` 有独立接口和独立权限
5. 服务端对任务完成条件以 `review` 镜头为唯一统计口径

---

## 3. 阶段拆分

### 阶段 1：协议字段与返回模型收口

1. 统一任务详情镜头参与类型字段命名
2. 明确新协议的正式字段
3. 兼容旧字段一段时间，但不得让新实现继续漂移

### 阶段 2：任务动作与状态机收口

1. 导演完成任务走 `complete`
2. 制片关闭任务走 `close`
3. `complete` 与 `close` 的权限、状态前置条件和返回值必须分清

### 阶段 3：context 镜头服务端硬保护

1. 拒绝 context 镜头的正式反馈写入
2. 拒绝 context 镜头的正式通过 / 返修结论写入
3. 任务统计、完成判定、自动状态推进只统计 review 镜头

### 阶段 4：测试与文档回填

1. 补测试
2. 更新协议文档与必要注释
3. 输出风险说明

## 5. 必做事项

1. 明确 `GET /api/review-tasks/{id}/detail` 中任务镜头参与类型的正式输出字段，并保证客户端可稳定识别
2. 确认 `submit`、`start`、`complete`、`close` 只对 `review` 镜头产生正式审片状态影响
3. 新增或修正导演任务完成接口，确保导演端调用的是 `complete`，不是 producer `close`
4. 在服务端完成校验中，统一按 `review` 镜头计算：
   - 未完成镜头数
   - 通过镜头数
   - 反馈镜头数
   - canComplete 判定
5. 对 `context` 镜头新增明确业务拒绝：
   - 创建正式反馈
   - 更新正式反馈
   - 删除正式反馈
   - 写入正式绘制事实
   - 写入 `DIRECTOR_APPROVED`
   - 写入任务内返修结论
6. 若当前镜头状态写接口缺少任务上下文，需补足必要校验路径，避免仅凭 lensId 越权写 context 镜头
7. 保持历史兼容入口可用，但新增正式逻辑必须以 `review-tasks` 基线路由为准
8. 更新相关测试，覆盖 mixed review/context task 场景

---

## 6. 禁止事项

1. 不要继续把导演完成任务复用为 producer `close`
2. 不要仅靠前端控制按钮显示，服务端必须做最终拦截
3. 不要把“播放源是 layout”当成“镜头就是 context”的自动判定真相源
4. 不要为兼容旧字段而放弃新字段规范化
5. 不要引入破坏现有已完成 / 已关闭可见性规则的临时捷径

---

## 7. 验收标准

1. 服务端存在清晰的 `complete` 与 `close` 边界
2. `context` 镜头不会进入正式结论写链路
3. 任务完成判定只统计 `review` 镜头
4. mixed task 中 `context` 镜头不会阻塞完成
5. 相关服务端测试通过
6. 协议文档中的任务动作与字段定义已同步更新
