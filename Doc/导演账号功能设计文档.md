# 导演账号功能设计文档

## 1. 文档目的

本文件用于固化导演账号的产品定位、页面原型、主链路、反馈形态与协同边界，作为后续导演功能迭代、UI 设计、接口设计、验收讨论的统一基线。

本版重点解决以下问题：

1. 导演账号的左侧主导航如何收口
2. 导演如何从项目切换到审片任务
3. 审片工作台如何围绕“画面即反馈”设计
4. 导演反馈如何以“图片 + 说明”形式沉淀到镜头中
5. 制作人员如何在镜头详情中消费导演反馈

补充说明：

1. 路由、字段、状态码以 `Doc/通信协议文档.md` 为唯一协议基线
2. 页面命名、按钮命名、状态中文文案以 `Doc/平台功能与UI统一命名清单.md` 为唯一显示基线
3. 本文档重点回答“导演如何进入任务、如何处理任务、反馈如何回流”，不单独维护另一套接口契约

## 2. 角色定位

导演账号不是通用项目管理账号，而是以“审片任务”为核心的工作账号。

导演的核心目标：

1. 查看自己参与的项目
2. 在当前项目下查看待处理审片任务
3. 按制片整理好的顺序联播镜头
4. 在视频画面上直接标注与填写反馈
5. 将反馈以图片配合说明的形式保存到对应镜头

补充说明：当前导演反馈编辑窗已支持粘贴图片与选择图片后直接显示缩略图，提交后图片会作为服务端正式资产回读到镜头详情，不再只显示文件名；编辑态看的是本地草稿缩略图，回看态看的是服务端正式图片地址。

## 3. 左侧主导航设计

导演账号左侧主导航建议收口为以下 3 项：

1. 仪表盘
2. 镜头
3. 审片

补充说明：

1. 导演主流程是“仪表盘 -> 审片任务 -> 审片工作台”
2. 镜头页面作为辅助查询入口保留
3. 若当前基线文档中仍存在导演可见“项目 / 工作台 / 日志 / 设置”等菜单，应视为旧基线，后续以本文件讨论结果为准再统一回写命名清单

## 4. 产品设计原则

1. 以审片任务为核心，不以零散镜头为核心
2. 审片顺序由制片整理，导演不负责拼装镜头顺序
3. 导演反馈入口必须在视频画面上，而不是脱离画面的表单区
4. 每条反馈必须形成可视成果，最小单位为“标注图片 + 说明文字”
5. 每条反馈都必须可回挂到镜头详情，供制作人员查看
6. 原始标注数据与导出图片都必须落服务器保存

## 5. 页面信息架构

导演账号第一期建议包含 3 个核心页面：

1. 仪表盘
2. 镜头
3. 审片工作台

其中：

1. 仪表盘负责项目切换与任务入口
2. 镜头负责单镜头查询与反馈回看
3. 审片工作台负责联播、逐帧、标注、反馈保存

补充说明：

1. 镜头详情页中的 `导演反馈` 分页属于辅助回看层，不是导演正式主编辑入口
2. 镜头详情页中的 `镜头反馈` 与 `导演反馈` 是同级分页；前者由 `记录` 分页承接一级返修记录，后者由 `导演反馈` 分页承接导演内部审片反馈卡片
3. 导演反馈卡片的图片展示已统一为“编辑态缩略图 + 回看态缩略图/放大预览”两层模型

## 5.1 导演与镜头二级状态的关系

加入内部审片后，导演账号不负责管理镜头主状态，而是通过审片动作驱动镜头二级状态。

导演侧需要遵循以下规则：

1. 二级状态只服务当前镜头版本
2. 当前版本变化后，不以“自动清空状态”作为默认策略，而是按本轮反馈是否已确认处理来继续推进
3. 导演保存反馈后，镜头进入 `待处理反馈`
4. 导演确认通过后，镜头进入 `内部通过`
5. 导演不负责把镜头推进到 `待提审`

### 5.2 当前正式任务协议口径

当前导演页正式围绕两类数据工作：

1. 导演任务列表：基于正式 `ReviewTask` 列表投影
2. 导演任务详情：基于正式 `ReviewTask` + 评论 + 当前镜头反馈集合拼装

当前正式路由口径：

1. `GET /api/review-tasks`
2. `GET /api/review-tasks/{id}`
3. `GET /api/review-tasks/{id}/comments`
4. `POST /api/review-tasks/{id}/approve`
5. `POST /api/review-tasks/{id}/reject`
6. `POST /api/review-tasks/{id}/comments`
7. `POST /api/review-feedbacks`
8. `GET /api/review-feedbacks/{id}`
9. `GET /api/review-feedbacks/lens/{lensId}`
10. `PUT /api/review-feedbacks/{id}`
11. `DELETE /api/review-feedbacks/{id}`

补充说明：

1. 导演页当前显示的 `通过 / 返修` 是页面结果投影名
2. 服务端真实任务状态 `completed / closed` 在导演页分别投影为 `approved / rejected`
3. 导演当前进入任务时，任务镜头队列的来源仍以正式任务详情为准，不能由本地镜头列表反推

## 6. 仪表盘设计

### 6.1 页面目标

导演打开系统后，先确定当前工作项目，再查看该项目下待处理审片任务。

### 6.2 页面区块

1. 页面头部
2. 我参与的项目
3. 当前激活项目的审片任务
4. 最近反馈记录

### 6.3 页面头部

字段建议：

1. 页面标题：导演仪表盘
2. 当前激活项目
3. 待审任务数量
4. 最近更新时间

### 6.4 我参与的项目

每个项目卡片建议展示：

1. 项目名称
2. 项目代号
3. 项目阶段
4. 待审任务数
5. 最近提审时间
6. 激活项目按钮

交互：

1. 点击项目卡可激活项目
2. 激活后刷新任务列表
3. 激活状态需全局记忆

### 6.5 当前激活项目的审片任务

任务列表字段建议：

1. 任务名称
2. 所属项目
3. 提审人
4. 镜头数量
5. 当前状态：待审 / 审阅中 / 已完成
6. 截止时间
7. 最近更新时间
8. 开始审片按钮

筛选建议：

1. 全部
2. 待审
3. 审阅中
4. 已完成

当前阶段状态映射说明：

1. 页面筛选 `待审` 对应服务端 `pending`
2. 页面筛选 `审阅中` 对应服务端 `in-review`
3. 页面筛选 `已完成` 在导演展示层可继续拆成 `通过 / 返修`
4. 若页面继续显示 `通过 / 返修`，实现层必须分别消费服务端 `completed / closed`

交互：

1. 点击任务进入审片工作台
2. 若存在上次审到一半的任务，优先显示“继续审片”

### 6.6 最近反馈记录

建议展示导演最近保存的反馈卡片记录：

1. 镜头号
2. 所属任务
3. 对应帧号
4. 保存时间
5. 快速回到该镜头按钮

### 7.2 页面区块

1. 搜索与筛选区
2. 镜头列表区
3. 镜头详情抽屉

### 7.3 镜头列表字段

1. 镜头号
2. 所属项目 / 集
3. 当前最新版本
4. 镜头主状态
5. 内部审片二级状态
6. 最近提审版本
7. 最近导演反馈时间
8. 反馈数量
9. 当前镜头结论

### 7.4 镜头详情抽屉

建议展示：

1. 基础信息
2. 版本时间线

4. 导演反馈列表
5. 所属审片任务列表

## 8. 审片任务模型

导演端必须围绕“审片任务”工作。

### 8.1 当前任务状态口径

导演端当前需要同时理解两层状态：

1. 服务端真实状态：`pending`、`in-review`、`completed`、`closed`
2. 导演页面投影状态：`pending`、`in-review`、`approved`、`rejected`

对应关系：

1. `pending` -> `待审`
2. `in-review` -> `审阅中`
3. `completed` -> `通过`
4. `closed` -> `返修`

一个审片任务建议包含：

1. 所属项目
2. 任务名称
3. 提审人
4. 提审说明
5. 镜头清单
6. 镜头顺序
7. 每个镜头的提审版本
8. 当前任务状态

任务状态建议：

1. 待审
2. 审阅中
3. 已完成
4. 已关闭

补充说明：

1. 导演页面可以继续用“通过 / 返修”作为结果显示文案
2. 但实现层不得误以为服务端会直接返回 `approved / rejected`

## 9. 审片工作台设计

## 9.1 页面目标

审片工作台是导演的主工作空间，核心是：

1. 联播整组镜头
2. 逐帧定位问题
3. 在画面上直接标注
4. 保存为镜头反馈卡片

## 9.2 布局原则

根据本轮调整，审片工作台不再采用左右三栏为主，而采用“上任务列表、中大播放器、下反馈区”的纵向结构，以便播放器尽可能接近全屏。

页面结构建议如下：

1. 顶部：任务镜头列表区
2. 中部：播放器与画面标注区
3. 底部：反馈区

这样在页面纵向滚动时，播放器区域仍能保持最大的可视面积。

## 9.3 顶部任务镜头列表区

### 定位

这一块承担“任务导航 + 审片进度 + 快速切镜头”的作用。

### 展示内容

每个镜头项建议展示：

1. 顺序号
2. 镜头号
3. 提审版本
4. 实际播放版本
5. 反馈数量
6. 审片任务内状态：未看 / 已看 / 待处理 / 已完成
7. 当前镜头二级状态

### 交互

1. 横向列表或可折叠网格展示
2. 点击镜头项切换当前镜头
3. 支持筛选“全部 / 未反馈 / 有反馈”
4. 显示整体进度，如 `12 / 36`
5. 支持固定在页面顶部，滚动时保持可见

当前阶段补充说明：

1. 任务镜头队列是导演正式主入口的一部分，不是辅助信息
2. 导演切换镜头时，应始终保持“仍处于当前任务上下文中”的心智模型
3. 任务详情中的镜头顺序必须直接来自服务端任务详情，不得由客户端自行排序猜测
4. 镜头队列颜色状态调整为：未匹配视频显示红色，已匹配视频但尚未完成处理显示黄色，已提交反馈或已执行镜头通过后显示绿色
5. `已匹配视频` 只表示素材可播放，不等于该镜头已审结；只有提交反馈或镜头通过后，才视为当前轮次已完成处理

## 9.4 中部播放器与画面标注区

### 页面目标

播放器区域必须是整个页面的视觉中心，优先保证联播和定帧标注体验。

### 顶部信息条

建议展示：

1. 项目名
2. 任务名
3. 当前镜头号
4. 当前使用版本
5. 素材来源提示
6. 自动保存状态

补充布局要求：

1. `制作人员`、`版本`、`当前帧` 这三个描述卡片不再占据播放器上方主信息区
2. 这三项应下移到页面最底部，或放在 `测试审片` 区域上方
3. 页面上半部分优先保证视频查看、进度定位、反馈编写与绘制操作

### 视频取片规则

联播或单镜头播放时，视频匹配规则固定为：

1. 优先使用提审版本号对应的视频
2. 若提审版本号无对应视频，则使用该镜头版本号最高的视频
3. 若完全没有版本视频，则使用 Layout 视频
4. 若连 Layout 视频也不存在，则标记为无可播放素材

界面必须显式展示本次实际使用来源，例如：

1. 使用指定版本 `V012`
2. 未找到 `V012`，已自动切换 `V015`
3. 无版本视频，已使用 `Layout`
4. 无可播放素材

补充说明：审片页与镜头详情页必须保持一致的素材结果。导演进入审片任务前，应先在镜头详情页完成路径匹配；审片页只消费镜头详情已经得到的版本视频 / Layout 结果，不再单独判断“有 Layout 即可播”。

补充说明：审片联播过程中，反馈列表、绘制内容和通过状态都必须按镜头独立隔离；播放器切换到下一镜头时，页面必须立即显示当前播放镜头自己的反馈列表，不能沿用上一镜头的列表或标注残留。

补充约定：

1. 导演审片页不再自行重新匹配素材路径，必须优先复用镜头详情页已经完成的版本匹配结果
2. 审片页只做“消费镜头详情匹配结果 + 选择最终可播放 URL”，不再单独拼一套与镜头详情不一致的解析逻辑
3. 播放列表项必须携带来源类型、实际版本号、来源提示文案，队列与播放器统一消费这些派生字段
4. 只有当版本视频不存在时，才允许使用 Layout 补位；Layout 不能再被当作唯一可播判定条件
5. `submitVersionNum` 表示提审版本，`actualVersionNum` 表示任务实际播放版本，`hasPlayableMedia` 仅作为弱摘要，不得作为最终可播放依据
6. 审片页播放器快捷键固定为：`Space` 播放/暂停，`←/→` 逐帧，`Shift+←/→` 切换上一镜头/下一镜头，`Home/End` 跳转当前镜头首尾帧

补充说明：镜头详情先完成路径匹配，审片页再直接消费匹配结果；这两页的素材结果必须一致，不能一页能播一页判缺失。

### 播放控制

建议支持：

1. 播放 / 暂停
2. 上一个镜头 / 下一个镜头
3. 联播全部
4. 从当前镜头继续联播
5. 循环当前镜头
6. 逐帧前进
7. 逐帧后退
8. 指定帧跳转
9. 播放速度切换
10. 全屏

补充要求：

1. 视频播放时，播放器底部必须持续显示进度条
2. 视频暂停时，画面绘制工具必须继续可见，不得因为暂停而隐藏
3. 进度条是导演定位问题帧的基础控件，不能仅依赖快捷键或外部时间码跳转

### 画面标注工具

导演需要直接在视频定帧画面上操作，建议工具包含：

1. 选择
2. 文字
3. 箭头
4. 圆圈
5. 矩形
6. 自由画笔
7. 橡皮
8. 撤销
9. 重做
10. 颜色
11. 线宽
12. 清空当前标注

### 标注模式交互

1. 正常播放时以预览为主
2. 暂停后进入可编辑状态
3. 标注图层叠加在当前帧图像上
4. 支持缩放查看与拖拽画布
5. 当前帧未保存时切镜头需提醒
6. 暂停状态下即使视频未截图，绘制工具也必须可直接作用于当前画面
7. 当前反馈的配图来源固定为“当前视频帧 + 导演绘制图层合成图”，不再提供手动选图入口

当前阶段绘制协同原则：

1. 绘制行为本身不自动生成反馈记录
2. 绘制数据在底层仍以 `单帧事件` 表达，但正式提交后的回读语义必须统一解释为“当前反馈轮次的完整绘制时间线”
3. 同一帧允许存在多个绘制对象，这些对象共同组成该帧的绘制集合
4. 若导演需要让相同绘制效果出现在后续多帧，草稿态可以通过本轮保留链继续显示；正式提交后，系统必须把该结果固化为可稳定回放的轮次时间线
5. 绘制数据与文字反馈保持弱绑定：允许有文字反馈但无绘制，也允许仅凭有效绘制构成正式提交
6. 不允许提交无文字且无绘制的空反馈
7. 导演在镜头内产生的绘制内容应始终保存为本地数据，直到导演主动清除或正式提交
8. 系统必须明确区分 `本轮新绘制保留链` 与 `历史已保存正式绘制时间线` 两类来源，二者不能混为同一条继承链
9. `本轮新绘制保留链` 只服务导演当前草稿编辑体验，可在后续未定稿帧上继续显示，直到被新的本轮绘制或清屏打断
10. `历史已保存正式绘制时间线` 是服务端正式事实，任意查看页都必须按同一规则逐帧回放
11. 提交成功后，导演工作台、镜头详情页、制作查看页都必须与正式时间线回放结果一致，不允许再因为“当前帧没有 feedback 条目”而丢失绘制
12. `清屏` 是当前帧动作，会从当前帧开始打断前序显示链
13. 清屏后直到再次出现新的 DRAWN 事件前，后续帧均保持无绘制状态
14. 若导演回到历史帧修改本轮绘制，则该轮正式提交前的保留显示结果应重新计算；正式提交后则以新的正式 timeline 为准
15. 点击反馈列表定位到历史反馈帧时，允许显示该帧正式结果，但该查看动作本身不得生成新的保留链，也不得影响后续帧显示

## 9.5 底部反馈区

### 整体布局

底部反馈区拆为上下两块：

1. 上方：当前反馈卡片编辑区
2. 下方：草稿与已提交反馈列表

这样可以保证反馈列表不挤占视频预览与绘制窗口，同时让“正在编辑的反馈”和“当前镜头已经确定的反馈列表”保持连续阅读关系。

补充说明：底部反馈区也必须按当前播放镜头实时切换，草稿、已提交反馈、绘制状态和 `镜头通过` 按钮都只对当前镜头生效。

补充说明：实现层必须维护一个“当前激活镜头 ID”，反馈列表和通过按钮都只认这个 ID；视频虽然可能继续预加载下一镜头，但当前反馈列表不能跟着视频预加载结果走，必须等真正切到该镜头后再切换展示。

### 上方：当前反馈卡片编辑区

当前反馈卡片不是传统表单，而是一条即将保存到镜头中的图文反馈。

建议包含：

1. 当前帧预览缩略图
2. 当前标注状态：未保存 / 已保存 / 已修改
3. 反馈说明输入框
4. 保存按钮组

字段建议：

1. 反馈说明
2. 对应时间点 / 当前帧信息
3. 当前绘制状态

字段说明：

1. `标注说明` 文案统一改为 `反馈说明`
2. `反馈说明` 用于描述当前帧或后续几帧的修改意见
3. 该区域不再编辑 `镜头结论`，因为进入反馈流程本身就代表该镜头需要修改
4. 该区域不再提供图片选择能力，反馈配图固定取当前视频帧并叠加导演绘制内容

按钮建议：

1. 保存为草稿
2. 提交反馈
3. 清除当前未保存反馈
4. 镜头通过

按钮规则：

1. 点击 `保存为草稿` 后，应在右侧反馈列表生成一条草稿记录
2. 点击 `提交反馈` 时，提交当前镜头下全部待提交反馈，并同步上传相关绘制数据
3. `镜头通过` 只检查当前镜头状态，只有当当前镜头没有已提交反馈、没有待提交草稿、没有待提交绘制修改时才允许点击
4. 播放器切换到下一镜头后，上一镜头的反馈列表和绘制内容不得继续显示在当前镜头上
5. 原 `取消镜头返修` 按钮移除，不再保留
6. 若当前镜头存在绘制内容但反馈列表为空，只要该绘制本身构成有效修改，点击 `提交反馈` 时仍允许提交
7. 只有当当前镜头既没有有效文字反馈，也没有有效绘制修改时，才不允许正式提交
8. `镜头通过` 按钮应与 `提交反馈` 放在同一操作区，便于统一管理当前镜头处理动作

### 下方：草稿与已提交反馈列表

每条反馈卡片建议展示：

1. 带标注图片缩略图
2. 帧号 / 时间码
3. 说明摘要
4. 版本号
5. 创建时间
6. 创建人
7. 当前状态：草稿 / 已提交

操作建议：

1. 查看大图
2. 回跳到对应帧
3. 编辑
4. 删除

### 反馈列表交互原则

1. 右侧区域是“当前镜头反馈列表”，统一承载本地草稿与已提交反馈，不再拆成两套心智模型
2. 同一镜头允许存在多条反馈卡片，但同一帧在同一镜头下只允许保留一条反馈项；再次编辑同一帧时应更新该项，而不是新增重复项
3. 保存草稿后即时刷新右侧反馈列表；若当前帧原本已有已提交反馈，则保存草稿后应以“待更新该条反馈”的草稿态覆盖显示
4. 点击 `编辑` 时，播放器必须自动定位到该反馈对应帧，并把该条反馈载入左侧编辑区
5. 已提交反馈也必须提供 `编辑` 与 `删除` 按钮；编辑后形成待提交修改，删除后形成待提交删除
6. 只要当前镜头存在新增、修改、删除中的任一待提交变化，或存在仅绘制修改，`提交反馈` 按钮就应可点击
7. 删除反馈后，若当前镜头反馈数量归零且不存在任何待提交绘制修改，则 `镜头通过` 按钮恢复可点击
8. 制作人员看到的正式反馈结果以提交后的反馈为准，草稿仅属于导演当前编辑态
9. 反馈列表是否为空不再是唯一判断条件；若当前镜头存在有效绘制修改但尚未形成文字草稿，仍允许正式 `提交反馈`
10. `当前镜头反馈列表` 的正式数据源必须按镜头实体 id 严格隔离；若同一审片任务包含多个镜头，请求当前镜头反馈时不得混入同任务其他镜头反馈
11. 客户端即使收到异常混合数据，也必须再按 `lensId`（必要时再结合 `taskShotId`）执行一次镜头级过滤后再渲染

## 10. 审片工作台主链路

导演在审片工作台的标准操作链路建议固化如下：

1. 从仪表盘进入一个审片任务
2. 在顶部任务镜头列表中确认当前镜头顺序
3. 通过联播查看整组镜头
4. 在发现问题的镜头处暂停
5. 逐帧定位到具体问题帧
6. 在视频画面上直接标注
7. 在底部左侧填写该条反馈说明
8. 若当前帧已有已提交反馈，播放或定位到该帧时，左侧编辑区自动回显该反馈内容
9. 导演可直接在回显内容上继续编辑，并保存为草稿；该草稿代表“对同帧既有反馈的待提交更新”
10. 若切到其他没有反馈的帧并保存草稿，则在当前镜头反馈列表中新增该帧的独立反馈项
11. 草稿记录出现在底部右侧反馈列表中，并可继续编辑或删除
12. 当需要正式提交时，点击 `提交反馈`
13. 客户端按当前镜头维度提交新增、更新、删除中的全部待提交反馈变化，并同步上传相关绘制数据
14. 若当前镜头无任何反馈且无任何待提交绘制修改，导演可直接执行 `镜头通过`
15. 点击 `镜头通过` 时，只上传镜头通过状态到服务器
16. 完成当前镜头处理后，继续下一处问题或切换下一镜头
17. 完成后将任务状态推进为已完成

当前阶段补充说明：

1. 导演“通过”动作当前作用在任务结果与镜头二级状态联动上
2. 导演“返修”动作在协议层仍对应 `reject`
3. 当前页面允许任务结果投影显示为 `通过 / 返修`，但协议动作名不得随意改成另一套枚举

## 11. 导演反馈对象定义

导演反馈建议定义为镜头下的独立记录对象。

一条反馈至少包含：

1. 项目 ID
2. 镜头 ID
3. 审片任务 ID
4. 版本 ID
5. 帧号
6. 时间码
7. 原始帧截图
8. 带标注导出图
9. 标注原始数据 JSON
10. 反馈说明
11. 当前状态：草稿 / 已提交
12. 创建人
13. 创建时间
14. 最近编辑时间

当前阶段补充说明：

1. 当前服务端稳定保存的反馈对象以 `reviewTaskId + lensId + versionNum + frameNumber/timecode` 为核心上下文；同一镜头同一帧默认只维护一条有效反馈事实
2. 导演反馈卡片与普通任务评论是两类对象：评论负责任务上下文沟通，反馈卡片负责镜头级图文反馈
3. 后续若要增强标签、附件、版本对照，必须继续在协议文档中扩展，而不是只在页面层追加本地字段
4. 当前反馈图片不再由用户手工选择，而是由客户端基于当前帧图像与绘制图层自动生成
5. `镜头结论` 不属于单条帧反馈字段，镜头结果由 `提交反馈` 或 `镜头通过` 动作体现
6. 绘制数据是镜头级时间轴上的本地编辑数据，提交时按镜头维度与反馈列表一并上传；若只有绘制修改也可独立形成正式提交
7. 已提交反馈在导演工作台中仍可再次编辑或删除；再次编辑后先形成本地草稿，提交后更新原反馈事实，而不是强制新增一条同帧重复记录
8. 反馈回读接口必须保持 `content`、`frameNumber`、`timecode`、`feedbackRoundId`、`taskShotId` 的语义一致；若历史数据出现 `frameNumber` 为空但 `timecode` 有值，客户端应以 `timecode` 作为兼容降级依据
9. `GET /api/review-feedbacks/lens/{lensId}` 的正式语义是“只回读该 `lensId` 对应镜头的反馈事实”；不得因为该反馈所属任务中包含当前镜头，就把其他镜头反馈一并返回

## 12. 存储与服务器要求

导演反馈相关内容必须全部保存到服务器。

### 12.1 服务器保存内容

1. 反馈主记录
2. 标注原始数据
3. 原始帧截图
4. 带标注导出图
5. 缩略图

### 12.2 保存原则

1. 不只保存文字说明
2. 不只保存最终截图
3. 需要同时保存可阅读成果和可编辑原始数据
4. `保存为草稿` 仅生成当前镜头下的本地草稿记录，不作为正式服务器入库时机
5. 正式上传服务器的触发动作只有 `提交反馈` 与 `镜头通过`
6. 尚未形成文字反馈的绘制内容，也必须作为本地数据保留，不能因为切镜头、刷新页面或关闭页面而丢失

### 12.2.0 本地绘制保存约定

1. 导演在暂停态产生的单帧绘制内容，必须立即纳入当前镜头的本地绘制数据
2. 切换镜头时，当前镜头的本地绘制数据必须保留
3. 刷新页面后，当前任务内镜头的本地绘制数据必须能够恢复
4. 关闭页面后再次进入，当前任务内镜头的本地绘制数据仍应保留，直到导演主动清空或完成正式提交
5. 后续绘制工具区应提供 `清屏` 按钮，由导演手动清除当前帧或当前镜头下的绘制内容
6. 本地绘制保存的目标是保证导演的视觉反馈不会在正式生成文字反馈前意外丢失

### 12.2.0.1 绘制继承链解释规则

1. 底层数据仍然只保存单帧绘制记录，不直接保存区间绘制对象
2. 显示层必须先判断当前帧是否存在 `历史已保存绘制结果`
3. 若当前帧存在 `历史已保存绘制结果`，则当前帧只显示该结果，且该结果不会自动带到下一帧
4. 若当前帧不存在 `历史已保存绘制结果`，则系统再判断当前帧之前是否存在仍然有效的 `本轮新绘制保留链`
5. `本轮新绘制保留链` 只来自当前编辑轮次中导演新画出的内容；通过反馈列表回看的历史结果、服务端已保存结果都不能成为新的保留链起点
6. 最近有效控制点若为 `本轮绘制帧`，则当前帧显示该保留链中的绘制集合
7. 最近有效控制点若为 `清屏帧`，则当前帧不显示任何绘制
8. `清屏帧` 是保留链中断点，不是普通绘制对象
9. 任何一个存在自身已保存绘制结果的帧，也视为前序保留链的天然中断点；其后一帧若没有新的本轮绘制，默认显示为空
10. 提交反馈时，客户端必须把当前镜头本轮最终绘制结果固化为可稳定回放的正式时间线；例如 `2-4 = a`、`5-6 = b`、`7 = x`、`8+ = 空`，提交后所有页面都应按同一正式时间线复现
11. 修改历史本轮绘制后，应重新计算其后续直到下一本轮绘制帧、清屏帧或已有保存结果帧之前的保留显示结果

### 12.2.1 上传时机约定

1. 点击 `提交反馈` 时，客户端必须上传当前镜头下本次待提交反馈与相关标注产物
2. 点击 `镜头通过` 时，客户端必须上传当前镜头的通过状态
3. `提交反馈` 应按当前镜头维度一次性提交全部待提交反馈，而不是只提交单条正在编辑的内容
4. 若当前镜头存在反馈记录，则禁止执行 `镜头通过`
5. 删除全部反馈后，当前镜头才重新具备 `镜头通过` 条件
6. 若当前镜头仅存在本地绘制数据而不存在任何文字反馈，只要该绘制构成有效修改，就允许提交纯绘制结果；只有文字与绘制都为空时才禁止提交
7. 绘制数据的正式提交边界是“当前镜头当前轮次的全部最终正式时间线结果”，而不是仅上传本次编辑增量，也不是只上传少量控制点
8. 若某段本轮绘制需要在多个连续帧上成立，提交前客户端必须将其展开并固化到正式时间线中；未被固化的后续空白帧不得因为前序查看或历史保存结果而被误写入同样内容
9. 当前轮反馈提交成功后，该镜头本地反馈内容与本地绘制内容应清空，进入下一轮空白编辑态

### 12.2.1.1 当前实现修正方案

1. 前端显示层把 `本轮新绘制保留链` 与 `历史已保存绘制结果` 分开建模，不再共用同一套 `drawingFrames` 回放语义
2. 点击反馈列表时只做 `定位到该帧 + 显示该帧正式结果`，不得把该帧历史绘制注入当前保留链
3. 当前帧显示优先级统一为：`当前帧本地草稿` > `当前帧历史已保存结果` > `前序仍有效的本轮新绘制保留链` > `空白`
4. 一旦命中 `当前帧历史已保存结果`，该帧之后默认不继续继承该结果；只有导演在该帧再次新绘制后，才会生成新的本轮保留链
5. `清屏` 既要清空当前帧显示，也要在本地状态中写入明确的链路中断事实，阻止旧保留链继续影响后续帧
6. 提交前需新增一层“逐帧展开器”，把本轮保留链按最终帧结果展开为单帧提交事实，再写入 `drawingFrames`
7. `reviewDrawingResolver` 需从“事件累计回放”改为“区分历史结果与本轮保留链的逐帧结算器”
8. `ReviewPage` 需补一套面向编辑态的帧结果解析函数，确保跳帧、播放、点击反馈列表、提交后刷新都走同一套逐帧显示规则
9. 服务端与客户端的回放口径都要按“提交后看到的就是逐帧最终结果”验收，不能再允许 `7 帧有 x，8 帧被自动回放出 x` 这类额外继承

### 12.2.2 镜头详情遮罩回放约定

1. 镜头详情页中的导演反馈遮罩只显示当前已提交轮次的正式绘制内容
2. 遮罩回放不显示导演工作台中的本地未提交绘制
3. 当导演完成一轮反馈并提交成功后，镜头详情页消费的是该轮正式提交产物
4. 导演工作台在提交成功后应清空本地反馈内容，避免新一轮反馈误叠加上一轮结果

### 12.3 文件组织建议

可按项目与镜头维度组织：

1. `project/{projectId}/shots/{shotId}/feedback/{feedbackId}/frame.png`
2. `project/{projectId}/shots/{shotId}/feedback/{feedbackId}/annotated.png`
3. `project/{projectId}/shots/{shotId}/feedback/{feedbackId}/annotation.json`

### 12.4 图片展示与回读约定

1. 导演提交反馈时，客户端必须先把图片预览为缩略图，不得仅显示文件名
2. 提交后的反馈图片必须以服务端正式可访问地址回读
3. 镜头详情页中的 `导演反馈` 分页只负责回看，不作为导演主编辑入口
4. 旧反馈回读与新反馈提交必须共用同一套图片地址解析规则，不能把服务端图片路径误转成本地 `file://`
5. 反馈卡片无论来自草稿还是回读，都必须保持缩略图可见，并支持点击放大查看

## 13. 与镜头详情页的协同关系

导演保存后的反馈必须自动回挂到镜头详情。

制作人员在镜头详情中应能看到：

1. 标注图片缩略图
2. 说明文字
3. 帧号 / 时间码
4. 对应版本号
5. 提交时间
6. 导演姓名
7. 来源审片任务

## 14. 权限边界

导演账号建议具备以下能力：

1. 查看参与项目
2. 激活项目
3. 查看审片任务
4. 联播镜头
5. 逐帧审片
6. 直接在画面上标注
7. 保存反馈卡片
8. 将当前镜头版本确认内部通过
9. 更新任务状态

导演账号不负责：

1. 创建项目
2. 管理项目成员
3. 拼装审片任务镜头顺序
4. 修改镜头主数据
