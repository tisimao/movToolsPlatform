## 2. 本次子任务目标

本次只聚焦“镜头视频连播功能优化”，不扩散到整批十九批的其他主题。

完成后，客户端必须具备：

1. 导演审片工作台按任务镜头顺序连续播放，不再以“手工逐镜头点播”作为主体验
2. 当前镜头播放结束后，播放器可自动无感切到下一个镜头并继续播放
3. 在当前镜头临近结束前，系统提前准备下一个镜头，尽量降低黑帧、白屏、等待和明显停顿
4. 镜头队列高亮跟随真实播放中的镜头，而不是只跟随点击态
5. 手动点击镜头队列切换时，仍能复用统一切换链路，不破坏自动连播能力
6. 连播优化后，暂停标注、本地草稿、反馈卡片、镜头通过/返修按钮语义不回退

本次不做：

1. 不重做导演工作台整体 UI 信息架构
2. 不重做反馈卡片业务模型
3. 不把多个镜头真正合成为一个物理视频文件
4. 不改变“提交反馈”和“任务结论”解耦的既有规则

## 3. 现状结论（必须按此理解）

当前客户端并不是完全没有连播基础，但还没有达到“无缝连播”标准。

### 3.1 当前已具备的基础

1. `ReviewPage.tsx` 已支持多镜头任务详情加载、镜头队列渲染、当前镜头索引状态
2. 当前视频播放结束时，已经会尝试自动切到下一个镜头
3. 页面中已经存在“预加载下一个镜头”的初步状态和 `nextVideoRef` 预留
4. 队列按钮激活态当前由 `currentShotIndex` 驱动，已经不是纯静态文案
5. 暂停标注、草稿卡片、本地图片、提交反馈等链路已接到当前 `ReviewPage`
6. 本次预加载不能只预加载 Layout 视频，必须基于最终解析后的真实播放 URL
7. 若审片页已先完成镜头详情预热，后续切镜头切换只允许复用已解析结果，不允许再以 Layout 视频作为唯一预加载目标

### 3.2 当前关键问题

1. 现有实现本质上仍是“单 video 元素 + 切镜头时重挂载 src”，不是双缓冲或预切换模型
2. `switchToShot(...)` 切换时会先停播、重置状态，再异步拉数据，切换时序天然容易产生停顿
3. `nextVideoRef` / `nextVideoSrc` 目前没有真正承担前后台播放器切换职责，预加载没有闭环
4. 当前镜头结束后虽然会切下一个镜头，但很难满足“像一条完整视频一样连着看”的体验要求
5. 当前切镜头相关的数据刷新分散在多个 effect 和事件里，链路不够收口，后续继续加功能风险高
6. 当前暂停标注、草稿编辑与播放器状态耦合较紧，若直接粗暴改播放器，容易引入回退
7. 当前播放源解析没有来源类型和来源文案，导致版本视频、Layout 补位与无素材混淆
8. 当前播放器没有统一快捷键说明区，新增键盘操作后需要在页面顶部固化提示，避免分散在播放器底部影响观看

## 4. 已固化业务边界（必须遵守）

1. 任务详情中的镜头顺序是连播顺序真相源，不允许本地自行重排后播放
2. 连播主体验必须成立，导演不能被迫逐镜头手工点击播放
3. 队列高亮必须跟随真实播放中的镜头推进
4. 暂停态才允许标注；播放态必须锁定画布
5. 本地草稿与服务端已提交反馈必须继续严格区分
6. “提交反馈”不自动触发通过、返修、关闭
7. 不允许为了连播优化破坏当前镜头反馈、标注、草稿、图片导入现有能力
8. 如果当前镜头或下一镜头没有可播放地址，必须优雅降级，而不是让整个任务卡死

## 6. 推荐实施拆解

建议把本次任务拆成以下 5 个子阶段：

### 阶段 1：状态收口

1. 梳理 `ReviewPage.tsx` 当前状态，把任务数据、播放器数据、镜头数据、反馈编辑数据分层
2. 明确哪个状态才代表“当前真实播放镜头”
3. 删除或改造无实际作用的半成品预加载状态

### 阶段 2：播放列表模型化

1. 基于 `taskDetail.shots` 派生稳定播放列表
2. 为每个镜头补齐可播放地址、可播放状态、预加载状态
3. 尽量避免切镜头时每次重新做整页级重置

### 阶段 3：预加载与接力切换

1. 在当前镜头进入尾段时开始准备下一镜头
2. 将 ended 后切换收敛为“接管已准备好的下一个镜头”
3. 统一手动切换与自动切换逻辑

### 阶段 4：编辑态对齐

1. 校正暂停标注与播放态边界
2. 校正草稿编辑与切镜头行为
3. 确认当前帧、时间码、镜头号在切换后不会串镜头

### 阶段 5：样式与验证

1. 补队列激活态、加载态、异常态提示
2. 验证多镜头、缺视频镜头、手动切换、自动切换、暂停标注、草稿提交等场景

## 7. 推荐的技术方案口径

执行方在实现时，优先按以下思路落地：

1. 不再把“当前 shot index”直接等同于所有业务状态真相源，而是单独维护“前台正在播放的播放项”
2. 引入“前台播放器 / 后台预加载播放器”双实例，或等价的双缓冲切换机制
3. 预加载触发时机建议使用：
   - 当前播放时长进入尾段阈值，例如剩余 1~3 秒
   - 或元数据加载完成后立即后台准备下一镜头
4. 切镜头时优先复用已准备好的 `videoSrc`，不要 ended 后再串行拉一整套依赖
5. 当前镜头详情和反馈列表可以异步补拉，但不能阻塞播放器尽快切换和恢复播放
6. 标注态仍以当前前台镜头为准；只要前台镜头变化，就必须清理上一镜头画布上下文
7. 预加载的 `videoSrc` 必须是“版本优先、Layout 兜底”后的最终结果，不可只按 `layoutVideoPreviewUrl` 预热

## 8. 风险提醒

1. 如果执行方在 `ReviewPage.tsx` 中继续堆叠 effect，而不先整理状态边界，后续极易出现连播、草稿、标注互相打架
2. 如果只做 ended 后 setState 切下一个镜头，而不做预准备，体验提升会非常有限
3. 如果把数据刷新链路绑死在切镜头主流程里，接口稍慢就仍会拖慢播放器切换
4. 如果双播放器接力处理不当，可能出现：
   - 队列高亮提前跳
   - 当前帧信息错位
   - 标注叠到错误镜头
   - 手动切换后自动连播断掉
