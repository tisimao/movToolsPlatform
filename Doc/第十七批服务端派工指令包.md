## 2. 本批次目标

这一批完成后，服务端必须具备：

1. 镜头返回二级状态字段与 `submissionAllowed` 判定结果
2. 镜头二级状态专用写接口与角色校验
3. 当前版本升级后二级状态自动重置为 `NOT_IN_REVIEW`
4. 正式提交时若当前二级状态不是 `DIRECTOR_APPROVED`，必须拒绝
5. 审片任务对象、任务镜头对象、导演反馈卡片对象正式落库
6. 导演反馈图片与标注数据可保存并回读
7. 制片 / 导演 / 制作人员三角色对二级状态写动作的权限边界稳定可测

这一批不做：

1. 不把客户端本地文件扫描迁回服务端
2. 不把视频素材本体托管到服务端
3. 不把镜头主状态和镜头二级状态合并为一个字段
4. 不允许仅靠客户端禁用按钮而不做服务端最终校验

---

## 3. 已固化业务边界（必须遵守）

1. 镜头主状态仍是正式业务状态，镜头二级状态是内部导演审片状态
2. 镜头二级状态只绑定当前镜头版本，版本升级后二级状态自动重置
3. 当前二级状态不是 `DIRECTOR_APPROVED` 时，正式提交必须被服务端拒绝
4. 制作人员只允许执行 `PENDING_FEEDBACK_FIX -> FIX_UPDATED`
5. 导演反馈保存后应刷新镜头反馈统计，并驱动镜头进入 `PENDING_FEEDBACK_FIX`
6. 导演执行内部通过动作应通过专用二级状态接口完成，不与镜头主状态接口混用
7. 审片任务由制片组织，导演反馈卡片归属镜头、任务、版本、帧

---

## 4. 问题定性

当前服务端需要解决的问题必须按以下结论理解：

1. 现有镜头模型只有主状态，不足以承载内部导演审片链路
2. 现有审片协议若只停留在普通 review/comment，不足以承载任务化审片与图文反馈卡片
3. 仅靠客户端控制“内部未通过不可提交”不可靠，必须由服务端在正式提交链路做最终拦截
4. 当前版本变化后二级状态若不自动重置，会导致错误继承旧版本通过结果
5. 反馈图片、标注图、标注 JSON 若没有统一存储契约，后续镜头详情无法稳定回看

## 6. 推荐实施顺序

1. 先补数据库迁移与实体字段
2. 再补镜头 DTO 与二级状态只读返回
3. 再补二级状态写接口与权限校验
4. 再补版本升级后的自动重置逻辑
5. 再补正式提交拦截
6. 最后补审片任务、反馈卡片与附件关联接口
