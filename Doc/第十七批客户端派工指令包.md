## 2. 本批次目标

这一批完成后，客户端必须具备：

1. 镜头列表与镜头详情稳定展示镜头主状态 + 镜头二级状态
2. 制片可在镜头模块中处理镜头二级状态：待提审、进入审片、重新提审
3. 导演拥有以审片任务为核心的审片入口与审片工作台
4. 导演可在视频画面上直接标注并保存“图片 + 说明”的反馈卡片
5. 制作人员可在镜头列表或详情中执行“标记已按反馈修改”
6. 当前镜头二级状态未进入 `内部通过` 时，正式提交入口必须被客户端收口
7. 当前镜头版本变化后二级状态展示必须自动刷新，不允许继续显示旧版本的内部通过结果

这一批不做：

1. 不改客户端本地文件扫描职责边界
2. 不把导图片标注结果只存本地而不走服务端
3. 不把镜头主状态和镜头二级状态合并成一个字段展示
4. 不允许在页面层私自判断提交权限而绕过服务端只读字段

---

## 3. 已固化业务边界（必须遵守）

1. 镜头主状态面向客户正式流程，镜头二级状态面向内部导演审片流程
2. 镜头二级状态只描述当前镜头版本；当前镜头处于 `待处理反馈` 时，新版本产出本身不自动清空状态，需待制作人员确认处理完成后再推进到 `已按反馈修改`
3. 当前镜头二级状态不是 `内部通过` 时，正式提交按钮必须不可用
4. 制片负责二级状态的组织性推进，导演负责反馈与内部通过，制作人员只负责 `已按反馈修改`
5. 导演反馈的最小产物是“标注图片 + 说明文字”的反馈卡片
6. 审片工作台必须采用“上任务镜头列表、中播放器、下反馈区”的纵向结构
7. 播放素材选择必须遵循：提审版本视频 -> 最高版本视频 -> Layout 视频

---

## 4. 问题定性

当前客户端需要解决的问题必须按以下结论理解：

1. 现有镜头页只有镜头主状态，缺少内部导演审片的二级状态展示与动作入口
2. 现有制作人员只读约束需要细化：制作人员仍不可改镜头主状态，但允许执行 `待处理反馈 -> 已按反馈修改`
3. 现有审片能力若仍停留在普通评论模式，无法满足导演“定帧标注并保存图文反馈卡片”的核心需求
4. 客户端必须用服务端返回的 `submissionAllowed` 收口正式提交入口，不能自己猜状态
5. 当前版本变化后若页面继续保留旧二级状态，会直接造成错误提交与错误展示
