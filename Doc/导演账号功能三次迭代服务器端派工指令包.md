# 导演账号功能三次迭代服务器端派工指令包

## 1. 文档用途

本文件用于把“导演账号功能三次迭代”的服务器端任务直接派发给其他 AI 或开发执行。

本轮目标不是继续修补单个报错，而是把以下三类规则一起正式落到服务端：

1. `正式审片镜头 / 上下文陪审镜头` 两类任务镜头语义
2. 制片加镜头入任务时，客户端默认参与类型必须遵守素材匹配规则；服务端只消费最终提交值并保留兼容兜底
3. 制片关闭任务时物理清空该任务下全部内部反馈，并把正式审片镜头回退到 `READY_FOR_REVIEW`

## 4. 问题背景

当前服务端在审片任务链路上同时存在三类缺口：

1. 任务提交后，系统会把任务内处于 `READY_FOR_REVIEW` / `FIX_UPDATED` 的镜头统一推进到 `IN_DIRECTOR_REVIEW`
2. 但实际业务中还存在一类镜头：

   - 镜头尚未完成正式版本
   - 当前只有 `layout` 或其他参考视频
   - 需要进入任务连播，用于导演判断前后镜头关系
   - 导演不会对该镜头提交反馈，也不会点击通过
3. 同一镜头在旧任务留下的反馈，在新任务重新提审时仍会被按镜头历史回读出来
4. 制片在装配任务时新增镜头，客户端默认参与类型逻辑必须遵守素材匹配规则，不能无条件落到 `review` 或 `context`

如果不一起收口，会产生以下问题：

1. 任务提交时被误推进到 `IN_DIRECTOR_REVIEW`
2. 导演工作台出现不应存在的“通过 / 返修 / 提交反馈”操作
3. 任务统计与完成判定被上下文镜头污染
4. 新任务仍显示已关闭旧任务的反馈残留
5. 制片新增镜头默认参与类型若不按素材匹配规则计算，会导致正式镜头和补位镜头被错误归类

## 5. 本阶段目标

在服务端正式引入“任务镜头参与类型”语义，确保：

1. `正式审片镜头` 参与内部审片状态机与导演结论
2. `上下文陪审镜头` 只参与任务播放和顺序组织，不参与正式审片状态推进
3. 任务统计、任务完成判定、导演反馈写入都只面向正式审片镜头
4. 制片新增任务镜头时，客户端默认参与类型稳定遵守素材匹配规则
5. 制片关闭任务后，旧任务反馈被物理清空，正式审片镜头回退到 `READY_FOR_REVIEW`

## 6. 必做事项

1. 为任务镜头模型增加参与类型字段，建议：
   - `ReviewParticipationMode`
   - 取值：`review` / `context`
2. 为现有数据提供兼容策略：
   - 存量任务镜头默认视为 `review`
3. 调整任务草稿创建、任务加镜头、任务详情回读，使每个任务镜头都带上参与类型
4. 调整任务提交 / 开始逻辑：
   - 只对 `review` 镜头执行 `READY_FOR_REVIEW` / `FIX_UPDATED` -> `IN_DIRECTOR_REVIEW`
   - `context` 镜头禁止自动推进镜头二级状态
5. 调整导演反馈写入、绘制写入、镜头通过 / 返修相关校验：
   - 若目标任务镜头为 `context`，必须拒绝正式写入
6. 调整任务级统计与完成判定：
   - 只统计 `review` 镜头
   - `context` 镜头不阻塞任务完成
7. 核对镜头最近任务回挂、任务详情 summary、反馈计数等投影，避免把 `context` 镜头误投影成“待导演处理”
8. 服务端仅保留参与类型兼容兜底：若旧客户端未传值，则按 `review` 入库；不在服务端实现素材匹配默认判定
9. 调整 `close` 链路：由制片关闭任务时，物理删除该任务下全部导演反馈与绘制事实
10. `close` 后把任务内 `review` 镜头统一回退到 `READY_FOR_REVIEW`
11. 清理关闭任务后的镜头聚合字段与任务回挂，避免新任务继续看到旧反馈残留

## 8. 推荐修改方向

### 8.1 数据模型与迁移

建议在任务镜头实体上增加正式字段，例如：

1. `ReviewParticipationMode: string`
2. 默认值：`review`

要求：

1. 新建数据库迁移
2. 老数据回填为 `review`
3. 域模型、EF 配置、DTO、响应投影统一带出该字段

### 8.2 任务提交 / 开始的自动状态推进收口

当前 `SetTaskStatusAsync` 中与镜头二级状态自动推进有关的逻辑必须改为：

1. 仅处理 `review` 镜头
2. 仅当当前二级状态为 `READY_FOR_REVIEW` 或 `FIX_UPDATED` 时推进到 `IN_DIRECTOR_REVIEW`
3. `context` 镜头只更新任务内播放信息、最近任务回挂等必要字段，不改 `InternalReviewStatusCode`

### 8.3 导演写动作校验收口

以下写动作都必须基于任务镜头参与类型做拦截：

1. 创建反馈卡片
2. 更新反馈卡片
3. 删除反馈卡片
4. 提交绘制遮罩
5. 镜头通过
6. 镜头返修 / 待处理反馈

若命中 `context` 镜头，应返回稳定业务错误，例如：

1. `review_context_shot_read_only`
2. 或等价可读错误码

### 8.4 任务统计与完成判定

需统一收口以下统计口径：

1. `approvedShotCount`
2. `feedbackCount`
3. `pending` / `completed` 类任务摘要
4. 任务完成前置条件

要求：

1. `context` 镜头不纳入“待导演处理数”
2. `context` 镜头不纳入“内部通过数”分母
3. 任务是否可完成，只看 `review` 镜头是否全部达到本轮结束条件

### 8.5 关闭任务即清空反馈

当前项目已明确采用以下业务口径：

1. 内部评审旧反馈不保留历史追溯价值
2. 制片关闭任务时，视为本轮内部评审正式清场

要求：

1. `CloseTaskAsync` 或等价链路中删除该任务下全部 `review_comments`
2. 与反馈一起持久化的绘制事实随反馈一并清空
3. 关闭后按当前任务维度回读反馈必须为空
4. 同镜头重新进入新任务时，不得再读到已关闭任务的旧反馈
5. 若镜头 `LatestReviewTaskId` 指向当前关闭任务，需做清理或重置
6. 任务内 `review` 镜头二级状态统一回退到 `READY_FOR_REVIEW`

### 8.6 新增镜头默认参与类型

要求统一收口：

1. 草稿创建时新增镜头，官方客户端必须按素材匹配规则显式传入参与类型
2. 规则固定为：当前版本视频已匹配 -> `review`；若当前镜头需要依赖 `layout` 视频补位连播 -> `context`
3. 服务端不负责根据素材状态自动推导默认参与类型，只消费客户端最终提交值
4. 任务详情追加镜头时，若请求未传 `reviewParticipationMode`，服务端仍按 `review` 兼容兜底
5. 不允许因为默认值缺失而把普通镜头误落到 `context`
6. 不允许把“仅有 layout 补位视频”的镜头默认落成 `review`

## 9. 禁止事项

1. 不要只在前端隐藏按钮，服务端必须成为正式语义真相源
2. 不要把 `context` 镜头简单塞进现有 `IN_DIRECTOR_REVIEW` 再靠页面文案规避
3. 不要为了兼容旧逻辑，让 `context` 镜头继续生成正式反馈或通过结论
4. 不要修改为“整个任务只要含一个 context 镜头就禁用自动推进”；自动推进必须精确到任务镜头级别
5. 不要把“关闭任务清空反馈”错误实现成“完成任务自动清空反馈”
6. 不要让任务新增镜头在默认情况下落成 `context`

## 10. 自测要求

至少补以下服务端测试：

1. 创建一个混合任务：
   - 镜头 A：`review`
   - 镜头 B：`context`
2. A 处于 `READY_FOR_REVIEW` 或 `FIX_UPDATED`，B 处于 `NOT_IN_REVIEW`
3. 提交任务后：
   - A 自动进入 `IN_DIRECTOR_REVIEW`
   - B 仍保持原二级状态
4. 导演尝试对 B 提交反馈，应被拒绝
5. 导演尝试对 B 点击通过，应被拒绝
6. 任务统计只统计 A，不统计 B
7. 仅当 A 达到完成条件时，任务即可完成，B 不构成阻塞
8. 关闭任务后，该任务下反馈回读结果为空
9. 关闭任务后，A 的镜头二级状态回到 `READY_FOR_REVIEW`
10. 用同一镜头重新建立新任务时，导演进入后默认看不到旧任务反馈残留
11. 客户端按素材匹配规则默认提交后，任务详情回读应稳定反映最终保存的参与类型

建议再补一组：

1. 先以 `context` 加入任务
2. 后续把该镜头切换为 `review`
3. 再次进入正式提交流程时，状态推进恢复正常

## 11. 交付标准

1. 任务镜头正式具备 `review / context` 参与类型语义
2. `context` 镜头不会被自动推进到 `IN_DIRECTOR_REVIEW`
3. `context` 镜头不能写入正式反馈、绘制或通过结论
4. 任务统计与完成判定不再受 `context` 镜头污染
5. 制片关闭任务后，该任务下反馈被物理清空
6. 制片关闭任务后，任务内 `review` 镜头回退到 `READY_FOR_REVIEW`
7. 服务端不擅自改写客户端按素材匹配规则提交的默认参与类型
8. 相关服务端测试通过
9. 服务端编译通过
