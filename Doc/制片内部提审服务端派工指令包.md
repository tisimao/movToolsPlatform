# 制片内部提审服务端派工指令包

## 1. 文档用途

本文件用于把“制片内部提审派工链路”的服务端部分单独拆包，直接派发给后端开发或其他 AI 执行。

本包只关注服务端能力补齐，重点解决：

1. 制片创建的是正式审片任务，而不是只改镜头二级状态
2. 一个任务可包含多个镜头、顺序、提审版本、目标导演
3. 导演任务列表来自正式任务，而不是镜头状态推导
4. 任务提交后系统自动推进镜头二级状态
5. 导演反馈、镜头状态、任务进度在服务端口径统一

目标目录：`D:\project\MovtoolsEx\movtools-server\`

## 2. 本批次目标

这一批完成后，服务端必须具备：

1. `ReviewTask` 正式支持多镜头任务，而不是仅单镜头任务兼容态
2. 制片可创建草稿任务、编辑任务、添加镜头、调整顺序、指定提审版本、提交任务、关闭任务
3. 导演获取到的是正式派发任务及其镜头队列，而不是“所有未关闭任务的平铺单镜头列表”
4. 任务提交后，任务内镜头二级状态统一推进到 `IN_DIRECTOR_REVIEW`
5. 导演开始任务后，任务状态可推进到 `in-review` 等价状态
6. 导演保存反馈、内部通过后，任务进度摘要和镜头摘要能同步刷新
7. 镜头详情可稳定回挂所属任务与最近任务

这一批不做：

1. 不改客户端页面结构
2. 不放松 `DIRECTOR_APPROVED` 才允许正式提交的规则
3. 不把镜头主状态与任务状态合并
4. 不仅补接口壳子而不补真实状态联动

## 3. 已固化业务边界（必须遵守）

1. `READY_FOR_REVIEW` 只表示可入任务，不表示任务已派给导演
2. 导演任务列表的真相源必须是正式 `ReviewTask`
3. `IN_DIRECTOR_REVIEW` 必须由任务提交或任务开始驱动，不允许继续只靠镜头手工写接口充当派工
4. 一个任务可包含多个镜头，镜头顺序由制片确定
5. 任务状态、镜头二级状态、镜头主状态各自独立，但要有明确驱动关系
6. 当前镜头处于 `PENDING_FEEDBACK_FIX` 时，新版本产出本身不自动重置为 `NOT_IN_REVIEW`；需待制作人员确认处理完成后再推进到 `FIX_UPDATED`

---

## 4. 服务端问题定性

当前服务端需要按以下结论理解：

1. 已有 `ReviewTaskShot` 雏形，但制片可用的完整任务装配和提交流程还不完整
2. 当前 `SubmitForReviewAsync(Guid lensId, ...)` 偏向单镜头提交，不足以承载制片成组派工
3. 当前导演页查询任务时更像把 task 当单镜头对象消费，任务详情粒度不够
4. 任务提交、任务开始、镜头状态推进、反馈汇总之间缺少统一编排

## 5. 本批必须完成的服务端能力

### 5.1 任务对象模型补齐

必须确保任务对象至少具备：

1. 任务 ID
2. 所属项目
3. 所属集（可空但建议支持）
4. 任务名称
5. 提审说明
6. 目标导演
7. 创建人
8. 截止时间
9. 任务状态
10. 创建时间 / 更新时间
11. 提交时间
12. 完成时间

### 5.2 任务镜头对象模型补齐

每个任务镜头至少具备：

1. 任务镜头 ID
2. 任务 ID
3. 镜头 ID
4. 顺序号
5. 提审版本号
6. 实际播放版本号或解析结果
7. 任务内处理状态，如未看 / 已看 / 已反馈 / 已完成
8. 反馈数量
9. 最近反馈时间
10. 是否内部通过

### 5.3 任务状态建议

建议统一为：

1. `draft`
2. `ready`
3. `pending`
4. `in-review`
5. `completed`
6. `closed`

如果当前数据库或契约还未完全一致，允许服务端内部做兼容映射，但 API 对外语义必须稳定。

## 6. 必须新增或补齐的接口能力

### 6.1 任务管理接口

必须支持：

1. `POST /api/review-tasks/draft`：创建草稿任务
2. `GET /api/review-tasks/producer`：制片任务列表
3. `GET /api/review-tasks/{id}/detail`：制片任务详情
4. `PUT /api/review-tasks/{id}`：更新任务名称、导演、说明、截止时间
5. `POST /api/review-tasks/{id}/shots`：批量添加镜头到任务
6. `DELETE /api/review-tasks/{id}/shots/{taskShotId}`：从任务移除镜头
7. `PUT /api/review-tasks/{id}/shots/reorder`：调整任务镜头顺序
8. `POST /api/review-tasks/{id}/submit`：提交任务给导演
9. `POST /api/review-tasks/{id}/start`：导演开始任务
10. `POST /api/review-tasks/{id}/complete`：任务完成
11. `POST /api/review-tasks/{id}/close`：任务关闭

补充说明：

1. 历史 `POST /api/review-tasks`、`GET /api/review-tasks/{id}` 仍可作为导演任务链路或兼容入口存在，但不再作为制片装配链路的正式新增基线
2. 制片任务列表 / 详情允许作为正式 `ReviewTask` 的 projection 存在，不要求与导演任务详情完全同形

### 6.2 查询接口要求

`GET /api/review-tasks/producer` 至少支持：

1. 按项目过滤
2. 按导演过滤
3. 按状态过滤
4. 按当前用户可访问项目过滤
5. 返回镜头数量、已反馈数量、内部通过数量、最近更新时间等摘要

`GET /api/review-tasks/{id}/detail` 至少返回：

1. 任务基础信息
2. 任务状态
3. 提交人
4. 目标导演
5. 截止时间
6. 镜头总数
7. 镜头队列
8. 每个镜头当前二级状态
9. 每个镜头反馈数量
10. 每个镜头最近反馈时间
11. 每个镜头提审版本

当前正式状态映射补充：

1. 服务端真实状态集合为：`draft`、`ready`、`pending`、`in-review`、`completed`、`closed`
2. 制片页展示文案 `待提交` 对应 `ready`
3. 导演页结果投影 `approved / rejected` 分别对应 `completed / closed`

## 7. 状态联动规则（必须实现）

### 7.1 任务提交时

1. 仅允许草稿或待提交任务进入提交动作
2. 校验任务内至少有一个镜头
3. 校验每个镜头必须处于允许入任务的状态：
   - `READY_FOR_REVIEW`
   - 或按业务允许 `FIX_UPDATED`
4. 校验每个镜头提审版本信息完整
5. 提交成功后：
   - 任务状态进入 `pending`
   - 任务提交时间写入
   - 任务内镜头二级状态推进到 `IN_DIRECTOR_REVIEW`
   - 镜头 `LatestReviewTaskId` 指向当前任务

### 7.2 导演开始任务时

1. 仅导演本人或管理员可执行
2. 任务状态从 `pending` 进入 `in-review`
3. 可写入开始时间或首开时间（如已有字段可复用）

### 7.3 导演保存反馈时

1. 反馈必须绑定：任务、镜头、版本、帧或时间码
2. 镜头进入 `PENDING_FEEDBACK_FIX`
3. 任务镜头反馈数量、最近反馈时间、任务汇总统计同步刷新
4. 反馈评论落库时必须同步写入 `review_comments.VersionNum`，避免反馈查询链路因版本字段缺失而报数据库结构错误

### 7.4 导演确认通过时

1. 对应镜头进入 `DIRECTOR_APPROVED`
2. 任务镜头状态标记为已完成或通过等价状态
3. 若任务内全部镜头都完成本轮处理，可允许任务进入 `completed`

### 7.5 制作回流时

1. 制作人员处理反馈并产生新版本后，版本升级先重置二级状态为 `NOT_IN_REVIEW`
2. 制作人员可把镜头推进到 `FIX_UPDATED`
3. 制片在下一轮任务里重新组织，不自动继承旧任务进行中状态

## 8. 权限边界

### 8.1 制片

制片必须具备：

1. 创建任务
2. 编辑草稿任务
3. 添加或移除任务镜头
4. 调整任务镜头顺序
5. 指定提审版本
6. 提交任务
7. 关闭任务
8. 查看任务反馈汇总

### 8.2 导演

导演必须具备：

1. 查看分配给自己的任务
2. 开始任务
3. 查看任务详情与镜头队列
4. 保存反馈卡片
5. 将镜头推进到 `PENDING_FEEDBACK_FIX`
6. 将镜头推进到 `DIRECTOR_APPROVED`

### 8.3 制作人员

制作人员不参与任务编排，只允许：

1. 查看回挂到镜头的反馈
2. 在合规条件下把镜头推进到 `FIX_UPDATED`
